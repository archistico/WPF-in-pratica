# 03.04.02 - `IValueConverter`

## L'interfaccia base dei converter in WPF

In WPF, il modo standard per creare un converter semplice è implementare l'interfaccia `IValueConverter`.

Questa interfaccia definisce due metodi:
- `Convert`
- `ConvertBack`

Questa coppia rappresenta le due direzioni possibili della conversione.

## Perché è una buona struttura

Il binding può lavorare:
- dalla source verso il target;
- dal target verso la source.

Per questo un converter ben progettato può sapere come trasformare il valore in entrambe le direzioni, quando serve.

## La struttura dell'interfaccia

Un converter tipico ha questa forma:

```csharp
using System;
using System.Globalization;
using System.Windows.Data;

public class MioConverter : IValueConverter
{
    public object Convert(object value, Type targetType, object parameter, CultureInfo culture)
    {
        // conversione source -> target
        throw new NotImplementedException();
    }

    public object ConvertBack(object value, Type targetType, object parameter, CultureInfo culture)
    {
        // conversione target -> source
        throw new NotImplementedException();
    }
}
```

## Come leggere i parametri

### `value`
È il valore in ingresso da convertire.

### `targetType`
È il tipo richiesto dalla proprietà target o dal ritorno atteso.

### `parameter`
È un parametro opzionale che puoi passare dal binding.

### `culture`
È la cultura usata per la conversione o formattazione.

Questi quattro parametri danno già molta flessibilità.

## `Convert`

`Convert` è il metodo usato quando il dato va dalla source al target.

Esempio tipico:
- nel modello hai `bool`;
- nella UI ti serve `Visibility`.

In questo caso, `Convert` fa la trasformazione.

## `ConvertBack`

`ConvertBack` è il metodo usato quando il dato deve tornare dal target alla source.

Serve solo quando il binding lavora in una direzione in cui il ritorno ha senso.
Per esempio:
- da testo UI a numero;
- da `Visibility` a `bool`, se davvero vuoi supportarlo;
- da stringa mostrata a valore interno.

Non tutti i converter devono implementarlo in modo significativo.

## Un esempio semplice

```csharp
public class UpperCaseConverter : IValueConverter
{
    public object Convert(object value, Type targetType, object parameter, CultureInfo culture)
    {
        return value?.ToString()?.ToUpper(culture) ?? string.Empty;
    }

    public object ConvertBack(object value, Type targetType, object parameter, CultureInfo culture)
    {
        return value?.ToString() ?? string.Empty;
    }
}
```

Questo esempio è semplice, ma mostra già il meccanismo:
- in avanti trasformi il testo;
- nel ritorno decidi come comportarti.

## Come si usa nello XAML

Prima lo registri come risorsa:

```xml
<Window.Resources>
    <local:UpperCaseConverter x:Key="UpperCaseConverter" />
</Window.Resources>
```

Poi lo usi nel binding:

```xml
<TextBlock Text="{Binding Nome, Converter={StaticResource UpperCaseConverter}}" />
```

## Quando `IValueConverter` è la scelta giusta

È la scelta giusta quando:
- la conversione riguarda un singolo valore;
- non hai bisogno di combinare più sorgenti;
- la trasformazione è chiara e contenuta;
- vuoi riusare il comportamento in più binding.

## Quando non basta

Non basta quando devi lavorare su più valori contemporaneamente.
In quei casi entra in gioco `IMultiValueConverter`, che però appartiene a un livello successivo.

## Errori tipici

### Pensare che un converter sia solo una scorciatoia grafica
In realtà è un vero componente del binding.

### Implementare `ConvertBack` senza criterio
Se non serve, è meglio dichiararlo in modo chiaro.

### Mettere logica troppo ampia dentro il converter
Un converter deve restare focalizzato.

## Regola pratica

`IValueConverter` è il meccanismo standard per adattare un singolo valore nel binding.
Quando il problema è “ho un valore, ma la UI lo vuole in un'altra forma”, sei spesso nel territorio giusto.
