# 03.07.09 - Come si dichiara una dependency property personalizzata

## Il momento in cui il concetto diventa concreto

Dopo aver capito il ruolo delle dependency property, è il momento di vedere come se ne dichiara una personalizzata.

Questo è uno dei passaggi più tecnici del capitolo, ma se lo leggi con ordine non è difficile.

L'obiettivo è capire il modello standard, non memorizzare tutto a forza.

## La struttura tipica

Una dependency property personalizzata viene di solito dichiarata con tre elementi:

1. un campo statico `DependencyProperty`;
2. una registrazione tramite `DependencyProperty.Register(...)`;
3. un wrapper CLR comodo da usare.

## Esempio completo

```csharp
using System.Windows;

public class MioControllo : Control
{
    public static readonly DependencyProperty TitoloProperty =
        DependencyProperty.Register(
            nameof(Titolo),
            typeof(string),
            typeof(MioControllo),
            new PropertyMetadata(string.Empty));

    public string Titolo
    {
        get => (string)GetValue(TitoloProperty);
        set => SetValue(TitoloProperty, value);
    }
}
```

## Come leggere l'esempio

### 1. Campo statico
```csharp
public static readonly DependencyProperty TitoloProperty
```

Questo campo rappresenta l'identità della dependency property nel property system.

### 2. Registrazione
```csharp
DependencyProperty.Register(...)
```

Qui stai dicendo al framework:
“esiste una dependency property chiamata `Titolo`, di tipo `string`, appartenente a `MioControllo`”.

### 3. Wrapper CLR
```csharp
public string Titolo
{
    get => (string)GetValue(TitoloProperty);
    set => SetValue(TitoloProperty, value);
}
```

Questo rende la proprietà comoda da usare come una normale proprietà .NET, pur delegando il valore al property system.

## Perché servono tutti e tre i pezzi

Servono perché ogni parte ha un ruolo diverso:

- il campo statico è la registrazione vera nel sistema WPF;
- il wrapper CLR rende la proprietà usabile in modo naturale;
- la chiamata a `Register` collega nome, tipo, owner e metadati.

## Un punto importante sul naming

C'è una convenzione molto importante:

- proprietà CLR: `Titolo`
- campo dependency property: `TitoloProperty`

Questa convenzione non è un dettaglio estetico.
È parte del modo normale in cui WPF e gli sviluppatori si aspettano di leggere questo codice.

## Quando userai davvero questo schema

Lo userai soprattutto quando creerai:
- controlli personalizzati;
- `UserControl` con proprietà bindabili e stilizzabili;
- componenti riusabili;
- proprietà che devono integrarsi con XAML e col property system.

## Perché all'inizio sembra più pesante di quanto sia

Perché rispetto a una proprietà normale:

```csharp
public string Titolo { get; set; }
```

questa struttura appare molto più articolata.

Ma è proprio questa struttura che abilita l'integrazione col sistema WPF.

## Errori tipici

### Dimenticare il wrapper CLR
La proprietà esiste nel sistema, ma il codice diventa meno naturale da usare.

### Sbagliare naming tra `Titolo` e `TitoloProperty`
Si rompe la convenzione e il codice diventa meno leggibile.

### Pensare che il campo statico sia “la proprietà da usare” nel codice applicativo
No: normalmente userai il wrapper CLR.

## Regola pratica

Per dichiarare una dependency property personalizzata, pensa sempre a questo trio:
- registrazione nel framework;
- identità statica della property;
- wrapper CLR per usarla bene.
