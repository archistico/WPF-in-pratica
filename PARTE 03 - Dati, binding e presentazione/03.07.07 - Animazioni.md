# 03.07.07 - Animazioni

## Uno dei punti in cui le dependency property mostrano tutta la loro natura WPF

Le animazioni in WPF lavorano molto bene sulle dependency property.
Questo non è un dettaglio: è uno dei segnali più chiari che queste proprietà fanno parte del cuore del framework UI.

## Perché le animazioni si legano così bene alle dependency property

Un'animazione in WPF non lavora genericamente “sull'oggetto”.
Lavora su una proprietà.

Per esempio:
- `Opacity`
- `Width`
- `Height`
- `RenderTransform.X`
- `ScaleX`
- `ScaleY`

Molte di queste proprietà sono dependency property, oppure vivono dentro oggetti che partecipano bene al property system.

## Un esempio semplice

```xml
<DoubleAnimation Storyboard.TargetName="btnSalva"
                 Storyboard.TargetProperty="Opacity"
                 From="0"
                 To="1"
                 Duration="0:0:0.4"/>
```

Qui il target è la proprietà `Opacity`.
`Opacity` è una dependency property, ed è proprio questo che permette all'animazione di lavorarci in modo naturale nel sistema WPF.

## Perché una proprietà normale non offre la stessa integrazione

Una proprietà CLR normale non partecipa al property system allo stesso livello.
Questo rende molto meno naturale gestirla nel contesto delle animazioni WPF standard.

Il framework è progettato per animare proprietà che capisce profondamente.

## Un esempio mentale utile

Immagina un controllo personalizzato con una proprietà `ValoreEvidenziazione`.
Se vuoi che questa proprietà sia:
- impostabile in XAML;
- stilizzabile;
- animabile;

allora dichiararla come dependency property può diventare la scelta giusta.

## Animazione e precedenza del valore

Le animazioni mostrano molto bene anche un altro aspetto fondamentale:
il valore finale di una dependency property può essere influenzato da fonti diverse, e l'animazione è una di queste.

Questo è molto diverso dall'idea di semplice proprietà con setter tradizionale.

## Quando questo è importante nei controlli personalizzati

È importante quando stai costruendo:
- controlli con stati visivi;
- componenti animabili;
- proprietà che devono partecipare a micro-animazioni o transizioni;
- elementi grafici o UI riusabili.

Se vuoi che il tuo controllo si comporti “come un vero controllo WPF”, l'animabilità delle proprietà può diventare un criterio importante.

## Un punto da non fraintendere

Non ogni proprietà che crei deve essere pensata per le animazioni.
Ma se una proprietà appartiene davvero al comportamento visivo del controllo, il fatto che possa essere animata è un forte indizio del valore del modello dependency property.

## Errori tipici

### Dimenticare che le animazioni lavorano soprattutto su proprietà che il framework tratta bene
Si cerca di usare lo stesso schema su proprietà che non partecipano al property system in modo naturale.

### Dichiarare proprietà visuali importanti come semplici proprietà CLR e poi aspettarsi la stessa integrazione
Il risultato è meno “WPF-native”.

## Regola pratica

Se una proprietà del tuo controllo deve partecipare seriamente ad animazioni o transizioni WPF, una dependency property è molto spesso la scelta naturale.
