# 03.07.03 - Esempi di dependency property che usi già

## Il modo migliore per togliere subito l'idea di “argomento esotico”

Uno dei modi più efficaci per capire le dependency property è rendersi conto che le stai già usando continuamente.

Non sono qualcosa che compare solo quando scrivi controlli personalizzati avanzati.
Sono ovunque nella UI WPF quotidiana.

## Alcuni esempi molto comuni

### `Width`

```xml
<Button Width="120" Content="Salva"/>
```

`Width` è una dependency property.

### `Height`

```xml
<TextBox Height="30"/>
```

Anche `Height` è una dependency property.

### `Background`

```xml
<Button Background="SteelBlue"/>
```

`Background` è una dependency property.

### `Foreground`

```xml
<TextBlock Foreground="DarkRed"/>
```

Anche questa è una dependency property.

### `Visibility`

```xml
<TextBlock Visibility="Collapsed"/>
```

`Visibility` è una dependency property.

### `Text`

```xml
<TextBox Text="{Binding Nome}"/>
```

La proprietà `Text` del `TextBox` è una dependency property.

### `ItemsSource`

```xml
<ListBox ItemsSource="{Binding Clienti}"/>
```

Anche `ItemsSource` è una dependency property.

## Perché è così importante vedere questi esempi

Perché mostrano che molte delle cose che rendono WPF “WPF” si appoggiano proprio sulle dependency property.

In particolare:
- binding su `Text`;
- stili su `Background`;
- animazioni su `Opacity`;
- ereditarietà su certe proprietà;
- template e sistema visivo.

Tutto questo vive bene proprio perché molte proprietà fondamentali del framework sono dependency property.

## Un esempio molto istruttivo: `Opacity`

```xml
<Button Opacity="0.8" Content="Salva"/>
```

`Opacity` può:
- essere impostata localmente;
- essere animata;
- essere influenzata da uno stile;
- essere cambiata da trigger.

È un ottimo esempio del tipo di proprietà che beneficia molto del property system.

## Un altro esempio molto istruttivo: `DataContext`

```xml
<Grid DataContext="{Binding ClienteCorrente}">
    <TextBox Text="{Binding Nome}" />
</Grid>
```

`DataContext` è una dependency property.

Questo è importantissimo, perché spiega anche come può:
- propagarsi ai figli;
- partecipare al sistema di binding;
- avere comportamento speciale nel framework.

## Che cosa devi notare

Non è solo che queste proprietà esistono.
È che molte delle funzionalità avanzate che hai già studiato si reggono proprio sul fatto che queste proprietà siano dependency property.

Questa osservazione collega molto bene il capitolo con tutto ciò che hai già visto.

## Un piccolo esercizio mentale utile

Ogni volta che incontri una proprietà WPF importante, chiediti:
- partecipa a binding?
- può essere stilizzata?
- può essere animata?
- ha comportamento speciale nel framework?

Se la risposta è spesso sì, è molto probabile che tu sia davanti a una dependency property.

## Errore tipico

L'errore classico è pensare alle dependency property solo come qualcosa che “dichiari tu”.
Prima ancora di dichiararne una, ne stai già usando centinaia dentro il framework.

## Regola pratica

Prima di studiare come crearle, è utile vedere bene che le dependency property sono già la base di gran parte delle proprietà che usi ogni giorno in WPF.
