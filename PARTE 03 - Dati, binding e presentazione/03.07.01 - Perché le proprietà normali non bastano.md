# 03.07.01 - Perché le proprietà normali non bastano

## Il problema che WPF doveva risolvere

In C#, una proprietà normale va benissimo in moltissimi casi.

Per esempio:

```csharp
public string Titolo { get; set; }
```

oppure, in forma più completa:

```csharp
private string _titolo = "";

public string Titolo
{
    get => _titolo;
    set => _titolo = value;
}
```

Questo modello è perfetto per tantissima logica applicativa. Ma WPF aveva un problema più ambizioso da risolvere.

I controlli WPF devono poter partecipare in modo nativo a meccanismi come:
- binding;
- stili;
- template;
- animazioni;
- ereditarietà di certi valori;
- metadati di comportamento;
- sistema centralizzato di risoluzione del valore.

Le proprietà CLR normali, da sole, non bastano a fornire tutto questo in modo coerente e integrato.

## Il limite delle proprietà normali

Una proprietà normale appartiene semplicemente a un oggetto C#.
Può leggere e scrivere un valore, ma non possiede automaticamente un sistema che sappia:
- da dove arriva quel valore;
- se è stato impostato localmente;
- se arriva da uno stile;
- se arriva da un binding;
- se è animato;
- se eredita il valore da un antenato;
- se ha callback speciali da eseguire quando cambia.

WPF, invece, ha bisogno proprio di questo livello di ricchezza.

## Un esempio mentale molto utile

Prendi una proprietà come `Background` di un `Button`.

Quella proprietà può essere influenzata da:
- valore locale;
- stile;
- trigger;
- template;
- tema;
- animazione.

Con una proprietà normale, coordinare tutto questo sarebbe molto più difficile e disomogeneo.
Con il sistema delle dependency property, WPF ha un modello unico per gestire la cosa.

## Non è solo una questione di “più potenza”

Il punto non è semplicemente che le dependency property sono “più potenti”.
Il punto è che permettono a WPF di avere un **property system** ricco e coerente.

Questa è la frase chiave:
le dependency property non sono solo proprietà speciali, sono parte di un sistema più grande.

## Un confronto molto concreto

### Proprietà normale
Contiene un valore e la logica del getter/setter.

### Dependency property
Partecipa a un sistema che gestisce:
- valori da sorgenti diverse;
- metadati;
- notifica interna al framework;
- supporto avanzato per UI e presentazione.

## Perché questo è fondamentale in WPF

Perché WPF non è una libreria UI che aggiunge solo controlli.
È una piattaforma che modella molto profondamente il comportamento delle proprietà della UI.

Se questo punto entra bene, il resto del capitolo diventa molto più chiaro.

## Quando il limite delle proprietà normali si vede davvero

Si vede quando vuoi che una proprietà di un controllo:
- sia bindabile in modo naturale;
- sia stilizzabile;
- sia animabile;
- possa reagire con callback interne;
- si integri col property system di WPF.

In questi casi una proprietà CLR semplice non dà al framework tutto ciò che serve.

## Errore tipico

L'errore classico è pensare:
“una dependency property è solo una proprietà più complicata”.

No.
È una proprietà inserita in un sistema molto più ricco, e tutta la sua complessità deriva da lì.

## Regola pratica

Le proprietà normali vanno benissimo per molta logica C#.
Le dependency property esistono quando una proprietà deve vivere davvero dentro il motore UI di WPF.
