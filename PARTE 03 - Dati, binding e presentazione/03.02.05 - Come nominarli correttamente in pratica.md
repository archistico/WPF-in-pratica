# 03.02.05 - Come nominarli correttamente in pratica

## La chiarezza dei nomi conta più di quanto sembri

Quando inizi a lavorare con più oggetti, sezioni con `DataContext` diversi e sorgenti esplicite, i nomi diventano cruciali.

Un naming poco chiaro rende il binding molto più difficile da leggere, anche se tecnicamente funziona.
Un naming buono, invece, chiarisce da solo gran parte della struttura della finestra.

Questo vale sia per:
- proprietà del modello o del view model;
- nomi dei controlli;
- risorse;
- nomi delle sezioni logiche.

## Come nominare gli oggetti del `DataContext`

Se hai un oggetto principale, il suo nome deve dire chiaramente che ruolo ha.

Buoni esempi:
- `ClienteCorrente`
- `OrdineSelezionato`
- `DocumentoCorrente`
- `ImpostazioniUtente`
- `FiltroRicerca`

Cattivi esempi:
- `Dati`
- `Info`
- `Oggetto`
- `Temp`
- `Varie`

I nomi vaghi fanno perdere contesto.

## Come nominare i sotto-oggetti

Se hai una classe radice che espone più proprietà, i nomi devono far capire subito la responsabilità di ciascuna.

Esempio buono:

```csharp
public class SchedaClienteViewModel
{
    public Cliente ClienteCorrente { get; set; } = new();
    public Indirizzo IndirizzoPrincipale { get; set; } = new();
    public ObservableCollection<Ordine> OrdiniRecenti { get; set; } = new();
    public string TitoloFinestra { get; set; } = "";
}
```

Qui il lettore capisce subito di cosa si parla.

## Come nominare i controlli quando servono

Non tutti i controlli devono avere un nome.
Ma quando un nome serve, deve essere utile.

Buoni esempi:
- `txtRicerca`
- `lstOrdini`
- `dgClienti`
- `cmbStati`
- `btnSalva`

Cattivi esempi:
- `control1`
- `boxA`
- `x1`
- `prova`

Nel caso di `ElementName`, il nome del controllo diventa parte del binding e quindi conta ancora di più.

## Come nominare le risorse

Anche le risorse usate con `Source` o `StaticResource` devono essere leggibili.

Buoni esempi:
- `ClienteSelezionato`
- `BrushPrimario`
- `StilePulsantePrincipale`
- `TitoloSchedaStyle`

Cattivi esempi:
- `res1`
- `myData`
- `styleBlue`
- `obj`

## Un esempio che si legge bene

```xml
<Window.Resources>
    <local:Cliente x:Key="ClienteSelezionato" />
</Window.Resources>

<Grid DataContext="{Binding SchedaCliente}">
    <TextBox Text="{Binding ClienteCorrente.Nome}" />

    <TextBlock Text="{Binding Text, ElementName=txtRicerca}" />

    <TextBlock Text="{Binding Nome, Source={StaticResource ClienteSelezionato}}" />
</Grid>
```

Anche senza sapere tutto del progetto, si capisce molto meglio perché i nomi aiutano il lettore.

## Un principio molto utile

Un buon nome deve rispondere almeno a una di queste domande:
- che oggetto è?
- che ruolo ha?
- rispetto a cosa è “corrente”, “selezionato”, “principale”, “attivo”?
- in quale parte della UI o del modello vive?

## Quando i nomi peggiorano il progetto

Peggiorano il progetto quando:
- sono troppo generici;
- sono troppo abbreviati senza criterio;
- non distinguono ruoli simili;
- cambiano stile in modo incoerente da file a file.

## Errori tipici

### Nomini brevi ma opachi
Il codice sembra veloce da scrivere ma difficile da mantenere.

### Nomi lunghissimi e contorti
Anche l'eccesso può peggiorare la leggibilità.

### Nessuna coerenza tra nomi di modello e nomi di UI
Il lettore fatica a collegare i pezzi.

## Regola pratica

In una schermata WPF, i nomi devono chiarire il contesto.
Se leggendo un binding capisci subito “da dove arriva questo dato”, allora il naming sta lavorando bene.
