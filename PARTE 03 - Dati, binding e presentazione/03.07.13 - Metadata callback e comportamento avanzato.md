# 03.07.13 - Metadata, callback e comportamento avanzato

## Qui la dependency property mostra tutta la sua profondità

Finora hai visto la dependency property nel suo schema base.
Ma uno dei suoi aspetti più potenti è che può essere registrata con **metadati** che definiscono comportamenti speciali.

Questo è uno dei motivi per cui il sistema delle dependency property è molto più ricco di una semplice proprietà CLR.

## Che cosa contengono i metadati

I metadati possono definire, tra le altre cose:
- valore predefinito;
- callback quando il valore cambia;
- callback di coercizione;
- indicazioni al framework sul layout;
- ereditarietà dei valori;
- altri comportamenti avanzati.

Per ora concentriamoci sui casi più utili da capire bene:
- default value;
- property changed callback;
- metadata flags.

## Un esempio con callback di cambiamento

```csharp
public static readonly DependencyProperty TitoloProperty =
    DependencyProperty.Register(
        nameof(Titolo),
        typeof(string),
        typeof(MioControllo),
        new PropertyMetadata(string.Empty, OnTitoloChanged));
```

Callback:

```csharp
private static void OnTitoloChanged(DependencyObject d, DependencyPropertyChangedEventArgs e)
{
    var controllo = (MioControllo)d;

    string vecchioValore = (string)e.OldValue;
    string nuovoValore = (string)e.NewValue;

    // qui puoi reagire al cambiamento
}
```

## Perché questa callback è importante

Perché permette al controllo di reagire quando la dependency property cambia, indipendentemente dal fatto che il cambio arrivi da:
- valore locale;
- binding;
- stile;
- altro meccanismo del property system.

Questo è un punto molto potente.

## Un caso tipico d'uso

Un controllo può usare la callback per:
- aggiornare parti interne;
- invalidare uno stato visivo;
- sincronizzare elementi interni;
- eseguire logica UI conseguente al cambiamento.

## Metadata flags

Oltre ai callback, i metadati possono includere flag utili al framework.

Per esempio, in registrazioni più ricche puoi incontrare metadati che dicono a WPF che il cambio di una proprietà deve causare:
- nuovo measure;
- nuovo arrange;
- nuovo rendering.

Questo è molto utile nei controlli visuali seri.

## Un esempio concettuale

Se hai una proprietà `ValoreEvidenziazione` che cambia l'aspetto del controllo, potresti voler dire al framework che un cambio di quella proprietà richiede un nuovo render.
Questo tipo di collaborazione col framework è esattamente uno dei motivi per cui esistono i metadati.

## Perché non conviene mettere tutta la logica nel wrapper CLR

Molti principianti pensano:
“metto la logica nel setter del wrapper”.

Ma il wrapper CLR non è il punto migliore per tutta la logica avanzata del property system.

Le callback dei metadati sono spesso più corrette perché reagiscono al cambio della dependency property a livello framework, non solo al passaggio dal setter CLR.

## Quando questi strumenti diventano davvero importanti

Diventano importanti quando stai costruendo:
- controlli personalizzati veri;
- componenti UI riusabili;
- proprietà con effetti visivi o strutturali;
- proprietà che devono attivare comportamento del controllo.

## Errori tipici

### Ignorare i metadati e mettere tutto nel wrapper
Si perde una parte importante del modello WPF.

### Usare callback senza uno scopo chiaro
Il controllo diventa più difficile da mantenere.

### Non distinguere tra comportamento locale del codice e comportamento di proprietà nel framework
Le dependency property richiedono un pensiero più sistemico.

## Regola pratica

I metadati e le callback esistono per collegare la proprietà al comportamento del framework e del controllo.
Se una dependency property deve fare più che contenere un valore, spesso la risposta passa proprio da lì.
