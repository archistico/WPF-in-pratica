# 03.07.05 - Binding

## Uno dei motivi più forti per cui le dependency property esistono

Tra tutti i vantaggi delle dependency property, il binding è uno dei più importanti e più facili da percepire.

Molte proprietà che usi nei binding quotidiani in WPF sono dependency property proprio perché il framework deve poterle trattare come punti naturali di collegamento tra dati e UI.

## Perché il binding si appoggia così bene alle dependency property

Il binding in WPF ha bisogno di un sistema di proprietà che sia:
- osservabile dal framework;
- integrato con il motore di presentazione;
- compatibile con stili, template e aggiornamenti dinamici;
- coerente con il resto della UI.

Le dependency property offrono proprio questo terreno comune.

## Un esempio che già conosci

```xml
<TextBox Text="{Binding Nome}" />
```

Qui la proprietà target è `Text`.
Nel caso del `TextBox`, `Text` è una dependency property.

Questo spiega perché il binding può lavorare così bene in modo nativo su di essa.

## Un altro esempio molto classico

```xml
<ListBox ItemsSource="{Binding Clienti}" />
```

Anche `ItemsSource` è una dependency property.

E infatti può partecipare naturalmente a:
- binding;
- aggiornamenti;
- view;
- presentazione dinamica dei dati.

## Perché questo conta anche quando crei proprietà tue

Se stai sviluppando un controllo o un `UserControl` e vuoi che una tua proprietà possa essere bindata in modo naturale da XAML, molto spesso dichiararla come dependency property è la scelta corretta.

Per esempio, se hai un controllo personalizzato con una proprietà `Titolo`, e vuoi usarlo così:

```xml
<local:MioControllo Titolo="{Binding NomeCliente}" />
```

allora è molto spesso sensato che `Titolo` sia una dependency property.

## Un esempio minimale di idea

Supponiamo di avere un `UserControl` con una proprietà `Titolo`.
Se questa proprietà è una dependency property, il controllo può partecipare meglio al sistema di binding WPF.

Questo è uno degli scenari più classici per iniziare a crearne di personalizzate.

## Il binding non è l'unico motivo, ma è uno dei più forti

Attenzione: il fatto che una proprietà sia bindabile non significa automaticamente che **debba** essere una dependency property in ogni situazione.
Ma nei controlli WPF, se la proprietà appartiene davvero alla UI e deve essere usata da XAML come proprietà del controllo, il binding è uno dei motivi più forti a favore.

## Un piccolo confronto utile

### Proprietà di view model
Qui di solito usi proprietà CLR normali + `INotifyPropertyChanged`.

### Proprietà del controllo WPF
Qui spesso una dependency property è la scelta naturale.

Questa distinzione è fondamentale e prepara molto bene anche il confronto finale con `INotifyPropertyChanged`.

## Perché il binding da solo non basta a definire tutto

Non ogni proprietà bindabile in senso lato deve diventare una dependency property.
La domanda giusta è:
“questa proprietà appartiene a un oggetto UI che deve vivere nel property system di WPF?”

Se sì, allora la dependency property è un'ottima candidata.

## Errori tipici

### Pensare che ogni proprietà coinvolta in binding debba essere una dependency property
Non è vero: nei view model spesso basta `INotifyPropertyChanged`.

### Non dichiarare come dependency property una proprietà di controllo che deve integrarsi bene nello XAML
Il controllo risulta meno naturale da usare.

## Regola pratica

Se la proprietà appartiene a un controllo WPF e vuoi che partecipi al binding in modo nativo e coerente con il framework, una dependency property è spesso la scelta giusta.
