# 03.07.17 - Il punto da portare a casa

## La sintesi più importante del capitolo

Se dovessi riassumere tutto il capitolo sulle dependency property in poche idee forti, sarebbero queste:

- una dependency property è una proprietà gestita dal property system di WPF;
- esiste perché le proprietà normali non bastano per tutte le esigenze della UI;
- supporta molto bene binding, stili, animazioni, template e metadati;
- ha senso soprattutto nei controlli e nei `UserControl`, non nei view model.

Queste sono le idee da fissare davvero bene.

## Il cambio di mentalità decisivo

Il lettore non deve vedere la dependency property come una semplice proprietà “più complicata”.
Deve vederla come una proprietà che entra nel sistema UI di WPF in modo pieno.

Questa è la differenza fondamentale.

Una proprietà CLR normale appartiene soprattutto all'oggetto.
Una dependency property appartiene anche al framework.

## La distinzione che chiarisce quasi tutto

Tra i vari punti del capitolo, uno va ricordato più di tutti:

- **proprietà di view model o modello** → proprietà CLR + `INotifyPropertyChanged`
- **proprietà di controllo WPF** → spesso dependency property

Questa regola pratica evita già moltissimi errori di progettazione.

## La struttura tecnica da ricordare

Quando dovrai dichiararne una, pensa sempre a questi tre elementi:
- registrazione nel framework;
- campo statico `DependencyProperty`;
- wrapper CLR.

Non serve memorizzare tutto in modo meccanico, ma devi sapere leggere questa struttura con sicurezza.

## I vantaggi veri da non dimenticare

Le dependency property hanno senso perché la proprietà può diventare:
- bindabile in modo nativo;
- stilizzabile;
- animabile;
- arricchita da metadati e callback;
- eventualmente ereditabile nel valore.

Questa è la vera giustificazione della loro complessità.

## Il criterio pratico finale

Ogni volta che devi creare una proprietà, chiediti:
“questa proprietà deve davvero comportarsi come una proprietà nativa WPF del controllo?”

Se sì, una dependency property è una candidata forte.
Se no, probabilmente una proprietà CLR normale è la scelta migliore.

## Regola finale

Le dependency property non servono a rendere il codice “più WPF” in modo astratto.
Servono quando vuoi che una proprietà entri davvero nel motore di presentazione di WPF.
Se il lettore porta a casa questa idea, il capitolo ha centrato il punto.
