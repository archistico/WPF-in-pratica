# 03.05.06 - Filtering

## Filtrare senza distruggere i dati originali

Il filtering è uno degli usi più interessanti di una view.

Permette di dire:
“mostrami solo gli elementi che soddisfano questa condizione”

senza dover modificare la collezione originale.

Questo è esattamente il tipo di problema in cui una view mostra tutta la sua eleganza.

## Dove si applica

Il filtro si applica alla view, non ai dati sottostanti.

Con una `ICollectionView`, la forma tipica è questa:

```csharp
view.Filter = item => ((Cliente)item).Attivo;
```

Qui la view mostrerà solo gli elementi per cui la condizione restituisce `true`.

## Un esempio concreto

Supponiamo di avere:

```csharp
public class Cliente
{
    public string Nome { get; set; } = "";
    public bool Attivo { get; set; }
}
```

e la collezione:

```csharp
public ObservableCollection<Cliente> Clienti { get; } = new();
```

Nel costruttore:

```csharp
Clienti.Add(new Cliente { Nome = "Anna", Attivo = true });
Clienti.Add(new Cliente { Nome = "Luca", Attivo = false });
Clienti.Add(new Cliente { Nome = "Marco", Attivo = true });

var view = CollectionViewSource.GetDefaultView(Clienti);
view.Filter = item => ((Cliente)item).Attivo;
```

XAML:

```xml
<ListBox ItemsSource="{Binding Clienti}"
         DisplayMemberPath="Nome"
         Width="220"
         Height="180"
         Margin="20"/>
```

La UI mostrerà solo:
- Anna
- Marco

## Perché questo è meglio che creare una nuova lista filtrata ogni volta

Perché:
- i dati originali restano intatti;
- il filtro vive nel livello di presentazione;
- puoi cambiare la condizione senza riscrivere la collection;
- la struttura resta più coerente con WPF.

## Filtri dinamici

Il filtering diventa ancora più interessante quando la condizione dipende dall'input dell'utente.

Per esempio:
- testo di ricerca;
- checkbox “solo attivi”;
- selezione categoria;
- intervallo date.

In questi casi puoi cambiare la logica del filtro e poi chiedere alla view di aggiornarsi.

Esempio:

```csharp
view.Refresh();
```

Questo forza la rivalutazione del filtro.

## Un piccolo caso realistico

Immagina una schermata clienti con una `TextBox` di ricerca.
La collezione resta sempre la stessa, ma la view mostra solo gli elementi il cui nome contiene il testo digitato.

Questo è un uso molto comune e molto pulito della view.

## Attenzione a `Refresh()`

Se il filtro dipende da uno stato esterno che cambia, la view potrebbe dover essere aggiornata esplicitamente con `Refresh()`.

Questo è un dettaglio pratico importante da ricordare.

## Errori tipici

### Filtrare ricreando continuamente la collezione
Si complica il codice senza necessità.

### Dimenticare che il filtro agisce sulla view
La collezione vera continua a contenere tutti gli elementi.

### Cambiare la condizione del filtro ma non aggiornare la view
Il controllo non riflette il nuovo criterio.

## Regola pratica

Se vuoi mostrare solo una parte dei dati, pensa prima a un filtro sulla view.
È quasi sempre più pulito che manipolare direttamente la collezione originale.
