# 03.03.06 - Attenzione: `ObservableCollection<T>` non sostituisce `INotifyPropertyChanged`

## Uno degli equivoci più comuni del capitolo

Dopo aver scoperto `ObservableCollection<T>`, molti pensano:
“Perfetto, adesso la mia lista è osservabile, quindi sono a posto”.

Non è così.

`ObservableCollection<T>` risolve un problema preciso:
**notifica i cambiamenti della collezione**.

Ma non sostituisce `INotifyPropertyChanged`, che risolve un altro problema:
**notificare i cambiamenti delle proprietà dei singoli oggetti**.

Questa distinzione è decisiva.

## Facciamo un esempio concreto

Supponiamo di avere:

```csharp
public class Cliente
{
    public string Nome { get; set; } = "";
}
```

e una collezione:

```csharp
public ObservableCollection<Cliente> Clienti { get; } = new();
```

Se fai:

```csharp
Clienti.Add(new Cliente { Nome = "Anna" });
```

la UI si aggiornerà mostrando il nuovo elemento.

Ma se poi fai:

```csharp
Clienti[0].Nome = "Luca";
```

la UI non necessariamente aggiornerà il nome mostrato, se `Cliente` non implementa `INotifyPropertyChanged`.

## Perché succede

Perché in questo secondo caso la collezione non è cambiata:
- non hai aggiunto un elemento;
- non ne hai rimosso uno;
- non hai modificato la composizione.

Hai cambiato una **proprietà interna di un elemento**.

Questo è un problema di notifica della proprietà, non della collezione.

## Il modello corretto

Quando lavori con liste bindate alla UI, spesso ti servono **entrambi** i livelli:

- `ObservableCollection<T>` per notificare aggiunte/rimozioni;
- `INotifyPropertyChanged` sugli elementi per notificare modifiche interne.

## La versione corretta dell'elemento

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;

public class Cliente : INotifyPropertyChanged
{
    private string _nome = "";

    public string Nome
    {
        get => _nome;
        set
        {
            if (_nome != value)
            {
                _nome = value;
                OnPropertyChanged();
            }
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

Ora hai:
- una collezione osservabile;
- elementi osservabili.

Ed è questa, molto spesso, la combinazione giusta nelle applicazioni reali.

## Un'immagine mentale utile

Pensa così:

### `ObservableCollection<T>`
Osserva il contenitore.

### `INotifyPropertyChanged`
Osserva il contenuto dei singoli elementi.

Questa frase riassume molto bene la differenza.

## Quando questo punto diventa decisivo

Diventa decisivo soprattutto in controlli come:
- `ListBox`
- `ListView`
- `DataGrid`

perché lì vuoi spesso entrambe le cose:
- vedere nuove righe quando le aggiungi;
- vedere aggiornate le celle quando cambia una proprietà dell'elemento.

## Errori tipici

### Pensare che la collezione notifichi anche i cambi interni degli elementi
Non lo fa.

### Implementare `ObservableCollection<T>` ma lasciare gli elementi come classi passive
La griglia o la lista si aggiornano solo a metà.

### Cercare refresh manuali quando il problema è strutturale
Spesso non serve “forzare”, serve notificare correttamente.

## Regola pratica

`ObservableCollection<T>` e `INotifyPropertyChanged` non sono alternative.
Molto spesso lavorano insieme:
la prima per l'elenco, la seconda per i singoli elementi.
