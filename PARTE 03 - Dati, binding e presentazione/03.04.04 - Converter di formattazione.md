# 03.04.04 - Converter di formattazione

## Non tutti i converter servono a cambiare tipo

Molti pensano ai converter solo come trasformazioni drastiche:
- `bool` → `Visibility`
- enum → colore
- numero → brush

Ma ci sono anche converter molto utili che servono soprattutto a **formattare** o rendere più leggibile un valore.

In questo caso il cuore del problema non è tanto cambiare il significato del dato, quanto adattarne la presentazione.

## Quando ha senso un converter di formattazione

Ha senso quando:
- il valore grezzo non è adatto così com'è alla UI;
- la formattazione è riusabile;
- la semplice `StringFormat` non basta;
- vuoi una logica di trasformazione più flessibile.

## Un primo esempio

Supponiamo di avere una proprietà numerica:

```csharp
public decimal Totale { get; set; }
```

e di voler mostrare un testo come:

- `€ 125,50`
- oppure `Totale: € 125,50`

Puoi farlo in vari modi. Un converter diventa utile se vuoi centralizzare questa presentazione.

## Esempio di converter

```csharp
using System;
using System.Globalization;
using System.Windows.Data;

public class EuroFormatConverter : IValueConverter
{
    public object Convert(object value, Type targetType, object parameter, CultureInfo culture)
    {
        if (value is decimal importo)
            return importo.ToString("C", culture);

        return string.Empty;
    }

    public object ConvertBack(object value, Type targetType, object parameter, CultureInfo culture)
    {
        throw new NotSupportedException();
    }
}
```

## Uso nello XAML

```xml
<Window.Resources>
    <local:EuroFormatConverter x:Key="EuroFormatConverter" />
</Window.Resources>

<TextBlock Text="{Binding Totale, Converter={StaticResource EuroFormatConverter}}" />
```

## Perché non sempre basta `StringFormat`

`StringFormat` è ottimo e spesso preferibile quando la formattazione è semplice.

Per esempio:

```xml
<TextBlock Text="{Binding Totale, StringFormat=Totale: {0:C}}" />
```

Questa è una soluzione molto buona.

Allora quando conviene davvero un converter?

### Usa `StringFormat` quando
- la formattazione è semplice;
- non serve logica aggiuntiva;
- vuoi mantenere lo XAML diretto e leggibile.

### Usa un converter quando
- la logica di presentazione è più articolata;
- vuoi riusare la stessa trasformazione in più punti;
- devi gestire casi particolari;
- la semplice formattazione inline non basta.

## Un secondo esempio: stato leggibile

Supponiamo di avere:

```csharp
public int StatoDocumento { get; set; }
```

e di voler mostrare:
- `Bozza`
- `Confermato`
- `Annullato`

Un converter può essere perfetto per questa traduzione di presentazione.

## Un piccolo esempio

```csharp
public class StatoDocumentoConverter : IValueConverter
{
    public object Convert(object value, Type targetType, object parameter, CultureInfo culture)
    {
        return value switch
        {
            0 => "Bozza",
            1 => "Confermato",
            2 => "Annullato",
            _ => "Sconosciuto"
        };
    }

    public object ConvertBack(object value, Type targetType, object parameter, CultureInfo culture)
    {
        throw new NotSupportedException();
    }
}
```

## Errore tipico

L'errore classico è usare un converter anche quando una semplice `StringFormat` sarebbe più chiara.
Questo appesantisce inutilmente il progetto.

L'errore opposto è cercare di fare con `StringFormat` una logica che in realtà richiede una vera conversione.

## Regola pratica

Per la sola formattazione semplice, valuta prima `StringFormat`.
Usa un converter di formattazione quando la trasformazione è riusabile, un po' più intelligente o chiaramente separabile come logica di presentazione.
