# 03.07.11 - Wrapper CLR

## La dependency property non si usa direttamente nel codice quotidiano

Quando dichiari una dependency property, il framework ha bisogno del campo statico `DependencyProperty`.
Ma il codice applicativo e il codice del controllo hanno spesso bisogno di una forma più naturale da usare.

Qui entra in gioco il wrapper CLR.

## Che cos'è

Il wrapper CLR è una proprietà .NET normale che fa da facciata comoda sopra il sistema delle dependency property.

Esempio:

```csharp
public string Titolo
{
    get => (string)GetValue(TitoloProperty);
    set => SetValue(TitoloProperty, value);
}
```

Questa proprietà non contiene il valore in un campo privato tradizionale.
Invece delega lettura e scrittura al property system di WPF tramite:
- `GetValue(...)`
- `SetValue(...)`

## Perché serve

Serve perché rende la property:
- comoda da leggere e usare in C#;
- coerente con le convenzioni .NET;
- esponibile come proprietà naturale del controllo;
- integrata con XAML e col framework.

In pratica, il wrapper CLR è il volto “normale” della dependency property.

## Un esempio completo

```csharp
public static readonly DependencyProperty TitoloProperty =
    DependencyProperty.Register(
        nameof(Titolo),
        typeof(string),
        typeof(MioControllo),
        new PropertyMetadata(string.Empty));

public string Titolo
{
    get => (string)GetValue(TitoloProperty);
    set => SetValue(TitoloProperty, value);
}
```

## Come leggere bene `GetValue` e `SetValue`

### `GetValue(TitoloProperty)`
Chiede al property system il valore corrente di quella dependency property.

### `SetValue(TitoloProperty, value)`
Dice al property system di impostare il valore della property.

Il punto chiave è questo:
non stai lavorando con un campo privato.
Stai lavorando con il sistema di proprietà di WPF.

## Perché si chiama “wrapper”

Perché avvolge il meccanismo più tecnico della dependency property in una forma CLR naturale e leggibile.

Questo rende la proprietà molto più ergonomica.

## Cosa non devi fare nel wrapper

Una regola molto importante è:
il wrapper CLR non dovrebbe contenere logiche arbitrarie complicate come se fosse una proprietà normale qualsiasi.

Per molte logiche avanzate, WPF offre strumenti più adatti nei metadati e nelle callback della dependency property.

## Quando il wrapper è particolarmente utile

È utile sempre, ma soprattutto quando:
- vuoi usare la proprietà da codice;
- vuoi mantenere convenzioni standard leggibili;
- vuoi esporre una proprietà naturale del controllo.

## Errore tipico

L'errore classico è pensare che il valore viva nel wrapper.
No: il valore vive nel property system.
Il wrapper è solo il modo pulito di arrivarci.

## Regola pratica

Il wrapper CLR è la porta comoda verso la dependency property.
Il campo statico registra la property, ma il wrapper è ciò che il codice leggerà come proprietà “normale”.
