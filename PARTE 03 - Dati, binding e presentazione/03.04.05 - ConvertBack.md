# 03.04.05 - `ConvertBack`

## La direzione di ritorno del converter

Finora hai visto soprattutto `Convert`, cioè la trasformazione dalla source verso il target.

Ma un converter può anche lavorare nella direzione opposta:
dal target verso la source.

Questo è il ruolo di `ConvertBack`.

## Quando entra in gioco

`ConvertBack` serve quando il binding non è solo di lettura.
Quindi tipicamente quando hai:
- `Mode=TwoWay`;
- un controllo editabile;
- un valore che deve tornare al modello dopo essere stato trasformato.

In altre parole:
`ConvertBack` è rilevante quando la UI non si limita a mostrare il valore, ma lo modifica e lo restituisce.

## Un esempio mentale semplice

Supponiamo che il modello esponga un numero:

```csharp
public int Quantità { get; set; }
```

e che il controllo lavori con una stringa o con una presentazione specifica.
All'andata puoi trasformare il numero in testo.
Al ritorno devi essere in grado di ritrasformare quel testo nel numero corretto.

Questo è il compito di `ConvertBack`.

## Un esempio semplice

```csharp
using System;
using System.Globalization;
using System.Windows.Data;

public class IntToStringConverter : IValueConverter
{
    public object Convert(object value, Type targetType, object parameter, CultureInfo culture)
    {
        if (value is int numero)
            return numero.ToString(culture);

        return "0";
    }

    public object ConvertBack(object value, Type targetType, object parameter, CultureInfo culture)
    {
        if (int.TryParse(value?.ToString(), NumberStyles.Integer, culture, out int risultato))
            return risultato;

        return 0;
    }
}
```

## Come leggere questo esempio

### `Convert`
Trasforma un `int` in una stringa.

### `ConvertBack`
Trasforma la stringa inserita dall'utente di nuovo in `int`.

Questa è una vera conversione bidirezionale.

## Quando `ConvertBack` è davvero utile

È utile quando:
- il binding è `TwoWay`;
- il controllo modifica il dato;
- la source e il target non parlano lo stesso tipo;
- vuoi gestire un ritorno pulito del valore.

## Quando non serve davvero

Non serve quando:
- il binding è `OneWay`;
- il controllo serve solo a visualizzare;
- il ritorno non ha senso;
- il converter è solo di presentazione in avanti.

In questi casi è perfettamente lecito non supportarlo davvero.

Per esempio:

```csharp
public object ConvertBack(object value, Type targetType, object parameter, CultureInfo culture)
{
    throw new NotSupportedException();
}
```

Questa scelta è spesso molto più onesta e leggibile di un finto `ConvertBack` improvvisato.

## Un caso tipico: `bool` → `Visibility`

Nel caso del classico converter `bool` → `Visibility`, `ConvertBack` può esistere, ma non sempre è necessario.
Se stai solo mostrando o nascondendo un elemento in `OneWay`, può anche non servire.

## Errori tipici

### Implementarlo anche quando non ha senso
Il codice si riempie di logica inutile.

### Non gestire gli errori di conversione
Per esempio input testuale che non può tornare nel tipo richiesto.

### Restituire un valore ambiguo senza criterio
Questo può produrre comportamenti strani nel binding.

## Regola pratica

Scrivi `ConvertBack` solo quando il binding deve davvero riportare un valore trasformato alla source.
Se il converter è solo di presentazione, dichiararlo non supportato è spesso la scelta più pulita.
