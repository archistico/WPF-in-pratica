# 03.02.01 - Un solo oggetto nel `DataContext`

## Il caso più semplice e più importante

Il modo più chiaro di usare il `DataContext` è questo:
assegni un solo oggetto alla finestra, o a un contenitore principale, e tutti i binding interni lavorano rispetto a quell'oggetto.

È il punto di partenza migliore perché:
- è facile da leggere;
- è facile da mantenere;
- riduce la confusione;
- rende naturale capire il binding.

Per una grande quantità di schermate iniziali o intermedie, questa impostazione è già più che sufficiente.

## Perché funziona così bene

Quando in una finestra stai lavorando attorno a un solo concetto centrale, avere un solo oggetto nel `DataContext` è molto naturale.

Esempi tipici:
- una scheda cliente;
- una scheda prodotto;
- una finestra impostazioni;
- una schermata dettaglio documento;
- un piccolo form anagrafico.

In tutti questi casi la UI ruota attorno a un singolo oggetto principale.

## Un esempio semplice

Supponiamo di avere questa classe:

```csharp
public class Cliente
{
    public string Nome { get; set; } = "";
    public string Cognome { get; set; } = "";
    public string Città { get; set; } = "";
}
```

Nel code-behind:

```csharp
public partial class MainWindow : Window
{
    public MainWindow()
    {
        InitializeComponent();

        DataContext = new Cliente
        {
            Nome = "Anna",
            Cognome = "Rossi",
            Città = "Torino"
        };
    }
}
```

XAML:

```xml
<Grid Margin="20">
    <Grid.RowDefinitions>
        <RowDefinition Height="Auto"/>
        <RowDefinition Height="Auto"/>
        <RowDefinition Height="Auto"/>
    </Grid.RowDefinitions>

    <TextBox Text="{Binding Nome}" Margin="0,0,0,8"/>
    <TextBox Grid.Row="1" Text="{Binding Cognome}" Margin="0,0,0,8"/>
    <TextBox Grid.Row="2" Text="{Binding Città}"/>
</Grid>
```

Qui la lettura mentale è molto pulita:
- il `DataContext` è un `Cliente`;
- ogni binding cerca una proprietà su quell'oggetto;
- la finestra è semplicemente la rappresentazione visiva del cliente.

## Perché questa soluzione è editorialmente la più sana

Dal punto di vista didattico e progettuale, partire da un solo oggetto nel `DataContext` è quasi sempre la scelta migliore.

Perché?
Perché insegna bene il modello mentale corretto:
- il binding non è un trucco locale;
- la finestra lavora rispetto a una sorgente dati coerente;
- i controlli sono proiezioni di proprietà dello stesso oggetto.

Questa impostazione prepara molto bene anche ai modelli più evoluti.

## Quando è davvero la scelta giusta

Usa un solo oggetto nel `DataContext` quando:
- la schermata ha un centro logico chiaro;
- i dati appartengono allo stesso dominio o alla stessa entità;
- vuoi la massima leggibilità;
- non hai ancora un bisogno reale di strutture più articolate.

## Un piccolo vantaggio pratico

Con un solo oggetto nel `DataContext`, i binding sono più corti:

```xml
<TextBox Text="{Binding Nome}" />
```

invece di qualcosa come:

```xml
<TextBox Text="{Binding ClienteCorrente.Nome}" />
```

Questo non è solo più breve: è anche più leggibile.

## Quando può non bastare

Non basta quando nella stessa finestra hai dati che appartengono a ruoli distinti o a contesti diversi.
Per esempio:
- un oggetto cliente;
- una lista ordini;
- uno stato della UI;
- un pannello laterale che lavora su un altro oggetto.

In quei casi bisogna strutturare meglio il `DataContext` o usare sorgenti esplicite.

## Errore tipico

L'errore non è usare un solo oggetto.
L'errore è **non usarlo quando sarebbe sufficiente**, e complicare la finestra troppo presto con più sorgenti non necessarie.

## Regola pratica

Se la schermata ruota davvero attorno a un solo oggetto, usa quello come `DataContext` principale.
È la soluzione più chiara, più didattica e spesso anche la più elegante.
