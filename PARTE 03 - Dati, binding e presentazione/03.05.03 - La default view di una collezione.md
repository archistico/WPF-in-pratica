# 03.05.03 - La default view di una collezione

## Il modo più naturale di entrare nel modello delle view

Una delle cose più eleganti di WPF è che una collezione usata in binding ha spesso già una vista associata: la cosiddetta **default view**.

Questo significa che, anche senza creare ogni volta una struttura complessa, puoi già ottenere una view della collezione e lavorarci sopra.

È spesso il primo passo migliore per capire il meccanismo.

## Che cos'è la default view

La default view è la vista predefinita che WPF associa a una collezione quando questa viene usata nel data binding.

In pratica:
- hai una collezione reale;
- WPF può fornirti una vista di quella collezione;
- su quella vista puoi applicare ordinamento, filtro e grouping.

## Come si ottiene

La forma più comune è questa:

```csharp
ICollectionView view = CollectionViewSource.GetDefaultView(Clienti);
```

Qui:
- `Clienti` è la collezione originale;
- `view` è la default view.

## Perché è così comoda

Perché ti permette di lavorare subito sulla presentazione dei dati senza dover riscrivere la struttura della collezione o creare meccanismi artificiali.

È una soluzione molto naturale per scenari come:
- ordinamento di una lista;
- filtro in una schermata di ricerca;
- grouping in una griglia o in una lista.

## Un esempio mentale semplice

Hai:

```csharp
public ObservableCollection<Cliente> Clienti { get; } = new();
```

Se la tua `ListBox` usa `ItemsSource="{Binding Clienti}"`, puoi comunque prendere la default view e dire:

- filtrami i clienti attivi;
- ordinali per nome.

Senza toccare la collezione vera.

## Un punto molto importante

La default view non è una cosa completamente scollegata dalla UI.
Spesso è proprio la view che il controllo utilizza dietro le quinte.

Questo rende il comportamento molto naturale:
modifichi la view e il controllo riflette la nuova presentazione.

## Un piccolo esempio

```csharp
var view = CollectionViewSource.GetDefaultView(Clienti);
view.Filter = item => ((Cliente)item).Attivo;
```

Qui stai dicendo:
“la view deve mostrare solo i clienti attivi”.

La collezione `Clienti` resta intatta.
La UI mostrerà solo gli elementi che passano il filtro.

## Quando la default view è la scelta giusta

È ottima quando:
- vuoi lavorare rapidamente su una collezione già bindata;
- ti basta una view associata alla collezione esistente;
- stai gestendo ordinamento e filtro da codice;
- non hai ancora bisogno di una struttura XAML più esplicita con `CollectionViewSource`.

## Quando può non bastare

Se vuoi definire in modo più dichiarativo la view, o usarla come risorsa, o configurarla in modo più strutturato nello XAML, allora `CollectionViewSource` può diventare una scelta migliore.

Ma per capire bene il modello, la default view è un punto di partenza eccellente.

## Errori tipici

### Non sapere che esiste già una view
Si finisce per manipolare direttamente la lista senza necessità.

### Pensare che la default view sia una copia indipendente
Non lo è: è legata alla collezione di origine.

### Modificare la collezione per esigenze di sola presentazione
Quando la default view poteva già risolvere il problema.

## Regola pratica

Prima di creare soluzioni più pesanti, chiediti sempre:
“posso semplicemente usare la default view della collezione?”
Molto spesso, la risposta è sì.
