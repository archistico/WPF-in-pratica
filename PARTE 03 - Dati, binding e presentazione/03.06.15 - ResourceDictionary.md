# 03.06.15 - `ResourceDictionary`

## Quando le risorse iniziano a meritare un loro file

All'inizio puoi mettere risorse in:
- una finestra;
- un `UserControl`;
- `App.xaml`.

Funziona bene, ma solo fino a un certo punto.

Quando le risorse crescono, conviene organizzarle in file dedicati.
Qui entra in gioco `ResourceDictionary`.

## Che cos'è in pratica

`ResourceDictionary` è il contenitore standard delle risorse in WPF.

In un certo senso, tutte le collezioni di risorse che hai già visto sono dizionari di risorse.
La novità qui è che puoi anche metterli in file XAML dedicati, separando meglio l'organizzazione del progetto.

## Perché è così utile

Perché ti permette di:
- togliere peso a `App.xaml`;
- separare colori, stili, template e converter;
- organizzare meglio il progetto;
- rendere più facile manutenzione e riuso.

## Un esempio tipico

File `Brushes.xaml`:

```xml
<ResourceDictionary xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
                    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml">

    <SolidColorBrush x:Key="PrimaryBrush" Color="#2D6CDF"/>
    <SolidColorBrush x:Key="DangerBrush" Color="#C62828"/>

</ResourceDictionary>
```

Questo file non è una finestra e non è un controllo.
È un contenitore puro di risorse.

## Perché è meglio di accumulare tutto in un solo posto

Perché in un progetto reale avrai spesso:
- molti brush;
- molti stili;
- più template;
- converter;
- risorse di supporto.

Mettere tutto in `App.xaml` porta rapidamente a un file difficile da leggere.

Con i dizionari separati puoi creare, per esempio:
- `Colors.xaml`
- `Brushes.xaml`
- `Styles.Buttons.xaml`
- `Styles.TextBox.xaml`
- `Templates.xaml`
- `Converters.xaml`

## Che cosa cambia per l'uso delle risorse

Dal punto di vista del controllo che usa la risorsa, non cambia quasi nulla.
Il vantaggio è soprattutto organizzativo e architetturale.

Continui a usare:
- `StaticResource`
- `DynamicResource`

ma le risorse ora vivono in file dedicati e meglio strutturati.

## Quando introdurlo

Introduci `ResourceDictionary` quando:
- le risorse iniziano a crescere;
- vuoi separare aree diverse di responsabilità;
- il progetto ha più schermate;
- la coerenza visiva merita una struttura più chiara.

## Quando puoi ancora aspettare

In esempi molto piccoli o schermate isolate puoi anche non separare subito tutto.
Ma sapere che `ResourceDictionary` esiste è molto importante, perché è lo strumento naturale quando il progetto evolve.

## Errori tipici

### Continuare a tenere tutto in un unico file enorme
La manutenzione peggiora.

### Spezzare troppo presto in troppi file minuscoli
Anche l'eccesso di frammentazione può confondere.

### Non dare una logica chiara ai file di risorse
Il progetto resta disordinato anche se i file sono separati.

## Regola pratica

Quando le risorse iniziano a diventare una parte seria del progetto, `ResourceDictionary` è il modo giusto per dare loro una casa ordinata e leggibile.
