# 05.02.10 - Risorse XAML localizzabili e `MergedDictionaries`

## Un approccio molto pratico e molto WPF

Oltre al modello classico con file `.resx` e satellite assemblies, in WPF esiste anche una pratica molto utile:
organizzare risorse localizzabili direttamente in XAML, usando dizionari di risorse e `MergedDictionaries`.

Questo approccio è particolarmente interessante quando vuoi localizzare:
- testi UI;
- brush o stili culturali specifici;
- template;
- risorse visive;
- contenuti XAML riusabili.

## Il principio di base

Invece di tenere tutto dentro un unico file o hardcoded nella view, puoi creare dizionari di risorse separati per lingua o cultura.

Per esempio:
- `Strings.it-IT.xaml`
- `Strings.en-US.xaml`

oppure strutture più articolate come:
- `Resources/it-IT/Strings.xaml`
- `Resources/en-US/Strings.xaml`

Il punto è che le risorse testuali o visive vivono in file XAML dedicati e possono essere caricati dinamicamente o in modo organizzato.

## Un esempio semplice di risorsa XAML

File `Strings.it-IT.xaml`:

```xml
<ResourceDictionary xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
                    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml">
    <sys:String x:Key="SaveButtonText"
                xmlns:sys="clr-namespace:System;assembly=mscorlib">Salva</sys:String>
</ResourceDictionary>
```

File `Strings.en-US.xaml`:

```xml
<ResourceDictionary xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
                    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml">
    <sys:String x:Key="SaveButtonText"
                xmlns:sys="clr-namespace:System;assembly=mscorlib">Save</sys:String>
</ResourceDictionary>
```

## Uso nella UI

```xml
<Button Content="{StaticResource SaveButtonText}" />
```

La chiave è sempre la stessa.
Cambia il dizionario caricato.

## Il ruolo di `MergedDictionaries`

`MergedDictionaries` permette di comporre il set di risorse attive.

Per esempio:

```xml
<Application.Resources>
    <ResourceDictionary>
        <ResourceDictionary.MergedDictionaries>
            <ResourceDictionary Source="Resources/Strings.it-IT.xaml"/>
        </ResourceDictionary.MergedDictionaries>
    </ResourceDictionary>
</Application.Resources>
```

In questo modo l'app usa il dizionario di risorse della cultura scelta.

## Perché è utile

Perché è molto naturale in WPF:
- resti nel mondo XAML;
- puoi localizzare non solo stringhe ma anche altre risorse;
- mantieni una struttura coerente con stili e template;
- puoi organizzare la UI in modo modulare e leggibile.

## Quando è particolarmente interessante

È particolarmente interessante quando:
- l'app ha forte componente XAML;
- vuoi localizzare anche risorse visive;
- lavori molto con dizionari di risorse;
- preferisci una pipeline localizzabile molto orientata al markup.

## Che cosa non sostituisce automaticamente

Questo approccio non elimina per magia il ruolo delle risorse compilate o di altri sistemi.
È una scelta progettuale che può convivere o essere usata in certi scenari al posto di altre soluzioni.

L'importante è capire bene il modello:
- chiavi stabili;
- dizionari separati;
- caricamento della variante giusta.

## Un punto molto importante

Localizzare risorse XAML non significa solo tradurre stringhe.
Può significare anche:
- adattare stili;
- cambiare spacing o layout per certe lingue;
- fornire template diversi;
- modificare contenuti visivi per cultura.

Questo è uno dei punti in cui WPF mostra bene la sua potenza.

## Errore tipico

L'errore classico è usare `MergedDictionaries` per localizzare, ma senza una struttura coerente di chiavi e file.
In quel caso il progetto si complica invece di semplificarsi.

## Regola pratica

Se la tua applicazione WPF vive molto di XAML, le risorse localizzabili in `ResourceDictionary` e `MergedDictionaries` sono un approccio molto naturale.
La chiave del successo sta nell'avere:
- chiavi stabili;
- file ben separati per cultura;
- una strategia chiara di caricamento della lingua attiva.
