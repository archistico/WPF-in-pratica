# 05.01.01 - Il binding non mostra nulla

## Uno dei problemi più frequenti in assoluto

Tra tutti gli errori che si incontrano in WPF, questo è probabilmente il più comune:
scrivi un binding che sembra corretto, avvii l'applicazione, ma il controllo resta vuoto.

Per esempio:

```xml
<TextBox Text="{Binding Nome}" />
```

eppure nella UI non compare niente.

Questo è un problema molto formativo, perché costringe a controllare bene i pezzi fondamentali del binding.

## Il primo controllo da fare: esiste davvero il `DataContext`?

La prima domanda da porsi è:
**il controllo ha davvero un `DataContext` valido?**

Se il `DataContext` non esiste, il binding non sa su quale oggetto cercare la proprietà `Nome`.

Esempio corretto:

```csharp
DataContext = new Cliente
{
    Nome = "Anna"
};
```

Se manca questo passaggio, il binding può risultare perfettamente scritto ma non avere nessuna sorgente.

## Il secondo controllo: la proprietà esiste davvero?

La seconda domanda è:
**la proprietà indicata nel binding esiste davvero sull'oggetto corrente?**

Se il `DataContext` è un `Cliente`, allora `Nome` deve essere una proprietà reale del tipo.

Per esempio:

```csharp
public class Cliente
{
    public string Nome { get; set; } = "";
}
```

Se invece hai scritto:
- `Nme`
- `nome`
- `ClienteNome` quando la proprietà si chiama `Nome`

il binding non troverà nulla.

## Il terzo controllo: il contesto è quello che pensi?

Molto spesso il problema non è che manca il `DataContext`, ma che il `DataContext` corrente non è quello che immagini.

Per esempio:

```xml
<StackPanel DataContext="{Binding ClienteCorrente}">
    <TextBox Text="{Binding Nome}" />
</StackPanel>
```

Qui il binding cerca `Nome` su `ClienteCorrente`, non sul view model esterno.

Se hai perso di vista un cambio di contesto locale, il binding può sembrare “misteriosamente rotto”.

## Il quarto controllo: c'è davvero un valore?

Può sembrare banale, ma è utile controllare anche questo:
la proprietà esiste, il binding è corretto, ma il valore è effettivamente valorizzato?

Per esempio:

```csharp
public string Nome { get; set; } = "";
```

Se il valore è stringa vuota, il controllo può apparire semplicemente vuoto senza che il binding sia in errore.

## Un esempio tipico corretto

```csharp
public class Cliente
{
    public string Nome { get; set; } = "";
}

public partial class MainWindow : Window
{
    public MainWindow()
    {
        InitializeComponent();

        DataContext = new Cliente
        {
            Nome = "Anna Rossi"
        };
    }
}
```

```xml
<TextBlock Text="{Binding Nome}" />
```

Qui tutto è allineato:
- esiste il `DataContext`;
- la proprietà `Nome` esiste;
- il valore è presente.

## Perché questo errore è così importante

Perché ti costringe a ragionare in ordine:
1. qual è la source?
2. qual è il contesto corrente?
3. il path è giusto?
4. il valore esiste?

Questa disciplina mentale è utilissima in tutto WPF.

## Un piccolo criterio di debug

Quando un binding non mostra nulla, chiediti sempre:
- il `DataContext` c'è?
- è del tipo giusto?
- il nome della proprietà è corretto?
- il valore non è semplicemente vuoto?
- non c'è un cambio locale di `DataContext` che mi inganna?

## Errore tipico reale

Molto spesso il problema non è il binding in sé, ma il fatto che il controllo si trovi dentro un contenitore che ha cambiato il `DataContext`.
Il lettore guarda la view model “principale”, ma il binding sta cercando su un sotto-oggetto diverso.

## Regola pratica

Quando un binding non mostra nulla, non partire da soluzioni complicate.
Controlla in ordine:
- `DataContext`
- proprietà
- path
- valore
- eventuale cambio di contesto locale

Nella maggior parte dei casi, il problema è lì.
