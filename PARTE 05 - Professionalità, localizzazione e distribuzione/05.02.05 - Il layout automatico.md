# 05.02.05 - Il layout automatico

## La localizzazione non riguarda solo i testi

Uno degli errori più comuni è pensare che localizzare significhi solo sostituire stringhe.

In realtà, quando cambi lingua, cambiano spesso anche:
- la lunghezza dei testi;
- il ritmo visivo dell'interfaccia;
- l'equilibrio tra etichette e campi;
- a volte perfino la direzione di lettura.

Per questo il layout conta moltissimo.

In WPF il concetto chiave è progettare un layout **adattabile**, non rigido.

## Che cosa significa layout automatico

Qui per layout automatico intendiamo un layout che si adatta in modo naturale al contenuto, invece di dipendere da misure fisse troppo rigide.

In pratica:
- contenitori che si allungano quando serve;
- colonne dimensionate bene;
- margini e spaziature coerenti;
- elementi che non si spezzano appena cambia la lingua.

Questo è molto importante in UI localizzabili.

## Perché WPF aiuta molto

WPF ha un sistema di layout molto più flessibile di altri modelli storici di interfaccia.

Con controlli e pannelli come:
- `Grid`
- `StackPanel`
- `DockPanel`
- `WrapPanel`

puoi progettare schermate che reagiscono meglio al contenuto.

Ma devi farlo consapevolmente.

## Un esempio da evitare

```xml
<Button Width="80" Content="Save" />
```

Questo può funzionare in inglese.
Ma se in un'altra lingua il testo è più lungo, il bottone può risultare:
- stretto;
- tagliato;
- visivamente sgradevole.

## Una soluzione più sana

```xml
<Button MinWidth="80"
        Padding="12,6"
        Content="{x:Static properties:Resources.SaveButtonText}" />
```

Qui il pulsante ha un minimo, ma non è completamente bloccato.
Il layout ha più respiro.

## Il ruolo delle colonne `Auto` e `*`

In una `Grid`, una progettazione sana può usare:
- `Auto` per contenuti che devono occupare lo spazio necessario;
- `*` per aree elastiche.

Per esempio:

```xml
<Grid.ColumnDefinitions>
    <ColumnDefinition Width="Auto"/>
    <ColumnDefinition Width="*"/>
</Grid.ColumnDefinitions>
```

Questa soluzione è molto più adatta a UI localizzabili rispetto a larghezze fisse arbitrarie.

## Perché i layout troppo rigidi soffrono tanto

Se usi:
- `Width` fissi ovunque;
- posizionamenti troppo rigidi;
- controlli compressi;
- allineamenti pensati solo per una lingua;

basta una traduzione un po' più lunga per rompere l'equilibrio della schermata.

## Un esempio mentale utile

Una label italiana può essere:
- “Nome”

la stessa label in tedesco o in altre lingue può occupare molto più spazio.

Se il layout non prevede questa possibilità, l'interfaccia si degrada subito.

## Il layout automatico è anche una scelta di professionalità

Una UI pronta alla localizzazione non è solo traducibile.
È anche progettata per sopportare bene la traduzione.

Questo è uno dei segnali più chiari di maturità progettuale.

## Quando fare attenzione

Soprattutto in:
- toolbar;
- form con etichette e campi;
- pulsanti stretti;
- finestre piccole;
- intestazioni di colonne;
- menu e comandi molto compatti.

Sono i punti in cui i testi più lunghi si fanno sentire subito.

## Errore tipico

L'errore classico è costruire la UI perfetta per una lingua sola, con misure troppo rigide.
Poi, quando cambia il testo, il problema viene scambiato per “difetto della traduzione”, mentre in realtà è un difetto del layout.

## Regola pratica

Se la UI deve essere localizzata, progetta layout elastici.
In WPF la regola sana è: lasciare spazio al contenuto, invece di costringerlo in misure pensate solo per una lingua.
