# 05.03.05 - Come mostrarla all’avvio

## Il punto tecnico che trasforma l'idea in flusso reale

Dopo aver deciso di usare una finestra splash personalizzata, la domanda pratica diventa:
come la mostro davvero all'avvio dell'applicazione?

Qui il punto importante non è solo “aprire una window”.
È costruire bene il flusso di startup:
- mostrare lo splash;
- eseguire l'inizializzazione;
- poi aprire la finestra principale;
- infine chiudere lo splash nel momento giusto.

Questo va fatto con ordine.

## Un approccio semplice e chiaro

Una strategia molto leggibile consiste nel gestire l'avvio in `App.xaml.cs`, togliendo `StartupUri` da `App.xaml` e prendendo il controllo manuale dello startup.

## `App.xaml`

```xml
<Application x:Class="MioProgetto.App"
             xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml">
    <Application.Resources>
    </Application.Resources>
</Application>
```

Qui non usiamo `StartupUri`.

## `App.xaml.cs`

```csharp
using System.Windows;

public partial class App : Application
{
    protected override async void OnStartup(StartupEventArgs e)
    {
        base.OnStartup(e);

        var splash = new SplashWindow();
        splash.Show();

        await Task.Delay(2000); // simulazione inizializzazione

        var mainWindow = new MainWindow();
        MainWindow = mainWindow;
        mainWindow.Show();

        splash.Close();
    }
}
```

## Come leggere questo esempio

### 1. Lo splash viene mostrato subito
La finestra iniziale compare prima della `MainWindow`.

### 2. L'inizializzazione avviene durante lo splash
Qui è simulata con `Task.Delay`, ma nella pratica può essere:
- caricamento configurazioni;
- inizializzazione servizi;
- lettura dati;
- setup iniziale.

### 3. Si apre la finestra principale
Quando il lavoro essenziale è finito.

### 4. Si chiude lo splash
Solo a quel punto la transizione è completa.

Questo è il flusso corretto di base.

## Perché togliere `StartupUri`

Perché `StartupUri` è il modello più semplice, ma qui hai bisogno di controllare tu:
- l'ordine di visualizzazione;
- il momento di apertura della `MainWindow`;
- il ciclo di vita dello splash.

Quindi la gestione esplicita in `OnStartup` è la soluzione più naturale.

## Un punto molto importante

La parte di inizializzazione deve essere progettata in modo da non bloccare inutilmente la UI.
Se fai lavoro lungo in modo sincrono e bloccante nello startup, puoi avere uno splash che compare male o una UI che sembra congelata.

Per questo il tema si collega anche al capitolo sul multithreading.

## Un criterio pratico

Mostra lo splash prima di fare l'inizializzazione visibile all'utente.
Poi esegui il minimo lavoro necessario per portare l'app in uno stato iniziale corretto.
Poi apri la finestra principale e chiudi lo splash.

Questo è il ritmo corretto.

## Quando può servire qualcosa di più strutturato

Se hai un bootstrap più articolato, potresti anche:
- aggiornare messaggi nello splash;
- mostrare progresso;
- separare fasi di inizializzazione;
- usare un view model dedicato allo splash.

Ma il modello base resta questo:
mostra → inizializza → apri → chiudi.

## Errore tipico

L'errore classico è aprire lo splash e poi bloccare subito il thread UI con lavoro sincrono pesante.
In quel caso lo splash esiste “sulla carta”, ma l'esperienza reale resta cattiva.

## Regola pratica

Per mostrare uno splash personalizzato all'avvio, la strada più chiara è gestire lo startup manualmente in `App.xaml.cs`.
Così controlli bene quando compare, quando resta visibile e quando lascia il posto alla finestra principale.
