# 05.04.05 - Framework-dependent o self-contained

## La scelta trasversale che cambia molto il deployment

Oltre a scegliere tra cartella, ClickOnce o MSIX, c'è una distinzione molto importante che attraversa il deployment .NET:

- **framework-dependent**
- **self-contained**

Questa scelta influisce su:
- che cosa pubblichi;
- che cosa deve avere la macchina target;
- dimensione dell'output;
- semplicità di esecuzione sul PC di destinazione. citeturn333254search3turn333254search6turn333254search10turn333254search18

## Framework-dependent

In modalità framework-dependent, l'app richiede che sul computer di destinazione sia installato un runtime .NET compatibile. La documentazione del comando `dotnet publish` descrive esplicitamente l'opzione `--no-self-contained` per questo scenario. citeturn333254search6turn333254search18

### Vantaggi
- output più leggero;
- distribuzione più piccola;
- meno runtime incluso nei file pubblicati. citeturn333254search6turn333254search18

### Svantaggio pratico
- devi assicurarti che il target abbia già il runtime corretto.

## Self-contained

In modalità self-contained, il runtime necessario viene distribuito insieme all'app. La documentazione di publishing .NET descrive questa modalità come una delle opzioni standard di deployment, e il single-file può essere prodotto sia in scenari self-contained sia framework-dependent in base alle opzioni scelte. citeturn333254search3turn333254search10

### Vantaggi
- meno dipendenza dal fatto che il target abbia .NET già installato;
- distribuzione più autonoma verso macchine eterogenee. citeturn333254search3turn333254search10

### Svantaggi
- output più grande;
- pacchetto più pesante.

## Un punto molto importante aggiornato

Dalla compatibilità SDK .NET 8, un'app runtime-specific non è più automaticamente self-contained solo perché ha un `RuntimeIdentifier`; per default resta framework-dependent, salvo scelta esplicita. Questo è un dettaglio molto importante perché in passato molti davano per scontato il contrario. citeturn333254search18

## Il legame con il single-file

La documentazione Microsoft sul single-file spiega che puoi configurare:
- deployment mode (`Self-contained` o `Framework-dependent`);
- target runtime;
- opzione “Produce single file”. citeturn333254search10

Questo significa che “single file” non va confuso automaticamente con “self-contained”, anche se spesso le due scelte vengono abbinate.

## Come scegliere in pratica

### Framework-dependent
Ha senso quando:
- il target è controllato;
- sai che il runtime giusto è già presente o può essere gestito facilmente;
- vuoi output più leggero.

### Self-contained
Ha senso quando:
- vuoi ridurre il più possibile i prerequisiti sul PC utente;
- distribuisci a macchine dove non controlli il runtime;
- accetti un output più grande in cambio di maggiore autonomia.

## Regola pratica

La domanda giusta è:
“posso dare per scontato il runtime giusto sul PC di destinazione?”
Se sì, framework-dependent è spesso più leggero.
Se no, self-contained è spesso più sicuro operativamente.
