# 05.01.02 - La UI non si aggiorna

## Il secondo grande classico di WPF

Un altro errore frequentissimo è questo:
il binding iniziale funziona, il valore compare correttamente, ma quando il dato cambia la UI non si aggiorna.

Questo è un problema molto tipico e molto istruttivo, perché porta quasi sempre a una domanda decisiva:
**chi sta notificando il cambiamento?**

## Il caso tipico

Supponiamo di avere:

```csharp
public class Cliente
{
    public string Nome { get; set; } = "";
}
```

e nella UI:

```xml
<TextBlock Text="{Binding Nome}" />
```

All'avvio vedi `"Anna"`, quindi pensi:
“il binding funziona”.

Poi esegui:

```csharp
cliente.Nome = "Luca";
```

ma il `TextBlock` resta fermo su `"Anna"`.

## Perché succede

Perché il binding iniziale ha letto il valore, ma l'oggetto non ha notificato a WPF che la proprietà è cambiata.

Qui entra in gioco `INotifyPropertyChanged`.

## La soluzione corretta

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;

public class Cliente : INotifyPropertyChanged
{
    private string _nome = "";

    public string Nome
    {
        get => _nome;
        set
        {
            if (_nome != value)
            {
                _nome = value;
                OnPropertyChanged();
            }
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

Ora, quando `Nome` cambia, la UI può aggiornarsi.

## Il problema reale non è il binding iniziale

Questo punto va capito molto bene:
se la UI mostra il valore iniziale, il binding non è “rotto”.
Il problema è che l'oggetto non sta collaborando col sistema di aggiornamento dinamico.

Questa distinzione è molto utile per diagnosticare bene.

## Un altro caso tipico: `CanExecute` o stato derivato che non si aggiorna

A volte non è solo un `TextBlock` a non aggiornarsi.
Può essere:
- un pulsante che non cambia abilitazione;
- un totale che non si aggiorna;
- un'etichetta che resta indietro.

Anche qui la causa è spesso la stessa:
qualcosa è cambiato, ma non è stato notificato bene.

## Un piccolo test mentale molto utile

Se la UI mostra bene il valore iniziale ma non i cambi successivi, pensa subito a:
- `INotifyPropertyChanged` mancante;
- `OnPropertyChanged()` dimenticato;
- property name sbagliato;
- proprietà calcolate che non vengono notificate.

## Il caso delle proprietà calcolate

Per esempio:

```csharp
public string NomeCompleto => Nome + " " + Cognome;
```

Se cambia `Nome`, ma non notifichi anche `NomeCompleto`, la UI che dipende da `NomeCompleto` non si aggiorna.

Questo è un errore molto reale e molto frequente.

## Un caso ancora più concreto

Se hai:

```xml
<TextBlock Text="{Binding NomeCompleto}" />
```

e modifichi solo `Nome`, allora devi notificare:
- `Nome`
- `NomeCompleto`

altrimenti il binding sul valore derivato resta fermo.

## Errore tipico reale

Il lettore implementa `INotifyPropertyChanged`, ma si dimentica la chiamata a `OnPropertyChanged()` in un setter.
Formalmente “sembra tutto giusto”, ma la UI non reagisce.

## Regola pratica

Se la UI mostra il valore iniziale ma non si aggiorna dopo, la prima ipotesi da controllare è quasi sempre:
manca o è incompleta la notifica del cambiamento della proprietà.
