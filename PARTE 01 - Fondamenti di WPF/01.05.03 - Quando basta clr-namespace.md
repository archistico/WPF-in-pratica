# 01.05.03 - Quando basta `clr-namespace`

Molto spesso, quando lavori in un normale progetto WPF, il tipo che vuoi usare nello XAML si trova nello stesso progetto in cui stai già lavorando. In questo caso, di solito basta usare `clr-namespace`.

Per esempio:

```xml
xmlns:local="clr-namespace:LibroWpf.Views"
```

Con questa dichiarazione stai dicendo a XAML che il prefisso `local:` corrisponde al namespace CLR `LibroWpf.Views`.

Se dentro quel namespace hai un `UserControl` chiamato `SchedaCliente`, potrai usarlo così:

```xml
<local:SchedaCliente />
```

Questo è il caso più comune quando:

- usi `UserControl` creati nello stesso progetto;
- usi converter definiti nella stessa applicazione;
- hai classi di supporto che vuoi referenziare nello XAML.

In pratica, se il tipo è nello stesso assembly del file XAML, `clr-namespace:` è spesso tutto quello che ti serve.

## Un esempio concreto

Immagina di avere questa classe:

```csharp
namespace LibroWpf.Converters
{
    public class BoolToTextConverter
    {
    }
}
```

Nel file XAML puoi dichiarare:

```xml
xmlns:conv="clr-namespace:LibroWpf.Converters"
```

Da lì in poi potrai usare `conv:` per referenziare i tipi di quel namespace.

## Errore tipico

Molti principianti pensano che `local` abbia un significato speciale imposto da WPF. Non è così. `local` è solo un alias molto comune. Potresti usare anche `views`, `conv`, `ui` o un altro nome sensato.

## Cosa portare a casa

Quando il tipo che vuoi usare sta nello stesso progetto, nella maggior parte dei casi basta `clr-namespace:...`. È la forma più semplice e più frequente del mapping tra XAML e codice.
