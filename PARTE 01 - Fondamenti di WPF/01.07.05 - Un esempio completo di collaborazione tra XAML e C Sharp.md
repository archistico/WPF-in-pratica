# 01.07.05 - Un esempio completo di collaborazione tra XAML e C#

Mettiamo insieme i pezzi visti fin qui in un esempio piccolo ma realistico.

`MainWindow.xaml`:

```xml
<Window x:Class="EsempioWpf.MainWindow"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        Title="Saluto" Height="180" Width="320">
    <StackPanel Margin="16">
        <TextBox x:Name="NomeTextBox" Margin="0,0,0,8" />
        <Button Content="Saluta" Click="SalutaButton_Click" Margin="0,0,0,8" />
        <TextBlock x:Name="RisultatoTextBlock" />
    </StackPanel>
</Window>
```

`MainWindow.xaml.cs`:

```csharp
namespace EsempioWpf;

public partial class MainWindow : Window
{
    public MainWindow()
    {
        InitializeComponent();
    }

    private void SalutaButton_Click(object sender, RoutedEventArgs e)
    {
        string nome = NomeTextBox.Text.Trim();

        if (string.IsNullOrWhiteSpace(nome))
        {
            RisultatoTextBlock.Text = "Inserisci un nome.";
            return;
        }

        RisultatoTextBlock.Text = $"Ciao, {nome}!";
    }
}
```

Questo esempio mostra bene la collaborazione tra i due livelli:

- lo XAML definisce la struttura dell'interfaccia;
- `x:Name` rende accessibili alcuni controlli dal codice;
- l'evento `Click` collega il pulsante a un metodo C#;
- il code-behind gestisce la reazione dell'interfaccia.

È un esempio semplice, ma già corretto come modello mentale. Non stai scrivendo tutta la UI in C#, e non stai neppure chiedendo allo XAML di fare ciò che spetta alla logica. Ogni parte fa il suo lavoro.

Questo è il modo sano di iniziare con WPF: usare il markup per dichiarare l'interfaccia, e il code-behind per gestire interazioni immediate e codice strettamente legato alla vista.
