# 01.07.07 - Quando il code-behind inizia a diventare un problema

Il code-behind diventa problematico quando smette di gestire la vista e comincia a inglobare troppe responsabilità diverse.

I segnali classici sono questi:

- nel file `.xaml.cs` finiscono regole di business;
- la finestra interroga direttamente il database;
- la logica applicativa è sparsa in molti gestori di eventi;
- il codice è difficile da riusare e da testare;
- ogni modifica alla UI rischia di rompere parti importanti dell'applicazione.

Per esempio, un gestore come questo è già un campanello d'allarme:

```csharp
private void SalvaButton_Click(object sender, RoutedEventArgs e)
{
    using var connessione = new SqlConnection(connectionString);
    connessione.Open();

    var comando = new SqlCommand("INSERT INTO Clienti ...", connessione);
    comando.ExecuteNonQuery();

    if (NomeTextBox.Text.Length < 3)
    {
        MessageBox.Show("Nome non valido");
    }

    AggiornaStatisticheGenerali();
}
```

Qui nello stesso punto si mescolano UI, validazione, accesso ai dati e logica applicativa. In un esempio didattico brevissimo può capitare; in un progetto reale diventa presto difficile da mantenere.

Questo non significa che ogni riga di code-behind sia sbagliata. Significa che bisogna riconoscere il momento in cui il file della vista sta prendendo in carico troppo lavoro.

L'errore tipico del principiante non è usare il code-behind: è continuare a metterci tutto, anche quando l'applicazione cresce.
