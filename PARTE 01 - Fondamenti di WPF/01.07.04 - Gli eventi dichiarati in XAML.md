# 01.07.04 - Gli eventi dichiarati in XAML

Uno dei modi più immediati per collegare XAML e C# è dichiarare un evento direttamente nel markup.

Esempio:

```xml
<Button Content="Calcola"
        Click="CalcolaButton_Click" />
```

Nel file `MainWindow.xaml.cs` puoi poi scrivere il metodo corrispondente:

```csharp
private void CalcolaButton_Click(object sender, RoutedEventArgs e)
{
    MessageBox.Show("Hai fatto clic sul pulsante.");
}
```

Questo meccanismo è molto comodo, soprattutto nei primi esempi o nelle interfacce semplici. Lo XAML dichiara che cosa deve succedere a livello di interazione, mentre il code-behind contiene l'azione concreta da eseguire.

Il vantaggio è la chiarezza iniziale: vedi subito quale controllo reagisce a quale evento.

C'è però una regola pratica importante: gli eventi dichiarati in XAML sono utili, ma non devono trasformarsi nel contenitore di tutta la logica dell'applicazione. Vanno bene per la gestione dell'interazione con la UI; diventano problematici quando dentro il gestore inizi a mettere regole di business, accesso ai dati, validazioni complesse o logiche riusabili.

Un altro errore tipico è sbagliare il nome del metodo o la firma del gestore. In questi casi puoi ottenere errori di compilazione oppure problemi nel caricamento dell'interfaccia.

La firma attesa, per un `Click`, è normalmente questa:

```csharp
private void CalcolaButton_Click(object sender, RoutedEventArgs e)
{
}
```

Nei capitoli iniziali questa tecnica è perfettamente lecita perché rende visibile il legame tra markup e codice. Più avanti vedremo che non tutto deve passare per gli eventi nel code-behind, ma per partire è un ponte molto naturale.
