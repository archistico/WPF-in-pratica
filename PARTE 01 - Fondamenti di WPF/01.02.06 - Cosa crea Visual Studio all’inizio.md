## 01.02.06 - Cosa crea Visual Studio all’inizio

Quando Visual Studio genera un nuovo progetto WPF, non crea solo una finestra vuota. Costruisce una piccola struttura di partenza, già pronta per compilare ed eseguire l’applicazione.

Per chi inizia, è importante capire una cosa: quei file iniziali non sono rumore. Sono la forma minima di un’app WPF funzionante.

Di solito, appena nasce il progetto, trovi elementi come questi:

- il file di progetto;
- `App.xaml`;
- `App.xaml.cs`;
- `MainWindow.xaml`;
- `MainWindow.xaml.cs`;
- cartelle e file di supporto generati dall’ambiente.

A seconda del template, della versione e delle opzioni scelte, la struttura può variare leggermente. Ma l’idea resta la stessa: Visual Studio prepara il minimo indispensabile perché l’app possa partire.

## La logica di questa struttura iniziale

Già qui si intravede una separazione molto importante in WPF:

- file XAML per descrivere la UI;
- file C# per il comportamento;
- file di progetto per definire come l’app viene compilata;
- file applicativi per il punto di ingresso e le risorse generali.

È una divisione che ritroverai continuamente nel resto del libro.

## Perché conviene non cancellare tutto subito

Molti, appena vedono il progetto generato, hanno la tentazione di ripulire, rinominare o riscrivere tutto. È comprensibile, ma all’inizio conviene prima capire il ruolo di ogni pezzo.

Quando conosci bene la struttura standard, puoi modificarla con criterio. Prima no.

### Errore tipico

Aprire il progetto, vedere tanti file, considerarli “generati automaticamente” e quindi ignorarli. In realtà sono parte del modello base di lavoro in WPF.

### Regola pratica

I file iniziali del progetto non vanno memorizzati a memoria, ma vanno capiti. Ogni file esiste per una ragione precisa.
