# 06.04.01 - Modelli

## Il primo passo del codice concreto

Dopo aver progettato il caso d'uso, il passaggio naturale è iniziare dall'implementazione dei modelli.

È una scelta corretta sia dal punto di vista tecnico sia da quello didattico:
prima definisci gli oggetti del dominio, poi costruisci sopra di loro il comportamento della UI.

Nel nostro caso finale, i modelli restano quelli già individuati in fase di progettazione:
- `Tavolo`
- `Piatto`
- `RigaConto`

Qui però li portiamo in una forma concreta e pronta a lavorare bene con WPF.

## Perché conviene renderli già adatti al binding

In teoria un modello potrebbe essere anche una semplice classe dati.
In pratica, in un caso WPF come questo, è molto utile che i modelli collaborino bene con il binding.

Questo significa che, quando serve, conviene implementare:
- `INotifyPropertyChanged`
- proprietà derivate leggibili
- strutture semplici ma vive

Così la UI può aggiornarsi in modo naturale quando lo stato cambia.

## Il modello `Piatto`

Partiamo dal modello più semplice:
il piatto.

Rappresenta un articolo ordinabile e, per il nostro caso didattico, ha poche proprietà essenziali:
- identificativo;
- nome;
- prezzo;
- categoria.

### Implementazione

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;

namespace CassaRistorante.Models
{
    public class Piatto : INotifyPropertyChanged
    {
        private int _id;
        private string _nome = "";
        private decimal _prezzo;
        private string _categoria = "";

        public int Id
        {
            get => _id;
            set
            {
                if (_id != value)
                {
                    _id = value;
                    OnPropertyChanged();
                }
            }
        }

        public string Nome
        {
            get => _nome;
            set
            {
                if (_nome != value)
                {
                    _nome = value;
                    OnPropertyChanged();
                }
            }
        }

        public decimal Prezzo
        {
            get => _prezzo;
            set
            {
                if (_prezzo != value)
                {
                    _prezzo = value;
                    OnPropertyChanged();
                }
            }
        }

        public string Categoria
        {
            get => _categoria;
            set
            {
                if (_categoria != value)
                {
                    _categoria = value;
                    OnPropertyChanged();
                }
            }
        }

        public event PropertyChangedEventHandler? PropertyChanged;

        protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
        {
            PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
        }
    }
}
```

## Perché `Piatto` va tenuto così semplice

Perché il suo ruolo è molto chiaro:
rappresentare ciò che si può ordinare.

Se in questa fase lo appesantisci con:
- gestione magazzino;
- IVA;
- cucina;
- disponibilità per fasce orarie;
- promozioni;

il caso finale perde leggibilità.

Qui l'obiettivo è avere un modello pulito, comprensibile e subito utile alla schermata.

## Il modello `RigaConto`

`RigaConto` è il modello più interessante del caso finale, perché rende operativo il conto.

Collega infatti:
- un piatto;
- una quantità;
- un prezzo unitario;
- un subtotale.

### Implementazione

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;

namespace CassaRistorante.Models
{
    public class RigaConto : INotifyPropertyChanged
    {
        private Piatto _piatto = new();
        private int _quantita = 1;

        public Piatto Piatto
        {
            get => _piatto;
            set
            {
                if (_piatto != value)
                {
                    _piatto = value;
                    OnPropertyChanged();
                    OnPropertyChanged(nameof(NomePiatto));
                    OnPropertyChanged(nameof(PrezzoUnitario));
                    OnPropertyChanged(nameof(Subtotale));
                }
            }
        }

        public int Quantita
        {
            get => _quantita;
            set
            {
                if (_quantita != value)
                {
                    _quantita = value;
                    OnPropertyChanged();
                    OnPropertyChanged(nameof(Subtotale));
                }
            }
        }

        public string NomePiatto => Piatto.Nome;

        public decimal PrezzoUnitario => Piatto.Prezzo;

        public decimal Subtotale => PrezzoUnitario * Quantita;

        public event PropertyChangedEventHandler? PropertyChanged;

        protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
        {
            PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
        }
    }
}
```

## Perché qui compaiono proprietà derivate

`NomePiatto`, `PrezzoUnitario` e `Subtotale` sono proprietà derivate.
Questo è molto importante, perché mostra una delle forze di WPF:
la UI non deve ricevere tutto “già impaginato”.
Può lavorare molto bene anche con proprietà calcolate in modo ordinato.

In più, questo modello rende molto naturale il binding di una griglia o di una lista ordine.

## Il modello `Tavolo`

Il tavolo è il contenitore logico dell'ordine.
È il contesto di lavoro attivo per l'operatore.

Nel nostro caso finale gli facciamo contenere:
- identificativo;
- nome;
- raccolta delle righe;
- stato derivato;
- totale derivato.

### Implementazione

```csharp
using System.Collections.ObjectModel;
using System.Collections.Specialized;
using System.ComponentModel;
using System.Linq;
using System.Runtime.CompilerServices;

namespace CassaRistorante.Models
{
    public class Tavolo : INotifyPropertyChanged
    {
        private int _id;
        private string _nome = "";
        private ObservableCollection<RigaConto> _righeConto = new();

        public Tavolo()
        {
            _righeConto.CollectionChanged += RigheConto_CollectionChanged;
        }

        public int Id
        {
            get => _id;
            set
            {
                if (_id != value)
                {
                    _id = value;
                    OnPropertyChanged();
                }
            }
        }

        public string Nome
        {
            get => _nome;
            set
            {
                if (_nome != value)
                {
                    _nome = value;
                    OnPropertyChanged();
                }
            }
        }

        public ObservableCollection<RigaConto> RigheConto
        {
            get => _righeConto;
            set
            {
                if (_righeConto != value)
                {
                    _righeConto.CollectionChanged -= RigheConto_CollectionChanged;
                    _righeConto = value;
                    _righeConto.CollectionChanged += RigheConto_CollectionChanged;

                    OnPropertyChanged();
                    OnPropertyChanged(nameof(IsOccupato));
                    OnPropertyChanged(nameof(Totale));
                }
            }
        }

        public bool IsOccupato => RigheConto.Any();

        public decimal Totale => RigheConto.Sum(r => r.Subtotale);

        private void RigheConto_CollectionChanged(object? sender, NotifyCollectionChangedEventArgs e)
        {
            OnPropertyChanged(nameof(IsOccupato));
            OnPropertyChanged(nameof(Totale));
        }

        public event PropertyChangedEventHandler? PropertyChanged;

        protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
        {
            PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
        }
    }
}
```

## Perché `Tavolo` contiene una `ObservableCollection<RigaConto>`

Perché qui stiamo modellando un contenitore operativo vivo.
Ogni tavolo non è solo un'etichetta.
Ha un conto che può:
- ricevere righe;
- perdere righe;
- cambiare totale;
- cambiare stato occupato/libero.

Questo rende il modello perfetto per una schermata WPF bindata.

## Un punto importante sui limiti di questo modello

Questa implementazione è molto buona per il caso finale del libro, ma non pretende di coprire ogni scenario reale.

Per esempio, in un progetto più avanzato potresti voler:
- separare `Ordine` da `Tavolo`;
- gestire stati più complessi;
- introdurre repository o persistenza;
- usare model più neutrali e view model più ricchi.

Ma per una implementazione didattica chiara, questa struttura è ottima.

## Che cosa ci portiamo a casa da questi modelli

Con questi tre modelli abbiamo già in mano:
- il contesto operativo (`Tavolo`);
- l'articolo ordinabile (`Piatto`);
- l'unità concreta del conto (`RigaConto`).

E soprattutto abbiamo modelli già pronti a dialogare bene con il binding WPF.

## Regola pratica

Nell'implementazione del caso finale, i modelli devono essere abbastanza semplici da restare leggibili, ma abbastanza vivi da sostenere binding, proprietà derivate e aggiornamento immediato della UI.
