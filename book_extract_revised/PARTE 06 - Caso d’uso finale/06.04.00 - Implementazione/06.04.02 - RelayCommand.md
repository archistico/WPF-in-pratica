# 06.04.02 - `RelayCommand`

## Il ponte pratico tra UI e operazioni

Una schermata come la cassa del ristorante vive di azioni ripetute:
- aggiungere un piatto;
- rimuovere una riga;
- aumentare quantità;
- diminuire quantità;
- chiudere il conto.

Per questo, il caso finale è un ottimo posto per usare davvero i comandi WPF in modo moderno.

E la forma più pratica e didatticamente chiara per farlo è `RelayCommand`.

## Perché qui non conviene usare solo eventi `Click`

Potremmo anche collegare ogni pulsante a un handler `Click` nel code-behind.
Ma così:
- la logica si sparge nella view;
- l'abilitazione/disabilitazione delle azioni diventa più manuale;
- la schermata si allontana dal modello MVVM.

Nel caso finale vogliamo invece mostrare una struttura più pulita:
la UI espone azioni, e il view model le governa.

## Il ruolo di `RelayCommand`

`RelayCommand` è una classe di supporto che implementa `ICommand` e permette di:
- definire che cosa fa un comando;
- definire quando il comando è disponibile;
- evitare di creare una classe separata per ogni singola azione.

È quindi un ottimo compromesso tra chiarezza e praticità.

## Implementazione

```csharp
using System;
using System.Windows.Input;

namespace CassaRistorante.Commands
{
    public class RelayCommand : ICommand
    {
        private readonly Action<object?> _execute;
        private readonly Func<object?, bool>? _canExecute;

        public RelayCommand(Action<object?> execute, Func<object?, bool>? canExecute = null)
        {
            _execute = execute;
            _canExecute = canExecute;
        }

        public event EventHandler? CanExecuteChanged;

        public bool CanExecute(object? parameter)
        {
            return _canExecute?.Invoke(parameter) ?? true;
        }

        public void Execute(object? parameter)
        {
            _execute(parameter);
        }

        public void RaiseCanExecuteChanged()
        {
            CanExecuteChanged?.Invoke(this, EventArgs.Empty);
        }
    }
}
```

## Come leggere questa implementazione

### `_execute`
Contiene l'azione da eseguire quando il comando parte.

### `_canExecute`
Contiene la logica che stabilisce se il comando è attualmente disponibile.

### `CanExecuteChanged`
Permette alla UI di rivalutare l'abilitazione dei controlli collegati.

### `RaiseCanExecuteChanged()`
È il metodo che useremo quando un cambiamento di stato può influenzare i comandi.

## Perché questa implementazione è adatta al caso finale

Perché il caso finale ha azioni molto chiare e relativamente locali al view model:
- aggiungi;
- rimuovi;
- incrementa;
- decrementa;
- chiudi.

Queste azioni si prestano benissimo a essere espresse come lambda nel costruttore del view model.

Questo rende il codice:
- leggibile;
- concentrato;
- semplice da seguire per il lettore.

## Un esempio di uso molto semplice

```csharp
public ICommand AggiungiPiattoCommand { get; }

public MioViewModel()
{
    AggiungiPiattoCommand = new RelayCommand(_ => AggiungiPiatto());
}
```

Questo è già molto più pulito di un pulsante con logica dispersa.

## Il ruolo di `CanExecute` nel caso finale

Il nostro caso d'uso è perfetto per far vedere perché `CanExecute` è utile.

Per esempio:
- non puoi aggiungere un piatto se non c'è un tavolo selezionato;
- non puoi rimuovere una riga se non c'è riga selezionata;
- non puoi chiudere un conto se non ci sono righe.

Queste regole sono molto più eleganti se vivono nei comandi.

## Esempio con `CanExecute`

```csharp
RimuoviRigaCommand = new RelayCommand(
    _ => RimuoviRiga(),
    _ => RigaSelezionata is not null);
```

Qui la UI può abilitare o disabilitare il pulsante senza logiche manuali sparse.

## Perché `RaiseCanExecuteChanged()` è importante

Se cambia lo stato della schermata, per esempio:
- cambia il tavolo selezionato;
- cambia la riga selezionata;
- cambia il numero di righe;

potresti dover dire ai comandi di rivalutare `CanExecute`.

Ed è qui che `RaiseCanExecuteChanged()` diventa prezioso.

## Un piccolo criterio utile

Nel caso finale, `RelayCommand` non è solo una comodità.
È anche un modo per far vedere che i comandi:
- rappresentano azioni dell'applicazione;
- non sono semplici click travestiti;
- danno ordine alla schermata.

Questo è un messaggio molto importante da consolidare alla fine del libro.

## Regola pratica

Nel caso d'uso finale, `RelayCommand` è la soluzione più naturale per esporre le azioni della schermata in modo pulito, bindabile e coerente con MVVM.
