# 06.07.05 - Introdurre i comandi e ripulire il code-behind

## Il momento in cui il progetto diventa davvero più ordinato

A questo punto del caso finale abbiamo già una schermata che:
- si apre;
- mostra i tavoli;
- mostra i piatti;
- costruisce il conto;
- aggiorna quantità e totale.

Ma c'è una domanda importante da chiarire:
dove vive davvero la logica di questa schermata?

Questo è il punto in cui il progetto smette di essere solo funzionante e comincia a diventare strutturato bene.

## Il problema del code-behind pieno

In una prima versione rapida, sarebbe facile scrivere tutto nel code-behind:
- click dei pulsanti;
- aggiunta dei piatti;
- modifica delle quantità;
- reset del conto.

Funzionerebbe.
Ma il progetto diventerebbe presto:
- più difficile da leggere;
- più accoppiato alla view;
- meno coerente con il resto del percorso del libro.

Per questo, in questo punto della costruzione guidata, ha molto senso ripulire il flusso e consolidare i comandi.

## Il ruolo del `RelayCommand`

Il `RelayCommand` serve proprio a questo:
trasformare le azioni operative della schermata in oggetti bindabili che vivono nel `ViewModel`.

In questo modo:
- la view invoca;
- il `ViewModel` decide;
- il code-behind resta pulito.

## Il code-behind finale deve restare minimale

Una forma molto sana del code-behind è questa:

```csharp
using System.Windows;
using CassaRistorante.ViewModels;

namespace CassaRistorante
{
    public partial class MainWindow : Window
    {
        public MainWindow()
        {
            InitializeComponent();
            DataContext = new CassaRistoranteViewModel();
        }
    }
}
```

Questo è un ottimo punto di arrivo per la finestra principale del caso finale.

## Il `ViewModel` come centro delle azioni

A questo punto, i comandi principali dovrebbero vivere tutti nel `ViewModel`:

```csharp
public ICommand AggiungiPiattoCommand { get; }
public ICommand AumentaQuantitaCommand { get; }
public ICommand DiminuisciQuantitaCommand { get; }
public ICommand RimuoviRigaCommand { get; }
public ICommand ChiudiContoCommand { get; }
```

Questa lista, da sola, racconta già molto bene il vocabolario della schermata.

## Perché è così importante

Perché mostra al lettore che una schermata WPF ben progettata può essere letta anche dal lato delle azioni:
- che cosa può fare l'utente?
- in che stato ha senso farlo?
- dove vive la logica di ciascuna azione?

Questa chiarezza è uno dei guadagni più grandi di MVVM.

## Il ruolo di `CanExecute`

Arrivati qui, è il momento giusto per usare davvero `CanExecute` come strumento di qualità della UI.

Per esempio:
- `AggiungiPiattoCommand` richiede un tavolo e un piatto selezionato;
- `RimuoviRigaCommand` richiede una riga selezionata;
- `ChiudiContoCommand` richiede un tavolo con almeno una riga.

Questa logica rende l'interfaccia più intelligente senza sporcare la view.

## Un esempio molto forte

```csharp
ChiudiContoCommand = new RelayCommand(
    _ => ChiudiConto(),
    _ => TavoloSelezionato is not null && RigheContoCorrenti.Any());
```

Qui si vede bene il vantaggio:
la UI non deve decidere manualmente quando abilitare il pulsante.
Il comando nasce già con la propria logica di disponibilità.

## Perché questo passaggio è fondamentale per il libro

Perché fino a qui il lettore ha studiato MVVM come concetto, commanding come meccanismo, e proprietà bindate come struttura.
Adesso li vede collaborare davvero in un progetto unico.

Questo è il punto in cui il progetto finale comincia a restituire al lettore anche una soddisfazione architetturale, non solo funzionale.

## Che cosa si guadagna davvero

Ripulendo il code-behind e consolidando i comandi, il progetto guadagna:
- leggibilità;
- ordine;
- testabilità futura;
- coerenza con il percorso del libro;
- migliore separazione delle responsabilità.

Ed è esattamente il tipo di salto qualitativo che un caso finale dovrebbe far vedere.

## Regola pratica

Il progetto finale non deve solo funzionare.
Deve anche mostrare al lettore dove conviene far vivere la logica.
Quando i comandi stanno nel `ViewModel` e il code-behind resta leggero, la schermata diventa molto più solida.
