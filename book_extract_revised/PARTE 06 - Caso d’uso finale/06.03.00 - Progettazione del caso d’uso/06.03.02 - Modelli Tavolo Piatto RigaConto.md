# 06.03.02 - Modelli: `Tavolo`, `Piatto`, `RigaConto`

## Dopo le regole, servono classi chiare e leggibili

Una volta definite le regole funzionali, il passo naturale è progettare i modelli del caso d'uso.

In questo progetto finale, i tre modelli fondamentali sono:
- `Tavolo`
- `Piatto`
- `RigaConto`

La scelta è molto sensata dal punto di vista didattico, perché questi tre oggetti permettono di rappresentare il dominio senza gonfiarlo inutilmente.

## Il modello `Tavolo`

Il `Tavolo` rappresenta il contesto operativo corrente su cui lavora l'utente.

Per il nostro caso, può avere proprietà come:
- identificativo;
- nome o numero tavolo;
- stato occupato/libero;
- eventualmente la collezione delle righe conto.

### Esempio

```csharp
using System.Collections.ObjectModel;
using System.ComponentModel;
using System.Runtime.CompilerServices;
using System.Linq;

public class Tavolo : INotifyPropertyChanged
{
    private int _id;
    private string _nome = "";
    private ObservableCollection<RigaConto> _righeConto = new();

    public int Id
    {
        get => _id;
        set
        {
            if (_id != value)
            {
                _id = value;
                OnPropertyChanged();
            }
        }
    }

    public string Nome
    {
        get => _nome;
        set
        {
            if (_nome != value)
            {
                _nome = value;
                OnPropertyChanged();
            }
        }
    }

    public ObservableCollection<RigaConto> RigheConto
    {
        get => _righeConto;
        set
        {
            if (_righeConto != value)
            {
                _righeConto = value;
                OnPropertyChanged();
                OnPropertyChanged(nameof(IsOccupato));
                OnPropertyChanged(nameof(Totale));
            }
        }
    }

    public bool IsOccupato => RigheConto.Any();

    public decimal Totale => RigheConto.Sum(r => r.Subtotale);

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

## Perché `Tavolo` è così importante

Perché non è solo un elemento grafico della sala.
È il contenitore logico dell'ordine corrente.

Didatticamente è prezioso perché collega:
- selezione;
- contesto corrente;
- stato del tavolo;
- totale del conto.

## Il modello `Piatto`

`Piatto` rappresenta un articolo ordinabile.

Per il nostro caso finale, ci basta tenerlo essenziale:
- identificativo;
- nome;
- prezzo;
- eventualmente categoria.

## Perché `Piatto` va tenuto semplice

Perché qui il suo ruolo è molto chiaro:
rappresentare ciò che si può aggiungere al conto.

Non serve appesantirlo con dettagli di magazzino, cucina o fiscalità.
Nel caso didattico basta che sia un'entità ordinabile, leggibile e bindabile.

## Il modello `RigaConto`

`RigaConto` è il punto più interessante del dominio, perché collega:
- un piatto;
- una quantità;
- un prezzo;
- un subtotale.

È il modello che rende il conto una struttura viva e dinamica.

### Esempio

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;

public class RigaConto : INotifyPropertyChanged
{
    private Piatto _piatto = new();
    private int _quantita = 1;

    public Piatto Piatto
    {
        get => _piatto;
        set
        {
            if (_piatto != value)
            {
                _piatto = value;
                OnPropertyChanged();
                OnPropertyChanged(nameof(NomePiatto));
                OnPropertyChanged(nameof(PrezzoUnitario));
                OnPropertyChanged(nameof(Subtotale));
            }
        }
    }

    public int Quantita
    {
        get => _quantita;
        set
        {
            if (_quantita != value)
            {
                _quantita = value;
                OnPropertyChanged();
                OnPropertyChanged(nameof(Subtotale));
            }
        }
    }

    public string NomePiatto => Piatto.Nome;

    public decimal PrezzoUnitario => Piatto.Prezzo;

    public decimal Subtotale => PrezzoUnitario * Quantita;

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

## Perché `RigaConto` è didatticamente fortissima

Perché obbliga a progettare bene:
- proprietà derivate;
- notifiche;
- aggiornamento del subtotale;
- rapporto tra modello base (`Piatto`) e riga operativa.

È anche uno dei punti migliori del caso finale per mostrare come WPF gestisca molto bene dati che cambiano e UI che si aggiorna.

## Perché usare `INotifyPropertyChanged` anche qui

Anche se alcuni modelli potrebbero sembrare “semplici”, in questo caso ha molto senso che tutti e tre siano bindabili in modo serio.

Questo permette:
- aggiornamento immediato dei dati mostrati;
- ricalcolo coerente del totale;
- stato visivo sempre allineato;
- maggiore naturalità nella UI WPF.

## Relazione tra i modelli

Il lettore può visualizzare così il rapporto tra i tre modelli:

- `Tavolo` contiene molte `RigaConto`
- ogni `RigaConto` punta a un `Piatto`
- il totale del `Tavolo` dipende dalla somma dei subtotali delle sue righe

Questa struttura è semplice ma molto espressiva.

## Perché questa triade è così buona per un caso finale

Perché è:
- realistica;
- leggibile;
- estendibile;
- abbastanza ricca da far emergere problemi veri;
- abbastanza semplice da restare didatticamente pulita.

È esattamente il tipo di modello che aiuta il lettore a vedere WPF in azione su un piccolo dominio credibile.

## Regola pratica

I modelli del caso finale devono restare pochi ma forti.
`Tavolo`, `Piatto` e `RigaConto` sono un'ottima triade perché rappresentano bene il dominio e fanno lavorare davvero binding, notifiche e dati derivati.
