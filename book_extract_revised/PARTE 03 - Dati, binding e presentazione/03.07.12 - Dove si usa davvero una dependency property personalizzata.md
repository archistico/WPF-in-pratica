# 03.07.12 - Dove si usa davvero una dependency property personalizzata

## Il punto pratico che evita di studiarle in astratto

Dopo tutta la teoria, arriva la domanda più importante:
dove si usa davvero una dependency property personalizzata?

Perché se il lettore non vede scenari reali, il rischio è pensare che sia solo un costrutto tecnico avanzato e lontano dalla pratica.

## Primo caso classico: proprietà di un controllo personalizzato

Questo è il caso più tipico.

Se stai creando un controllo WPF vero e proprio e vuoi esporre proprietà che devono essere:
- bindabili;
- stilizzabili;
- animabili;
- usabili da XAML come proprietà native del controllo;

allora una dependency property è una candidata naturale.

Per esempio:
- `Titolo`
- `Valore`
- `IsEvidenziato`
- `Icona`
- `ColorePrimario`

## Secondo caso classico: proprietà di un `UserControl`

Anche in un `UserControl`, se vuoi esporre verso l'esterno proprietà che il consumatore del controllo possa usare in modo naturale da XAML, molto spesso una dependency property è la scelta giusta.

Per esempio:

```xml
<local:SchedaCliente Titolo="{Binding NomeCliente}" />
```

Se `Titolo` deve comportarsi come una vera proprietà WPF del controllo, la dependency property è molto adatta.

## Terzo caso: proprietà che devono entrare davvero nel sistema WPF

Anche fuori dai casi più scolastici, la dependency property è giustificata quando la proprietà deve partecipare seriamente a:
- binding target-side del controllo;
- stili;
- template;
- animazioni;
- metadati di comportamento;
- eventuale ereditarietà dei valori.

In altre parole, quando la proprietà deve essere “cittadina piena” del property system.

## Un esempio realistico

Immagina un `UserControl` chiamato `InfoBadge` con proprietà come:
- `Titolo`
- `Valore`
- `IsInErrore`

Vuoi che l'utente del controllo possa:
- impostarle da XAML;
- bindarle;
- magari stilizzare il controllo in base ad esse.

Questo è un buon scenario per dependency property personalizzate.

## Dove invece non sono la prima scelta

Non sono la prima scelta per:
- proprietà del view model;
- modelli dati;
- classi di business;
- DTO;
- classi che non partecipano al sistema UI di WPF.

In quei casi di solito ha più senso usare proprietà CLR normali e, se serve notifica, `INotifyPropertyChanged`.

## Un buon criterio decisionale

Chiediti:
“questa proprietà appartiene davvero a un controllo o a un oggetto UI WPF che deve integrarsi profondamente col framework?”

Se la risposta è sì, la dependency property è una candidata forte.
Se la risposta è no, probabilmente stai guardando al livello sbagliato.

## Errore tipico

L'errore più comune è creare dependency property in classi che non ne hanno davvero bisogno, solo perché la sintassi è stata appena imparata.

Questo porta solo più complessità, non più qualità.

## Regola pratica

Una dependency property personalizzata si usa davvero quando una proprietà del tuo controllo deve comportarsi come una proprietà nativa di WPF.
Se non hai questa esigenza, spesso non è lo strumento giusto.
