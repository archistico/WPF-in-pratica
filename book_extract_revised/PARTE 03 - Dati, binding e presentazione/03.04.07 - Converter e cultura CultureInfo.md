# 03.04.07 - Converter e cultura (`CultureInfo`)

## Perché la cultura conta più di quanto sembri

Quando un converter lavora con:
- numeri;
- date;
- valute;
- stringhe formattate;
- parsing da testo;

la cultura può cambiare il risultato in modo molto concreto.

Per esempio:
- `1,50` e `1.50` non significano la stessa cosa in tutte le culture;
- una data può essere interpretata in modi diversi;
- una valuta può essere formattata con simboli e convenzioni differenti.

Per questo `CultureInfo` non è un dettaglio secondario.

## Dove compare nel converter

Sia `Convert` sia `ConvertBack` ricevono il parametro:

```csharp
CultureInfo culture
```

Questo permette al converter di:
- formattare correttamente un valore;
- interpretare correttamente l'input;
- evitare conversioni culturalmente sbagliate.

## Un esempio con valuta

```csharp
using System;
using System.Globalization;
using System.Windows.Data;

public class CurrencyConverter : IValueConverter
{
    public object Convert(object value, Type targetType, object parameter, CultureInfo culture)
    {
        if (value is decimal importo)
            return importo.ToString("C", culture);

        return string.Empty;
    }

    public object ConvertBack(object value, Type targetType, object parameter, CultureInfo culture)
    {
        if (decimal.TryParse(value?.ToString(), NumberStyles.Currency, culture, out decimal risultato))
            return risultato;

        return 0m;
    }
}
```

## Che cosa c'è di importante qui

### `Convert`
Usa la cultura per formattare la valuta nel modo corretto.

### `ConvertBack`
Usa la cultura per interpretare il testo inserito nel modo corretto.

Se la cultura è italiana, ad esempio, il separatore decimale e il formato monetario si comporteranno come ci si aspetta in quel contesto.

## Perché è importante soprattutto in `ConvertBack`

Se il converter riceve input dall'utente e lo deve riportare al modello, ignorare la cultura può produrre:
- parse errati;
- eccezioni;
- valori interpretati in modo scorretto.

Questo è particolarmente importante in form numerici e date.

## Un esempio con testo in maiuscolo

Anche in trasformazioni apparentemente semplici, la cultura può contare:

```csharp
return value?.ToString()?.ToUpper(culture) ?? string.Empty;
```

Questo è più corretto di una trasformazione cieca in contesti internazionali.

## Quando devi pensarci davvero

Devi pensarci soprattutto quando il converter lavora con:
- importi;
- percentuali;
- numeri decimali;
- date;
- parsing di testo verso tipi numerici;
- stringhe che dipendono da convenzioni linguistiche.

## Quando puoi quasi ignorarla

Se il converter fa una trasformazione puramente logica, per esempio:
- `bool` → `Visibility`
- enum → brush

allora la cultura tende a essere irrilevante.

## Errori tipici

### Fare parsing usando convenzioni implicite non controllate
Il codice funziona in un contesto e fallisce in un altro.

### Formattare numeri senza considerare la cultura
La UI mostra valori in modo poco naturale per l'utente.

### Ignorare del tutto il parametro `culture` in converter che gestiscono testo numerico o monetario
È un errore più serio di quanto sembri.

## Regola pratica

Se un converter formatta o interpreta valori dipendenti dalla lingua o dalle convenzioni locali, `CultureInfo` non è un accessorio: è parte della correttezza della conversione.
