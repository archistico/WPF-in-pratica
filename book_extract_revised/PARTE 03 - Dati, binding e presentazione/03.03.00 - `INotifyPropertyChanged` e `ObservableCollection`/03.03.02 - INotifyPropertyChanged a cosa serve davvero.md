# 03.03.02 - `INotifyPropertyChanged`: a cosa serve davvero

## Non è solo una formalità tecnica

Molti leggono `INotifyPropertyChanged` come una specie di passaggio obbligatorio da imparare a memoria.
In realtà è molto di più.

Serve a risolvere un problema centrale:
**come fa la UI a sapere che una proprietà dell'oggetto è cambiata dopo il caricamento iniziale?**

Se non capisci bene questa domanda, `INotifyPropertyChanged` sembra solo una piccola interfaccia tecnica.
Se la capisci, diventa uno dei pilastri del binding in WPF.

## Il problema concreto

Supponiamo di avere un modello molto semplice:

```csharp
public class Persona
{
    public string Nome { get; set; } = "";
}
```

E una UI con binding:

```xml
<TextBlock Text="{Binding Nome}" />
```

Se all'avvio il `DataContext` contiene `"Anna"`, il testo appare correttamente.

Ma se dopo scrivi da codice:

```csharp
persona.Nome = "Luca";
```

la UI non si aggiorna automaticamente.

Perché?
Perché l'oggetto non ha notificato il cambiamento.

## Che cosa fa `INotifyPropertyChanged`

`INotifyPropertyChanged` permette a un oggetto di emettere un segnale quando una sua proprietà cambia.

WPF ascolta quel segnale.
Quando lo riceve, può aggiornare i binding collegati a quella proprietà.

In altre parole:
- l'oggetto cambia;
- notifica il cambiamento;
- la UI si aggiorna.

## La visione corretta

`INotifyPropertyChanged` non “fa il binding”.
Il binding esiste già.

Quello che fa è rendere **reattivo** il binding rispetto ai cambi di proprietà del modello.

Questa distinzione è importante.

## Un'implementazione classica

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;

public class Persona : INotifyPropertyChanged
{
    private string _nome = "";

    public string Nome
    {
        get => _nome;
        set
        {
            if (_nome != value)
            {
                _nome = value;
                OnPropertyChanged();
            }
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

## Come leggere davvero questo codice

### 1. Non usi più una auto-property banale
Perché devi intercettare il cambio.

### 2. Verifichi se il valore è davvero diverso
Così eviti notifiche inutili.

### 3. Chiami `OnPropertyChanged()`
È qui che avviene la notifica.

### 4. La UI può reagire
I controlli bindati a `Nome` sanno che devono aggiornarsi.

## Quando serve davvero

Serve quando:
- il valore può cambiare nel tempo;
- la UI deve riflettere quel cambiamento;
- il cambiamento può avvenire anche da codice e non solo dalla UI;
- vuoi una schermata sincronizzata con il modello.

## Quando non è il focus principale

Se il valore viene impostato una volta sola prima che la UI venga mostrata e poi non cambia mai, `INotifyPropertyChanged` può non sembrare indispensabile.

Ma nelle applicazioni reali, questa situazione pura è rara.

## Un esempio mentale molto utile

### Senza notifica
La UI legge il dato all'inizio, poi resta ferma.

### Con notifica
La UI continua a seguire il dato anche quando cambia.

Questo è il vero valore dell'interfaccia.

## Errori tipici

### Pensare che basti il binding iniziale
Non basta se il dato cambia dopo.

### Pensare che `TwoWay` risolva anche questo
`TwoWay` regola la direzione del binding, ma non sostituisce la notifica.

### Dimenticare la chiamata a `OnPropertyChanged()`
È uno degli errori più classici.

## Regola pratica

`INotifyPropertyChanged` serve quando vuoi che la UI non mostri solo il valore iniziale, ma resti allineata ai cambiamenti reali dell'oggetto.
