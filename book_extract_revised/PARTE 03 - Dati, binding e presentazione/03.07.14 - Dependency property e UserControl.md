# 03.07.14 - Dependency property e `UserControl`

## Uno degli scenari più pratici per iniziare a usarle davvero

Tra tutti i casi in cui potresti dichiarare una dependency property personalizzata, il `UserControl` è uno dei più concreti e più didattici.

Perché?
Perché ti permette di costruire un componente riusabile e di esporre proprietà che il consumatore del controllo può usare in XAML come se fossero native.

## Un esempio tipico

Immagina un `UserControl` chiamato `SchedaInfo` che deve mostrare:
- un titolo;
- un valore;
- eventualmente uno stato.

Vuoi poterlo usare così:

```xml
<local:SchedaInfo Titolo="{Binding NomeCliente}"
                  Valore="{Binding CodiceCliente}" />
```

Per rendere queste proprietà naturali, bindabili e coerenti con WPF, le dependency property sono spesso la soluzione corretta.

## Esempio completo minimale

Code-behind del `UserControl`:

```csharp
using System.Windows;
using System.Windows.Controls;

public partial class SchedaInfo : UserControl
{
    public static readonly DependencyProperty TitoloProperty =
        DependencyProperty.Register(
            nameof(Titolo),
            typeof(string),
            typeof(SchedaInfo),
            new PropertyMetadata(string.Empty));

    public string Titolo
    {
        get => (string)GetValue(TitoloProperty);
        set => SetValue(TitoloProperty, value);
    }

    public static readonly DependencyProperty ValoreProperty =
        DependencyProperty.Register(
            nameof(Valore),
            typeof(string),
            typeof(SchedaInfo),
            new PropertyMetadata(string.Empty));

    public string Valore
    {
        get => (string)GetValue(ValoreProperty);
        set => SetValue(ValoreProperty, value);
    }

    public SchedaInfo()
    {
        InitializeComponent();
    }
}
```

XAML del controllo:

```xml
<UserControl x:Class="MioProgetto.SchedaInfo"
             xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml">
    <Border Padding="12" BorderThickness="1">
        <StackPanel>
            <TextBlock Text="{Binding Titolo, RelativeSource={RelativeSource AncestorType=UserControl}}"
                       FontWeight="SemiBold"/>
            <TextBlock Text="{Binding Valore, RelativeSource={RelativeSource AncestorType=UserControl}}"
                       Margin="0,6,0,0"/>
        </StackPanel>
    </Border>
</UserControl>
```

## Perché questo esempio è importante

Perché mostra un caso molto reale:
- il controllo espone proprietà proprie;
- quelle proprietà sono usabili da XAML esterno;
- il contenuto interno del controllo le legge tramite binding.

Questo è uno dei modi più comuni in cui le dependency property entrano davvero nel lavoro pratico su WPF.

## Perché non basta sempre una proprietà CLR normale

Una semplice proprietà CLR potrebbe sembrare sufficiente, ma perderesti molta integrazione con il modello WPF:
- binding più naturale;
- uso come proprietà XAML del controllo;
- partecipazione migliore al property system;
- compatibilità più piena con stili e template.

## Un punto importante sul binding interno

Nel XAML interno del `UserControl`, spesso devi usare `RelativeSource` o soluzioni equivalenti per riferirti alle proprietà del controllo stesso.
Questo è un punto pratico che torna spesso.

## Quando questo approccio è davvero giusto

È giusto quando il `UserControl` è pensato come componente riusabile e configurabile dall'esterno.

Se invece il controllo è ultra-locale e non ha vere proprietà pubbliche da esporre, può non servire.

## Errori tipici

### Creare proprietà CLR normali e aspettarsi che si comportino da vere proprietà WPF del controllo
Il risultato è meno naturale.

### Non capire come bindare internamente le property del `UserControl`
Si crea confusione tra `DataContext` interno e proprietà del controllo.

### Usare dependency property anche per dettagli che non devono essere esposti
Serve sempre criterio.

## Regola pratica

Nel `UserControl`, usa dependency property quando vuoi esporre proprietà pubbliche del componente in modo naturale, bindabile e coerente con WPF.
