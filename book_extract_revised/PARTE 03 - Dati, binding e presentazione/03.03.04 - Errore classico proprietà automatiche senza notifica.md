# 03.03.04 - Errore classico: proprietà automatiche senza notifica

## Uno degli errori più frequenti in assoluto

Questo errore è così comune che merita un sottocapitolo dedicato.

Il caso tipico è questo:
definisci un modello con auto-property molto pulite, il binding sembra corretto, il dato iniziale compare bene, ma quando cambi la proprietà da codice la UI non si aggiorna.

Il motivo è quasi sempre lo stesso:
**la proprietà non notifica il cambiamento**.

## L'esempio che sembra giusto ma non basta

```csharp
public class Cliente
{
    public string Nome { get; set; } = "";
    public string Città { get; set; } = "";
}
```

Questa classe è formalmente corretta come classe C#.
Ma per un binding vivo in WPF è insufficiente se i valori devono cambiare nel tempo.

## Perché l'errore inganna così tanto

Perché all'inizio tutto sembra funzionare.

Se il `DataContext` contiene:

```csharp
new Cliente
{
    Nome = "Anna",
    Città = "Torino"
}
```

e hai lo XAML:

```xml
<TextBlock Text="{Binding Nome}" />
<TextBlock Text="{Binding Città}" />
```

la UI mostrerà correttamente i valori.

Quindi il lettore pensa:
“Perfetto, il binding funziona”.

Poi però cambia il valore:

```csharp
cliente.Nome = "Luca";
```

e non succede nulla nella UI.

## Il vero problema

Il problema non è il binding.
Il problema è che l'oggetto non ha detto a WPF:
“Ehi, `Nome` è cambiato”.

Senza `INotifyPropertyChanged`, la UI non ha motivo di sapere che deve aggiornarsi.

## La versione corretta

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;

public class Cliente : INotifyPropertyChanged
{
    private string _nome = "";
    private string _città = "";

    public string Nome
    {
        get => _nome;
        set
        {
            if (_nome != value)
            {
                _nome = value;
                OnPropertyChanged();
            }
        }
    }

    public string Città
    {
        get => _città;
        set
        {
            if (_città != value)
            {
                _città = value;
                OnPropertyChanged();
            }
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

## Quando le auto-property vanno bene e quando no

Le auto-property vanno bene quando:
- il dato è statico o quasi statico;
- non ti serve notifica;
- la proprietà non deve aggiornare la UI dopo il caricamento iniziale.

Non vanno bene quando:
- il valore può cambiare;
- la UI deve reagire;
- stai costruendo una schermata viva.

## Perché è importante non banalizzare questo errore

Perché non è solo un dettaglio di sintassi C#.
È una differenza strutturale tra:
- un oggetto passivo;
- un oggetto che collabora col sistema di binding.

Questo è il vero salto.

## Segnale tipico che sei in questo errore

Il dato iniziale si vede.
Le modifiche fatte dall'utente magari arrivano al modello.
Ma le modifiche fatte dal codice non tornano nella UI.

Quando vedi questo pattern, pensa subito a qui.

## Regola pratica

Se una proprietà deve partecipare seriamente al data binding dinamico, una auto-property semplice quasi mai basta da sola.
