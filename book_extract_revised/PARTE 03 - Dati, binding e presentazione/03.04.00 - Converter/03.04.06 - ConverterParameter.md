# 03.04.06 - `ConverterParameter`

## Come rendere un converter più flessibile senza moltiplicarlo inutilmente

A volte un converter fa quasi quello che ti serve, ma con una piccola variazione:
- invertire un comportamento;
- scegliere una modalità diversa;
- cambiare un'etichetta;
- applicare una logica parametrica.

In questi casi creare tanti converter quasi uguali sarebbe scomodo.
Qui entra in gioco `ConverterParameter`.

## Che cos'è

`ConverterParameter` è un valore opzionale che puoi passare al converter dal binding.

Il converter lo riceve nel parametro `parameter` dei metodi:
- `Convert`
- `ConvertBack`

Questo gli permette di adattare il comportamento senza dover creare una nuova classe per ogni minima variante.

## Un esempio classico: inversione logica

Supponiamo di avere un converter `bool` → `Visibility` e di volerlo usare in due modi:
- `true` mostra;
- `true` nasconde.

Potresti usare un parametro per distinguere i due comportamenti.

## Esempio

```csharp
using System;
using System.Globalization;
using System.Windows;
using System.Windows.Data;

public class BoolToVisibilityConverter : IValueConverter
{
    public object Convert(object value, Type targetType, object parameter, CultureInfo culture)
    {
        bool flag = value is bool b && b;
        bool invert = string.Equals(parameter?.ToString(), "Invert", StringComparison.OrdinalIgnoreCase);

        if (invert)
            flag = !flag;

        return flag ? Visibility.Visible : Visibility.Collapsed;
    }

    public object ConvertBack(object value, Type targetType, object parameter, CultureInfo culture)
    {
        if (value is Visibility visibility)
        {
            bool result = visibility == Visibility.Visible;
            bool invert = string.Equals(parameter?.ToString(), "Invert", StringComparison.OrdinalIgnoreCase);

            return invert ? !result : result;
        }

        return false;
    }
}
```

## Uso nello XAML

Caso normale:

```xml
<TextBlock Visibility="{Binding MostraMessaggio,
                                Converter={StaticResource BoolToVisibilityConverter}}" />
```

Caso invertito:

```xml
<TextBlock Visibility="{Binding MostraMessaggio,
                                Converter={StaticResource BoolToVisibilityConverter},
                                ConverterParameter=Invert}" />
```

## Perché è utile

Perché con un solo converter puoi coprire più varianti coerenti dello stesso comportamento.

Questo mantiene il progetto più ordinato.

## Quando usarlo davvero

Usalo quando:
- la variazione è piccola e naturale;
- il converter resta leggibile;
- il parametro migliora il riuso;
- la logica non diventa ambigua.

## Quando fermarti

Se il converter inizia a gestire:
- troppi parametri;
- modalità troppo diverse;
- rami logici troppo complessi;

allora probabilmente non stai più usando bene `ConverterParameter`.
In quel caso è meglio:
- separare i converter;
- o ripensare la logica nel view model.

## Un altro esempio: testo con prefisso

```csharp
public class PrefixConverter : IValueConverter
{
    public object Convert(object value, Type targetType, object parameter, CultureInfo culture)
    {
        string prefisso = parameter?.ToString() ?? "";
        return prefisso + (value?.ToString() ?? "");
    }

    public object ConvertBack(object value, Type targetType, object parameter, CultureInfo culture)
    {
        return value?.ToString() ?? "";
    }
}
```

Uso:

```xml
<TextBlock Text="{Binding Nome,
                         Converter={StaticResource PrefixConverter},
                         ConverterParameter=Cliente: }" />
```

## Errori tipici

### Usare il parametro per fare troppe cose diverse
Il converter diventa difficile da leggere.

### Affidarsi a stringhe “magiche” ovunque
È comodo, ma può diventare fragile se abusato.

### Usare `ConverterParameter` dove servirebbe una soluzione più chiara
Per esempio una vera proprietà nel view model.

## Regola pratica

`ConverterParameter` è utile per piccole variazioni coerenti dello stesso converter.
Se il comportamento diventa troppo ricco o troppo ramificato, è il segnale che devi cambiare approccio.
