# 03.01.05 - Binding `OneWay` e `TwoWay`

## Non tutti i binding lavorano nelle due direzioni

Uno degli aspetti più importanti del data binding è capire **in che direzione** scorre il dato.

Molti principianti vedono un binding come un collegamento generico, ma in realtà la direzione conta molto.

I due casi più importanti da conoscere subito sono:
- `OneWay`
- `TwoWay`

## `OneWay`

Con `OneWay`, il dato va dalla source al target.

In pratica:
- l'oggetto sorgente espone un valore;
- il controllo lo mostra;
- se il valore nella source cambia e il binding è notificato correttamente, la UI si aggiorna;
- ma la modifica fatta nel controllo non torna automaticamente nella source.

Esempio tipico:

```xml
<TextBlock Text="{Binding Nome, Mode=OneWay}" />
```

Questo è perfetto per controlli che devono solo visualizzare.

## `TwoWay`

Con `TwoWay`, il collegamento è bidirezionale.

In pratica:
- la UI mostra il valore della source;
- se l'utente modifica il controllo, il valore torna nella source;
- se la source cambia, la UI può aggiornarsi.

Esempio:

```xml
<TextBox Text="{Binding Nome, Mode=TwoWay}" />
```

Questo è il caso classico dei form editabili.

## Perché questa distinzione è fondamentale

Perché scegliere la modalità giusta cambia completamente il comportamento del controllo.

Se usi `OneWay` dove ti serve modifica, l'utente cambia la UI ma l'oggetto dati non viene aggiornato come ti aspetti.
Se usi `TwoWay` dove devi solo visualizzare, stai introducendo una capacità inutile.

## Un esempio molto chiaro

Supponiamo di avere:

```csharp
public class Persona
{
    public string Nome { get; set; }
}
```

E nel `DataContext`:

```csharp
DataContext = new Persona
{
    Nome = "Anna"
};
```

### Caso `OneWay`

```xml
<TextBox Text="{Binding Nome, Mode=OneWay}" Width="200" />
```

La `TextBox` mostrerà `"Anna"`, ma il valore non verrà usato come collegamento bidirezionale nel modo che ti aspetti per l'editing.

### Caso `TwoWay`

```xml
<TextBox Text="{Binding Nome, Mode=TwoWay}" Width="200" />
```

Ora il campo è progettato per lavorare come input editabile collegato alla proprietà.

## Attenzione: alcuni controlli hanno comportamenti predefiniti diversi

In WPF molte proprietà di dipendenza hanno una modalità di binding predefinita sensata.
Per esempio, proprietà come `TextBox.Text` sono spesso pensate per l'editing e quindi si prestano bene al `TwoWay`.

Ma didatticamente è utile esplicitare il `Mode`, almeno finché i concetti non sono diventati naturali.

## Quando usare `OneWay`

Usalo quando:
- il controllo serve a visualizzare;
- non vuoi che l'utente scriva nella source tramite quel controllo;
- il dato è di sola lettura;
- vuoi esprimere chiaramente che la direzione è solo source → target.

## Quando usare `TwoWay`

Usalo quando:
- il controllo serve a modificare i dati;
- stai costruendo un form;
- il valore scritto nella UI deve aggiornare l'oggetto sorgente;
- vuoi una vera sincronizzazione tra dato e input.

## Errori tipici

### Dimenticare la direzione del binding
Si pensa che ogni binding sia automaticamente bidirezionale.

### Usare `TwoWay` ovunque
Non è sempre sbagliato, ma spesso è superfluo.

### Usare `OneWay` in un form e poi stupirsi che l'oggetto non si aggiorni come previsto
È uno degli errori più formativi da vedere.

## Regola pratica

Se il controllo **mostra soltanto**, pensa a `OneWay`.
Se il controllo **mostra e modifica**, pensa a `TwoWay`.
