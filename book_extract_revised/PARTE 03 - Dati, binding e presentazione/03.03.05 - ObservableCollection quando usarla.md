# 03.03.05 - `ObservableCollection<T>`: quando usarla

## Il problema delle liste vive

Dopo aver capito `INotifyPropertyChanged`, molti pensano di aver risolto tutto il discorso degli aggiornamenti.
Ma appena entrano in gioco gli elenchi, compare un nuovo problema.

Se una UI mostra una collezione e tu:
- aggiungi un elemento;
- ne rimuovi uno;
- svuoti la lista;

la UI non sempre si aggiorna automaticamente se stai usando una lista normale.

È qui che entra in gioco `ObservableCollection<T>`.

## Che cos'è in pratica

`ObservableCollection<T>` è una collezione che notifica quando la sua composizione cambia.

In altre parole, avvisa la UI quando:
- aggiungi elementi;
- rimuovi elementi;
- riorganizzi la collezione;
- la svuoti.

Questo la rende molto adatta per binding verso controlli come:
- `ListBox`
- `ListView`
- `DataGrid`
- `ItemsControl`
- `ComboBox`

## Un esempio semplice

```csharp
public ObservableCollection<string> Nomi { get; } = new();
```

Se un controllo è bindato a `Nomi` e fai:

```csharp
Nomi.Add("Anna");
Nomi.Add("Luca");
```

la UI può aggiornarsi automaticamente mostrando i nuovi elementi.

## Perché una `List<T>` normale spesso non basta

Supponiamo di avere:

```csharp
public List<string> Nomi { get; } = new();
```

Se aggiungi elementi dopo che la UI ha già fatto il binding, il controllo non viene automaticamente informato nel modo giusto.

Quindi il problema non è che la lista contenga dati sbagliati.
Il problema è che non sta notificando alla UI il cambiamento della collezione.

## Quando usare `ObservableCollection<T>`

Usala quando:
- la collezione deve essere mostrata in un controllo UI;
- gli elementi possono essere aggiunti o rimossi dopo il caricamento iniziale;
- vuoi che il controllo resti sincronizzato con la collezione;
- la lista è viva e non solo iniziale.

## Quando una `List<T>` può ancora bastare

Una `List<T>` può bastare quando:
- i dati vengono caricati una volta sola;
- non prevedi modifiche successive;
- assegni un'intera nuova collection al binding in un altro modo gestito;
- la lista non deve notificare cambiamenti incrementali.

Ma appena l'elenco diventa dinamico, `ObservableCollection<T>` è quasi sempre la scelta naturale.

## Un esempio realistico

Immagina una schermata ordini con un pulsante “Aggiungi riga”.

Se il `DataGrid` è bindato a una `ObservableCollection<RigaDocumento>`, l'aggiunta di una riga può comparire subito in UI senza refresh manuali.

Questo è esattamente il tipo di scenario per cui la classe è stata pensata.

## Un punto importante

`ObservableCollection<T>` notifica che la **collezione** è cambiata.
Non sta dicendo nulla, da sola, sulle proprietà interne dei singoli elementi.
Questo sarà il punto centrale del sottocapitolo successivo.

## Errori tipici

### Usare `List<T>` in schermate dinamiche
La UI non reagisce alle aggiunte o rimozioni come ti aspetti.

### Usare `ObservableCollection<T>` anche dove la collezione è del tutto statica
Non è grave, ma a volte è superfluo.

### Pensare che notifichi tutto
No: notifica i cambi della collezione, non automaticamente le proprietà di ogni elemento.

## Regola pratica

Se una lista mostrata in UI deve cambiare nel tempo, `ObservableCollection<T>` è quasi sempre il contenitore giusto da usare.
