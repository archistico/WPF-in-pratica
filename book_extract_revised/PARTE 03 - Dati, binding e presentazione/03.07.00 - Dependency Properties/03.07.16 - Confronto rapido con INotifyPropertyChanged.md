# 03.07.16 - Confronto rapido con `INotifyPropertyChanged`

## Una distinzione che va fissata molto bene

Questo è uno dei punti che confondono di più all'inizio:
che rapporto c'è tra dependency property e `INotifyPropertyChanged`?

La risposta breve è:
non sono la stessa cosa e non servono allo stesso livello.

Capire questa differenza è fondamentale.

## Il livello di `INotifyPropertyChanged`

`INotifyPropertyChanged` è lo strumento tipico per dire:
“una proprietà del mio oggetto .NET è cambiata”.

È usato soprattutto in:
- view model;
- modelli che devono notificare cambiamenti alla UI;
- classi di supporto dati.

È perfetto quando lavori nel mondo del modello dati o della logica di presentazione lato oggetto.

## Il livello della dependency property

La dependency property è invece uno strumento del property system di WPF.

È pensata soprattutto per:
- controlli;
- `UserControl`;
- componenti UI;
- proprietà che devono integrarsi con binding, stili, animazioni, template e metadati del framework.

Quindi lavora a un livello più vicino al motore della UI.

## Una formula molto utile

Puoi memorizzarla così:

- **view model / modello notificante** → `INotifyPropertyChanged`
- **proprietà di un controllo WPF** → spesso dependency property

Questa è una delle formule più utili dell'intero capitolo.

## Un confronto concreto

### Caso 1: proprietà `Nome` nel view model
Qui userai normalmente:

- proprietà CLR
- `INotifyPropertyChanged`

### Caso 2: proprietà `Titolo` di un `UserControl`
Qui spesso userai:

- dependency property

Perché `Titolo` deve comportarsi come una vera proprietà del controllo in XAML.

## Perché non sono in concorrenza diretta

Un errore frequente è pensare che una sostituisca l'altra.

In realtà:
- `INotifyPropertyChanged` notifica cambiamenti di proprietà di oggetti .NET;
- la dependency property integra la proprietà nel sistema di WPF.

Possono anche convivere nello stesso progetto, ma con responsabilità diverse.

## Un esempio molto realistico

Immagina una schermata con:
- un view model `ClienteViewModel` che implementa `INotifyPropertyChanged`;
- un `UserControl` `SchedaCliente` che espone `Titolo` e `Valore` come dependency property.

Qui entrambe le tecniche convivono perfettamente, ma su piani diversi.

## Quando il confronto chiarisce davvero

Il confronto è utile soprattutto per rispondere a questa domanda:
“devo creare una dependency property o mi basta una proprietà normale con `INotifyPropertyChanged`?”

La risposta dipende quasi sempre da dove vive la proprietà:
- nel controllo UI?
- o nel view model/dato?

## Errori tipici

### Usare dependency property nei view model
Quasi sempre è fuori posto.

### Cercare di usare `INotifyPropertyChanged` per far sembrare nativa una proprietà di controllo WPF
Si perde integrazione col framework.

### Pensare che “notificare un cambio” e “far parte del property system WPF” siano la stessa cosa
Non lo sono.

## Regola pratica

Se la proprietà appartiene al modello o al view model, pensa a `INotifyPropertyChanged`.
Se appartiene davvero al controllo WPF come proprietà UI di primo livello, pensa a una dependency property.
