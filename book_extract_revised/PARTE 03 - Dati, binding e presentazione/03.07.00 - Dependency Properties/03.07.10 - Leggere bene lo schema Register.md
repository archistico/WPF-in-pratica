# 03.07.10 - Leggere bene lo schema `Register`

## Il cuore tecnico della dichiarazione

Nel sottocapitolo precedente hai visto la struttura completa di una dependency property.
Ora conviene soffermarsi sulla parte più importante e più tecnica:
la chiamata a `DependencyProperty.Register(...)`.

È qui che molti lettori si bloccano, ma se la leggi a pezzi diventa molto più chiara.

## Esempio di riferimento

```csharp
public static readonly DependencyProperty TitoloProperty =
    DependencyProperty.Register(
        nameof(Titolo),
        typeof(string),
        typeof(MioControllo),
        new PropertyMetadata(string.Empty));
```

## Primo parametro: il nome della proprietà

```csharp
nameof(Titolo)
```

Questo è il nome logico della proprietà, cioè il nome con cui la proprietà CLR viene esposta.

Usare `nameof(Titolo)` è meglio che scrivere `"Titolo"` a mano, perché è più sicuro in caso di refactoring.

## Secondo parametro: il tipo del valore

```csharp
typeof(string)
```

Qui stai dicendo che il valore della dependency property è di tipo `string`.

Altri esempi potrebbero essere:
- `typeof(bool)`
- `typeof(int)`
- `typeof(Brush)`
- `typeof(Visibility)`

## Terzo parametro: il tipo proprietario

```csharp
typeof(MioControllo)
```

Questo indica il tipo che possiede la dependency property.

È il “proprietario” della property nel sistema WPF.

## Quarto parametro: i metadati

```csharp
new PropertyMetadata(string.Empty)
```

Qui definisci metadati associati alla property.

Nel caso più semplice, stai dicendo:
“il valore predefinito della proprietà è stringa vuota”.

Ma i metadati possono fare anche molto di più:
- callback;
- invalidazione layout;
- ereditarietà del valore;
- comportamenti avanzati.

Questo sarà il tema di un sottocapitolo successivo.

## Una lettura in linguaggio naturale

Questa registrazione può essere letta così:

“Registra una dependency property chiamata `Titolo`, di tipo `string`, appartenente a `MioControllo`, con valore predefinito stringa vuota.”

Questa frase è un ottimo modo di allenarsi a leggere lo schema.

## Perché è utile saperlo leggere bene

Perché quando avrai davanti una dependency property reale, molto spesso potrai capirne subito:
- il nome;
- il tipo;
- chi la possiede;
- che default ha;
- se ha comportamento speciale nei metadati.

Questa capacità di lettura vale quasi più della memoria meccanica della sintassi.

## Un altro esempio

```csharp
public static readonly DependencyProperty IsEvidenziatoProperty =
    DependencyProperty.Register(
        nameof(IsEvidenziato),
        typeof(bool),
        typeof(MioControllo),
        new PropertyMetadata(false));
```

Qui la lettura è:
- nome = `IsEvidenziato`
- tipo = `bool`
- owner = `MioControllo`
- default = `false`

## Errori tipici

### Leggere `Register` come una formula opaca
Meglio tradurlo mentalmente in frase.

### Confondere tipo del valore e tipo del proprietario
È uno degli errori più comuni all'inizio.

### Non dare importanza ai metadati
In realtà spesso sono il punto più ricco della registrazione.

## Regola pratica

Quando incontri `DependencyProperty.Register(...)`, scomponilo sempre in quattro domande:
- come si chiama?
- di che tipo è?
- a chi appartiene?
- quali metadati ha?

Se sai rispondere a queste domande, stai già leggendo bene la dependency property.
