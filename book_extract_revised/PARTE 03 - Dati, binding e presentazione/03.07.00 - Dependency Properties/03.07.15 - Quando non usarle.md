# 03.07.15 - Quando non usarle

## Uno dei sottocapitoli più utili in assoluto

Le dependency property sono potenti, ma non sono la risposta giusta per ogni proprietà.

Questo è un punto molto importante, perché appena impari a dichiararle potresti essere tentato di usarle troppo.

In realtà, sapere **quando non usarle** è una parte essenziale della maturità su WPF.

## Primo caso: proprietà del view model

Se stai scrivendo un view model, di solito non ti servono dependency property.

In quel contesto, la scelta normale è:
- proprietà CLR normali;
- `INotifyPropertyChanged` se serve notifica.

Per esempio:

```csharp
public class ClienteViewModel : INotifyPropertyChanged
{
    public string Nome { get; set; } = "";
}
```

Qui una dependency property sarebbe quasi sempre fuori posto.

## Secondo caso: modelli dati e classi di business

DTO, modelli di dominio, entità applicative, classi di servizio:
anche qui le dependency property non sono normalmente la scelta giusta.

Sono strumenti del sistema UI WPF, non del dominio applicativo.

## Terzo caso: proprietà che non devono partecipare davvero al property system

Se una proprietà:
- non deve essere bindabile come proprietà del controllo;
- non deve essere stilizzabile;
- non deve essere animabile;
- non deve partecipare a template o metadati;

allora una semplice proprietà CLR è spesso la scelta più pulita.

## Quarto caso: dettagli interni del controllo

Non ogni stato interno o valore tecnico del controllo deve diventare una dependency property.

Se una cosa è puramente interna e non deve essere esposta come parte del contratto del controllo, una proprietà o un campo normali possono essere più sensati.

## Perché abusarne è un problema

Perché una dependency property porta con sé:
- più codice;
- più complessità;
- più integrazione col framework;
- più responsabilità di progettazione.

Se il vantaggio non è reale, stai solo aumentando il peso del progetto.

## Un buon criterio decisionale

Chiediti:
“questa proprietà deve davvero comportarsi come una proprietà nativa WPF del controllo?”

Se la risposta è no, fermati e rivaluta.

## Un confronto molto utile

### Candidato forte a dependency property
`Titolo` di un `UserControl` riusabile, bindabile e configurabile da XAML.

### Cattivo candidato
`_cacheInternaCalcolo` o `NumeroTentativiRefresh` usati solo internamente dal controllo.

### Altro cattivo candidato
`Nome` o `Città` in un view model o in una classe dati.

## Perché questo punto è così importante

Perché una delle confusioni più frequenti è mescolare:
- livello UI WPF;
- livello view model;
- livello dominio.

Le dependency property appartengono soprattutto al primo livello.

## Errori tipici

### Usarle nei view model “perché così tutto è uniforme”
In realtà stai mescolando livelli diversi.

### Dichiarare come dependency property anche dettagli interni
Il controllo diventa più pesante senza bisogno.

### Pensare che siano un sostituto generale di `INotifyPropertyChanged`
Non è il loro ruolo.

## Regola pratica

Usa una dependency property solo quando la proprietà deve partecipare davvero al sistema UI di WPF.
Se il vantaggio non è concreto, una proprietà CLR normale è spesso meglio.
