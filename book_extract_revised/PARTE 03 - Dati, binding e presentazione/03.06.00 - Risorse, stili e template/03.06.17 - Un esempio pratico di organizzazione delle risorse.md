# 03.06.17 - Un esempio pratico di organizzazione delle risorse

## Dalla teoria alla struttura reale di progetto

A questo punto del capitolo, il lettore ha visto:
- risorse semplici;
- stili;
- template;
- dizionari separati;
- `MergedDictionaries`.

Ora conviene mettere tutto insieme in un esempio di organizzazione pratica.

L'obiettivo non è creare la struttura perfetta universale, ma mostrare una struttura sana e leggibile per un progetto reale.

## Un'organizzazione possibile

Immagina questa cartella nel progetto:

- `Resources/Colors.xaml`
- `Resources/Brushes.xaml`
- `Resources/Styles.Buttons.xaml`
- `Resources/Styles.Inputs.xaml`
- `Resources/Templates.xaml`
- `Resources/Converters.xaml`

Questa struttura ha una logica molto leggibile:
- colori e brush separati;
- stili distinti per famiglie di controlli;
- template in un file dedicato;
- converter in un loro file.

## Esempio di `App.xaml`

```xml
<Application.Resources>
    <ResourceDictionary>
        <ResourceDictionary.MergedDictionaries>
            <ResourceDictionary Source="Resources/Colors.xaml"/>
            <ResourceDictionary Source="Resources/Brushes.xaml"/>
            <ResourceDictionary Source="Resources/Styles.Buttons.xaml"/>
            <ResourceDictionary Source="Resources/Styles.Inputs.xaml"/>
            <ResourceDictionary Source="Resources/Templates.xaml"/>
            <ResourceDictionary Source="Resources/Converters.xaml"/>
        </ResourceDictionary.MergedDictionaries>
    </ResourceDictionary>
</Application.Resources>
```

## Esempio di `Brushes.xaml`

```xml
<ResourceDictionary xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
                    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml">

    <SolidColorBrush x:Key="PrimaryBrush" Color="#2D6CDF"/>
    <SolidColorBrush x:Key="DangerBrush" Color="#C62828"/>

</ResourceDictionary>
```

## Esempio di `Styles.Buttons.xaml`

```xml
<ResourceDictionary xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
                    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml">

    <Style x:Key="PrimaryButtonStyle" TargetType="Button">
        <Setter Property="Background" Value="{StaticResource PrimaryBrush}"/>
        <Setter Property="Foreground" Value="White"/>
        <Setter Property="Padding" Value="12,6"/>
        <Setter Property="MinWidth" Value="120"/>
    </Style>

</ResourceDictionary>
```

## Esempio di `Converters.xaml`

```xml
<ResourceDictionary xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
                    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
                    xmlns:local="clr-namespace:MioProgetto.Converters">

    <local:BoolToVisibilityConverter x:Key="BoolToVisibilityConverter"/>

</ResourceDictionary>
```

## Perché questa struttura è buona

Perché separa per responsabilità.
Chi legge il progetto capisce subito:
- dove cercare un brush;
- dove cercare uno stile;
- dove cercare un template;
- dove cercare un converter.

Questo è molto meglio di un unico grande contenitore confuso.

## Quando adattarla

Naturalmente ogni progetto può richiedere varianti.
Per esempio:
- in progetti piccoli potresti avere meno file;
- in progetti grandi potresti dividere ancora di più per modulo o area funzionale.

Il punto non è copiare una struttura rigida.
Il punto è capire il criterio:
**separa in modo leggibile, non in modo casuale**.

## Errori tipici

### Tutto in `App.xaml`
Comodo all'inizio, ingestibile dopo.

### Troppi file minuscoli senza logica chiara
Si perde il vantaggio della separazione.

### Nomi dei file incoerenti
Chi legge non capisce dove cercare.

## Regola pratica

Una buona organizzazione delle risorse deve permettere a chi legge il progetto di trovare rapidamente:
- che cosa esiste;
- dove si trova;
- come viene riusato.

Se questa ricerca è facile, la struttura sta funzionando bene.
