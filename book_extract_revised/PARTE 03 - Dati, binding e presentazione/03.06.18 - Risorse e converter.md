# 03.06.18 - Risorse e converter

## Perché i converter vivono così bene tra le risorse

I converter sono uno dei casi più naturali di oggetti che meritano di diventare risorse.

Il motivo è semplice:
- vengono usati nel markup;
- devono essere richiamabili da più binding;
- hanno senso come oggetti riusabili;
- appartengono al livello di presentazione.

Per questo, in WPF, è normalissimo registrarli nelle risorse.

## Un esempio classico

Converter:

```csharp
public class BoolToVisibilityConverter : IValueConverter
{
    public object Convert(object value, Type targetType, object parameter, CultureInfo culture)
    {
        if (value is bool flag && flag)
            return Visibility.Visible;

        return Visibility.Collapsed;
    }

    public object ConvertBack(object value, Type targetType, object parameter, CultureInfo culture)
    {
        throw new NotSupportedException();
    }
}
```

Registrazione nelle risorse:

```xml
<Window.Resources>
    <local:BoolToVisibilityConverter x:Key="BoolToVisibilityConverter"/>
</Window.Resources>
```

Uso:

```xml
<TextBlock Visibility="{Binding MostraMessaggio,
                                Converter={StaticResource BoolToVisibilityConverter}}" />
```

## Perché questa soluzione è così naturale

Perché il converter:
- è un oggetto dichiarativo;
- non deve essere creato a mano in ogni binding;
- può essere condiviso da più controlli;
- si integra perfettamente col sistema di risorse.

In pratica, le risorse sono il posto giusto in cui i converter “vivono” nel progetto XAML.

## Dove dichiararli

Dipende da quanto sono condivisi.

### Solo in una finestra
`Window.Resources`

### Riutilizzati in più schermate
`App.xaml` o un `ResourceDictionary` dedicato

### Organizzazione più pulita
Spesso si usa un file come `Converters.xaml`

Questa è una pratica molto sensata nei progetti reali.

## Perché conviene separarli

Se i converter iniziano a crescere, tenerli tutti mescolati a brush, stili e template può appesantire la lettura.
Un dizionario dedicato li rende più facili da trovare e manutenere.

## Un esempio con dizionario dedicato

```xml
<ResourceDictionary xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
                    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
                    xmlns:local="clr-namespace:MioProgetto.Converters">

    <local:BoolToVisibilityConverter x:Key="BoolToVisibilityConverter"/>
    <local:EuroFormatConverter x:Key="EuroFormatConverter"/>

</ResourceDictionary>
```

## Un punto importante sulla responsabilità

Il fatto che un converter sia una risorsa rafforza bene la sua natura:
non è logica di dominio, ma un componente di presentazione riusabile.

Questo è un buon modo di pensarci.

## Errori tipici

### Crearli “al volo” mentalmente senza considerarli risorse di progetto
Si perde coerenza.

### Tenerli sparsi in più finestre quando sono generici
Si duplicano.

### Metterli in risorse globali anche quando sono ultra-specifici
Anche qui serve misura.

## Regola pratica

Se un converter viene usato dal markup e ha senso riusarlo, trattalo come una risorsa vera e propria.
È quasi sempre il posto giusto per lui.
