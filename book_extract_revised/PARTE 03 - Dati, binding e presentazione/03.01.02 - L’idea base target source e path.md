# 03.01.02 - L’idea base: target, source e path

## Il binding diventa semplice quando vedi i suoi tre pezzi

Molti trovano il data binding difficile perché lo vedono come una sintassi compatta e un po' astratta.

In realtà, quasi tutto il binding in WPF si può capire bene se lo scomponi in tre elementi:

- **target**
- **source**
- **path**

Questi sono i tre pezzi fondamentali.

## Il target

Il **target** è il controllo o, più precisamente, la proprietà del controllo che riceve o mostra un valore.

Per esempio:

```xml
<TextBox Text="{Binding Nome}" />
```

Qui il target non è genericamente la `TextBox`, ma la sua proprietà `Text`.

Altro esempio:

```xml
<TextBlock Text="{Binding Cognome}" />
```

Qui il target è la proprietà `Text` del `TextBlock`.

## La source

La **source** è l'oggetto da cui arriva il dato.

Può essere:
- l'oggetto nel `DataContext`;
- un oggetto esplicitamente assegnato;
- un elemento della UI;
- una risorsa;
- un'altra origine specifica.

Nella maggior parte dei casi iniziali, la source sarà semplicemente l'oggetto esposto tramite `DataContext`.

## Il path

Il **path** è il percorso che indica quale proprietà leggere o scrivere nella source.

Esempio:

```xml
<TextBox Text="{Binding Nome}" />
```

Qui il path è `Nome`.

Se hai una struttura più ricca, può anche essere qualcosa come:

```xml
<TextBlock Text="{Binding Cliente.Città}" />
```

Qui il path punta a una proprietà annidata.

## Un esempio completo molto semplice

Supponiamo di avere questa classe:

```csharp
public class Persona
{
    public string Nome { get; set; }
}
```

E nel code-behind:

```csharp
public MainWindow()
{
    InitializeComponent();

    DataContext = new Persona
    {
        Nome = "Anna"
    };
}
```

XAML:

```xml
<TextBox Text="{Binding Nome}" Width="200" />
```

In questo caso:
- target = `TextBox.Text`
- source = l'oggetto `Persona` assegnato al `DataContext`
- path = `Nome`

Questa tripla lettura è una delle chiavi più utili del capitolo.

## Perché è così importante

Se un binding non funziona, quasi sempre uno dei problemi sta qui:
- il target non è la proprietà giusta;
- la source non è quella che pensavi;
- il path è errato o non esiste.

Per questo conviene abituarsi da subito a ragionare in questi termini.

## Un secondo esempio utile

```xml
<CheckBox Content="Cliente attivo"
          IsChecked="{Binding Attivo}" />
```

Qui:
- target = `CheckBox.IsChecked`
- source = l'oggetto nel `DataContext`
- path = `Attivo`

Il principio è sempre lo stesso, anche se cambia il controllo.

## Il binding non è un “cavo invisibile qualsiasi”

Molti all'inizio immaginano il binding come qualcosa di un po' magico che “collega cose”.
È meglio pensarlo come una regola precisa:
una proprietà target si collega a una proprietà della source seguendo un path.

Questa visione lo rende molto più leggibile.

## Quando il path diventa più interessante

Il path non deve per forza essere una proprietà semplice.

Può essere:
- una proprietà annidata;
- una collezione;
- un valore corrente di una struttura più grande;
- in casi più avanzati, anche qualcosa di più articolato.

Ma all'inizio conviene fissare molto bene il modello più semplice:
**proprietà target ↔ proprietà source**.

## Errori tipici

### Pensare solo al controllo e non alla sua proprietà
Il binding lavora su proprietà, non su controlli in astratto.

### Non sapere qual è la source reale
Molti errori di binding nascono da un `DataContext` inatteso.

### Scrivere un path sbagliato
Anche un piccolo errore di nome basta a rompere il collegamento.

## Regola pratica

Quando leggi o scrivi un binding, chiediti sempre:
- qual è il target?
- qual è la source?
- qual è il path?

Se questi tre punti sono chiari, metà del lavoro mentale è già fatto.
