# 03.06.09 - Lo stile: applicare un insieme di proprietà senza ripeterle

## Il passo successivo naturale dopo le risorse semplici

Dopo aver visto brush e valori condivisi, arriva uno strumento ancora più importante:
lo stile.

Uno stile serve a raccogliere un insieme di proprietà da applicare a uno o più controlli senza doverle riscrivere ogni volta.

Questo è uno dei modi più efficaci per mantenere coerenza visiva e ridurre duplicazioni.

## Il problema che risolve

Senza stile, è facile ritrovarsi con markup ripetitivo:

```xml
<Button Background="#2D6CDF"
        Foreground="White"
        Padding="12,6"
        MinWidth="120"
        Content="Salva"/>

<Button Background="#2D6CDF"
        Foreground="White"
        Padding="12,6"
        MinWidth="120"
        Content="Conferma"/>
```

Funziona, ma stai ripetendo la stessa decisione visiva più volte.

Con uno stile, quelle proprietà comuni vivono in un punto solo.

## Esempio base

```xml
<Window.Resources>
    <Style x:Key="PrimaryButtonStyle" TargetType="Button">
        <Setter Property="Background" Value="#2D6CDF"/>
        <Setter Property="Foreground" Value="White"/>
        <Setter Property="Padding" Value="12,6"/>
        <Setter Property="MinWidth" Value="120"/>
    </Style>
</Window.Resources>
```

Uso:

```xml
<Button Style="{StaticResource PrimaryButtonStyle}" Content="Salva"/>
<Button Style="{StaticResource PrimaryButtonStyle}" Content="Conferma"/>
```

## Come leggere bene uno stile

### `TargetType`
Indica il tipo di controllo a cui lo stile si applica.

### `Setter`
Ogni setter definisce una proprietà e il valore da assegnare.

In pratica, uno stile è un insieme dichiarativo di proprietà condivise.

## Perché lo stile è così importante

Perché fa tre cose molto preziose:
- elimina duplicazioni;
- centralizza scelte visive;
- rende la UI molto più coerente.

In progetti reali, questo è uno dei modi più forti per migliorare la manutenzione del markup.

## Quando usare uno stile

Usalo quando:
- più controlli condividono lo stesso aspetto o comportamento base;
- vuoi definire una grammatica visiva coerente;
- vuoi evitare ripetizioni;
- vuoi poter modificare facilmente l'aspetto comune di un gruppo di controlli.

## Quando non serve complicare tutto

Se una proprietà è davvero unica e locale a un solo controllo, non serve forzare uno stile.
Come sempre, la misura conta.

Ma appena la ripetizione compare, lo stile diventa un ottimo candidato.

## Un esempio realistico

```xml
<Window.Resources>
    <Style x:Key="SecondaryTextBoxStyle" TargetType="TextBox">
        <Setter Property="Padding" Value="8,4"/>
        <Setter Property="Margin" Value="0,0,0,10"/>
        <Setter Property="MinWidth" Value="220"/>
    </Style>
</Window.Resources>

<TextBox Style="{StaticResource SecondaryTextBoxStyle}"/>
<TextBox Style="{StaticResource SecondaryTextBoxStyle}"/>
<TextBox Style="{StaticResource SecondaryTextBoxStyle}"/>
```

Qui il vantaggio è subito evidente.

## Errore tipico

L'errore classico è continuare a copiare blocchi di proprietà uguali in più controlli invece di fermarsi a creare uno stile.

## Regola pratica

Quando un insieme di proprietà inizia a ripetersi, molto spesso è il momento di trasformarlo in uno stile.
