# 03.02.02 - Più oggetti diversi nella stessa finestra

## Quando il caso semplice non basta più

Il modello “un solo oggetto nel `DataContext`” è ottimo, ma non copre tutto.
Ci sono finestre reali in cui convivono più dati di natura diversa.

Per esempio:
- un cliente corrente;
- una lista di ordini;
- un riepilogo;
- uno stato della schermata;
- un pannello filtri.

Qui sorge una domanda importante:
come gestire più oggetti diversi nella stessa finestra senza creare confusione?

## Prima idea corretta: non improvvisare

La prima tentazione del principiante è questa:
- mettere un `DataContext` qui;
- poi un altro lì;
- poi aggiungere qualche binding con source esplicita;
- poi qualche riferimento diretto a controllo.

Il risultato può funzionare, ma diventa rapidamente difficile da leggere.

La soluzione migliore è sempre partire da una struttura pensata.

## La scelta più sana: un oggetto contenitore

Molto spesso la soluzione più pulita è usare un oggetto principale che contiene più proprietà.

Per esempio:

```csharp
public class FinestraClienteViewModel
{
    public Cliente ClienteCorrente { get; set; } = new();
    public List<Ordine> Ordini { get; set; } = new();
    public string TitoloFinestra { get; set; } = "";
}
```

Nel code-behind o nel view model principale:

```csharp
DataContext = new FinestraClienteViewModel
{
    ClienteCorrente = new Cliente
    {
        Nome = "Anna",
        Cognome = "Rossi"
    },
    Ordini = new List<Ordine>
    {
        new Ordine { Numero = "A001" },
        new Ordine { Numero = "A002" }
    },
    TitoloFinestra = "Scheda cliente"
};
```

XAML:

```xml
<Grid Margin="20">
    <Grid.RowDefinitions>
        <RowDefinition Height="Auto"/>
        <RowDefinition Height="Auto"/>
        <RowDefinition Height="*"/>
    </Grid.RowDefinitions>

    <TextBlock Text="{Binding TitoloFinestra}"
               FontSize="22"
               FontWeight="SemiBold"
               Margin="0,0,0,12"/>

    <TextBox Grid.Row="1"
             Text="{Binding ClienteCorrente.Nome}"
             Width="220"
             Margin="0,0,0,12"/>

    <ListBox Grid.Row="2"
             ItemsSource="{Binding Ordini}"
             DisplayMemberPath="Numero"/>
</Grid>
```

## Perché questa è spesso la scelta migliore

Perché mantieni:
- un solo `DataContext` principale;
- una struttura leggibile;
- nomi chiari;
- una finestra organizzata attorno a un oggetto radice coerente.

Questo è molto meglio di una giungla di `DataContext` secondari non pianificati.

## Altre possibilità esistono, ma vanno usate con criterio

Puoi anche avere:
- contenitori con `DataContext` diversi;
- binding con `Source`;
- binding con `ElementName`;
- binding con `RelativeSource`.

Ma prima conviene chiedersi:
“non posso semplicemente costruire un oggetto principale più chiaro?”

Molto spesso la risposta è sì.

## Un esempio realistico

Immagina una finestra ordine con:
- testata ordine;
- righe ordine;
- filtro di ricerca articoli;
- stato di caricamento.

Un contenitore principale come questo è spesso molto sensato:

```csharp
public class OrdineViewModel
{
    public Documento DocumentoCorrente { get; set; } = new();
    public List<RigaDocumento> Righe { get; set; } = new();
    public string TestoRicercaArticolo { get; set; } = "";
    public bool InCaricamento { get; set; }
}
```

Questo evita di frammentare troppo il senso della schermata.

## Quando conviene davvero avere più oggetti in gioco

Conviene quando la finestra ha parti con responsabilità diverse.
Ma questo non significa automaticamente che devi avere tanti `DataContext` sparsi.
Molto spesso basta un oggetto radice che li organizza.

## Errori tipici

### Usare più oggetti senza una struttura chiara
La finestra diventa difficile da leggere.

### Cambiare continuamente `DataContext` perché “così il binding è più corto”
Il codice diventa meno prevedibile.

### Non dare un nome logico ai sotto-oggetti
`Oggetto1`, `Dati`, `Info`, `Varie` sono nomi poco utili.

## Regola pratica

Se nella stessa finestra ci sono più oggetti diversi, la prima scelta da considerare è quasi sempre questa:
usa un oggetto radice che li raccolga e li renda leggibili tramite proprietà ben nominate.
