# 03.01.07 - Perché a volte il binding funziona solo a metà

## Uno dei primi misteri reali del data binding

Questo è uno dei punti in cui molti lettori si bloccano per davvero.

Vedono che:
- il dato iniziale compare bene nella UI;
- l'utente modifica una `TextBox`;
- il valore sembra passare all'oggetto;

e quindi pensano che tutto sia a posto.

Poi cambiano la proprietà da codice e la UI non si aggiorna come si aspettavano.

Da qui nasce la sensazione che il binding “funzioni solo a metà”.

## Che cosa sta succedendo davvero

Nella maggior parte dei casi, il problema è questo:
il binding sa leggere il valore iniziale e, con `TwoWay`, può anche riportare modifiche dal controllo alla source.

Ma se la source cambia **dopo**, la UI non ha alcun modo automatico di sapere che deve aggiornarsi, a meno che il modello non notifichi il cambiamento.

Questo è esattamente il motivo per cui esiste `INotifyPropertyChanged`.

## Un esempio classico

Supponiamo di avere:

```csharp
public class Persona
{
    public string Nome { get; set; }
}
```

E il `DataContext`:

```csharp
private Persona _persona;

public MainWindow()
{
    InitializeComponent();

    _persona = new Persona
    {
        Nome = "Anna"
    };

    DataContext = _persona;
}
```

XAML:

```xml
<TextBlock Text="{Binding Nome}" FontSize="18"/>
```

Se dopo scrivi:

```csharp
_persona.Nome = "Luca";
```

potresti aspettarti che il `TextBlock` si aggiorni subito.
Ma non succede.

## Perché la UI non si aggiorna

Perché il binding non legge continuamente tutte le proprietà per vedere se qualcosa è cambiato.
Sarebbe inefficiente e poco elegante.

Invece si aspetta che l'oggetto dica:
“questa proprietà è cambiata”.

Se l'oggetto non lo comunica, la UI resta con il vecchio valore.

## Cosa significa “funziona solo a metà”

Nella pratica significa questo:

### Funziona in lettura iniziale
Quando la UI viene costruita, il valore compare correttamente.

### Funziona spesso in scrittura dalla UI alla source
Con `TwoWay`, l'utente modifica il campo e l'oggetto riceve il nuovo valore.

### Non funziona bene in aggiornamento dinamico source → UI
Se cambi il modello da codice, la UI non si aggiorna automaticamente.

Ed è qui che il binding appare “mezzo rotto”, anche se in realtà sta solo aspettando una notifica che non arriva.

## Un'altra causa frequente

A volte il binding sembra funzionare a metà anche per altri motivi:
- `Mode` sbagliato;
- `UpdateSourceTrigger` non compreso;
- `DataContext` non corretto;
- path errato;
- proprietà non pubblica o scritta male.

Ma tra tutte le cause, la mancanza di `INotifyPropertyChanged` è una delle più importanti.

## Perché questo passaggio è così formativo

Perché fa capire che il binding non è solo “collegare due proprietà”.
È anche un patto:
- WPF si occupa del collegamento;
- ma l'oggetto deve notificare i cambiamenti quando serve.

Questa collaborazione è il cuore del modello.

## Errori tipici

### Pensare che il binding osservi automaticamente ogni cambio di proprietà
Non funziona così.

### Credere che `TwoWay` basti da solo
`TwoWay` non sostituisce la notifica dei cambiamenti.

### Pensare che sia un bug di WPF
Molto spesso è semplicemente il modello che non notifica.

## Regola pratica

Se il binding mostra il valore iniziale ma non aggiorna la UI quando cambi l'oggetto da codice, quasi sempre devi pensare subito a `INotifyPropertyChanged`.
