# 03.01.09 - Esempio completo con aggiornamento reale della UI

## Ora mettiamo insieme i pezzi importanti

Dopo aver introdotto `DataContext`, `TwoWay` binding e `INotifyPropertyChanged`, è il momento di vedere un esempio completo che mostri davvero l'aggiornamento automatico della UI.

L'obiettivo è questo:
- l'utente modifica un valore;
- il modello si aggiorna;
- il modello cambia anche da codice;
- la UI riflette tutto in tempo reale.

Qui il binding smette di essere un concetto teorico e diventa comportamento visibile.

## La classe del modello

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;

public class Persona : INotifyPropertyChanged
{
    private string _nome = "";
    private string _città = "";

    public string Nome
    {
        get => _nome;
        set
        {
            if (_nome != value)
            {
                _nome = value;
                OnPropertyChanged();
            }
        }
    }

    public string Città
    {
        get => _città;
        set
        {
            if (_città != value)
            {
                _città = value;
                OnPropertyChanged();
            }
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

## Il code-behind

```csharp
public partial class MainWindow : Window
{
    private Persona _persona;

    public MainWindow()
    {
        InitializeComponent();

        _persona = new Persona
        {
            Nome = "Anna",
            Città = "Torino"
        };

        DataContext = _persona;
    }

    private void CambiaDaCodice_Click(object sender, RoutedEventArgs e)
    {
        _persona.Nome = "Luca";
        _persona.Città = "Aosta";
    }
}
```

## Lo XAML

```xml
<StackPanel Margin="20">
    <TextBlock Text="Nome" Margin="0,0,0,4"/>
    <TextBox Text="{Binding Nome, Mode=TwoWay}"
             Width="220"
             Margin="0,0,0,10"/>

    <TextBlock Text="Città" Margin="0,0,0,4"/>
    <TextBox Text="{Binding Città, Mode=TwoWay}"
             Width="220"
             Margin="0,0,0,12"/>

    <TextBlock Text="{Binding Nome, StringFormat=Cliente: {0}}"
               FontWeight="SemiBold"
               Margin="0,0,0,4"/>

    <TextBlock Text="{Binding Città, StringFormat=Città: {0}}"
               Margin="0,0,0,12"/>

    <Button Content="Cambia da codice"
            Width="150"
            Click="CambiaDaCodice_Click"/>
</StackPanel>
```

## Che cosa mostra questo esempio

Questo esempio è molto importante perché mostra tutte e due le direzioni del binding in modo visibile.

### 1. La UI legge il modello
All'avvio, i campi mostrano `"Anna"` e `"Torino"`.

### 2. La UI modifica il modello
Se l'utente scrive nei `TextBox`, le proprietà si aggiornano.

### 3. Il modello modifica la UI
Quando premi il pulsante, il code-behind cambia le proprietà del modello e i controlli si aggiornano automaticamente.

Questo è esattamente il punto che mancava nei modelli senza notifica.

## Perché è un esempio formativo

Perché mostra che il binding non è solo “scrivo qualcosa nello XAML”.
Mostra una collaborazione tra:
- modello;
- meccanismo di notifica;
- controlli;
- `DataContext`.

Ed è proprio questa collaborazione che rende WPF così potente sul piano della presentazione dei dati.

## Dove si vede il vero vantaggio

Il vantaggio reale è che non devi fare codice come questo:

```csharp
txtNome.Text = _persona.Nome;
txtCittà.Text = _persona.Città;
```

ogni volta che qualcosa cambia.

La UI reagisce da sola, a patto che il modello notifichi correttamente.

## Errori tipici

### Copiare l'esempio ma togliere `INotifyPropertyChanged`
A quel punto metà del comportamento sparisce.

### Dimenticare il `DataContext`
Il binding non trova le proprietà.

### Pensare che il pulsante aggiorni direttamente la UI
No: aggiorna il modello, e la UI segue grazie al binding.

## Regola pratica

Quando vuoi verificare se hai davvero capito il data binding, fai sempre un test doppio:
- cambia il dato dalla UI;
- cambia il dato dal codice.

Se entrambe le direzioni funzionano come previsto, il modello mentale è corretto.
