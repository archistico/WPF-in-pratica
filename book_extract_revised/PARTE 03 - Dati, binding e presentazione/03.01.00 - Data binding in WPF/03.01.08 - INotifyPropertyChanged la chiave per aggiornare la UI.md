# 03.01.08 - `INotifyPropertyChanged`: la chiave per aggiornare la UI

## Qui il binding diventa davvero vivo

Se c'è un'interfaccia che devi imparare presto in WPF, è `INotifyPropertyChanged`.

Il motivo è semplice:
è uno dei meccanismi principali che permette alla UI di sapere che una proprietà della source è cambiata.

Senza di essa, il binding legge bene il valore iniziale, ma fatica a reagire agli aggiornamenti dinamici del modello.

Con essa, la UI diventa molto più viva e coerente.

## Che cosa fa in pratica

`INotifyPropertyChanged` permette a un oggetto di notificare che una sua proprietà è cambiata.

WPF ascolta questa notifica e, se c'è un binding interessato, aggiorna il target.

Questa è una delle basi tecniche più importanti di tutto il modello di presentazione in WPF.

## Implementazione minima

Ecco un esempio classico:

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;

public class Persona : INotifyPropertyChanged
{
    private string _nome;

    public string Nome
    {
        get => _nome;
        set
        {
            if (_nome != value)
            {
                _nome = value;
                OnPropertyChanged();
            }
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

## Come leggere bene questo codice

### 1. La proprietà usa un campo privato
Non usiamo più l'auto-property semplice, perché dobbiamo intercettare il cambio di valore.

### 2. Il setter controlla se il valore è davvero cambiato
Questo evita notifiche inutili.

### 3. `OnPropertyChanged()` emette l'evento
È questo il momento in cui la UI viene informata.

### 4. `CallerMemberName` semplifica la chiamata
Non serve scrivere il nome della proprietà a mano ogni volta.

## Perché questa implementazione è così importante

Perché è il ponte tra:
- il modello oggetto;
- il sistema di binding di WPF.

Se questa notifica manca, WPF non sa che la proprietà è cambiata.

## Un esempio pratico

```csharp
private Persona _persona;

public MainWindow()
{
    InitializeComponent();

    _persona = new Persona
    {
        Nome = "Anna"
    };

    DataContext = _persona;
}

private void CambiaNome()
{
    _persona.Nome = "Luca";
}
```

Se `Persona` implementa `INotifyPropertyChanged`, la UI aggiornerà automaticamente i controlli bindati a `Nome`.

Se non lo implementa, la UI resterà col valore precedente.

## Quando serve davvero

Serve praticamente ogni volta che:
- una proprietà può cambiare dopo il caricamento iniziale;
- vuoi che la UI rifletta quei cambiamenti;
- stai costruendo una schermata che deve restare sincronizzata con lo stato dell'oggetto.

## Quando non serve in senso stretto

Se il valore viene assegnato una sola volta prima della visualizzazione e poi non cambia più, il binding base può bastare.

Ma nelle applicazioni reali, questa situazione è meno frequente di quanto sembri.

## Errori tipici

### Continuare a usare auto-property semplici su modelli che devono notificare
La UI non si aggiorna come atteso.

### Dimenticare la chiamata a `OnPropertyChanged()`
Il codice sembra corretto, ma il binding non reagisce.

### Notificare il nome proprietà sbagliato
Errore sottile ma reale, se non usi `CallerMemberName`.

## Regola pratica

Se una proprietà deve poter cambiare e la UI deve accorgersene, `INotifyPropertyChanged` non è un dettaglio: è il meccanismo chiave.
