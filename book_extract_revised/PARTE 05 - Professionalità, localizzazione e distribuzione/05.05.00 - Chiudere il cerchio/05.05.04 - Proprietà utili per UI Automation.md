# 05.05.04 - Proprietà utili per UI Automation

## L'interfaccia non deve parlare solo agli occhi

Una schermata accessibile non deve essere comprensibile solo visivamente.
Deve anche esporre abbastanza informazione agli strumenti che leggono o automatizzano la UI.

Qui entra in gioco il mondo di **UI Automation**.

Non serve trasformare questo capitolo in una trattazione specialistica completa, ma è importante che il lettore capisca il principio:
una UI WPF ben progettata non mostra soltanto contenuti, ma li rende anche riconoscibili agli strumenti assistivi e di automazione.

## Che cos'è UI Automation in pratica

UI Automation è il modello attraverso cui strumenti esterni possono interrogare e usare l'interfaccia:
- screen reader;
- strumenti di test UI;
- software assistivi;
- tool di automazione.

Per farlo bene, questi strumenti hanno bisogno di nomi, ruoli e informazioni coerenti sui controlli.

## Perché questo riguarda anche chi non usa screen reader

Perché le stesse informazioni che aiutano l'accessibilità aiutano spesso anche:
- test automatici dell'interfaccia;
- strumenti di quality assurance;
- diagnostica;
- manutenzione di schermate complesse.

Quindi è un tema più ampio di quanto sembri.

## Un principio molto semplice

Se un controllo svolge un ruolo importante, dovrebbe essere identificabile in modo chiaro anche oltre l'aspetto visivo.

Per esempio:
- un pulsante con sola icona dovrebbe avere un nome comprensibile;
- una casella di input dovrebbe essere chiaramente associata alla sua funzione;
- una lista o una griglia dovrebbero esporre una struttura sensata.

## Il ruolo di `AutomationProperties`

In WPF esiste una serie di proprietà utili sotto `AutomationProperties`.

Per esempio:

```xml
<Button Content="Salva"
        AutomationProperties.Name="Salva il documento" />
```

oppure:

```xml
<TextBox AutomationProperties.Name="Nome cliente" />
```

Queste proprietà aiutano gli strumenti assistivi a comprendere meglio il controllo.

## Quando sono particolarmente utili

Sono particolarmente utili quando:
- il testo visibile non basta;
- il controllo usa solo icone;
- il significato non è chiaro dal contesto;
- la UI è molto personalizzata;
- vuoi migliorare leggibilità per strumenti assistivi o di automazione.

## Un altro esempio utile

```xml
<Button Width="40"
        Height="40"
        ToolTip="Cerca"
        AutomationProperties.Name="Avvia ricerca">
    <TextBlock Text="🔍" />
</Button>
```

Qui il contenuto visivo da solo potrebbe non essere abbastanza robusto per tutti gli strumenti.
L'attributo di automazione aiuta molto.

## Il rapporto con etichette e contesto

Naturalmente non tutto va risolto con proprietà di automazione aggiunte ovunque.
Molto spesso conviene prima progettare bene:
- etichette visibili;
- struttura chiara;
- naming sensato;
- layout leggibile.

Le proprietà di automazione diventano allora un rafforzamento, non un rimedio disperato.

## Quando una UI personalizzata è più a rischio

Più la UI viene personalizzata con:
- template complessi;
- icone senza testo;
- controlli compositi;
- riduzione delle etichette;

più cresce il rischio che il significato del controllo diventi chiaro solo “a occhio”.
Ed è proprio lì che le proprietà di automazione diventano preziose.

## Un criterio molto utile

Chiediti:
- questo controllo ha un significato chiaro anche per uno strumento assistivo?
- il suo ruolo e il suo nome sono leggibili?
- se togliessi il contesto visivo, resterebbe comprensibile?

Se la risposta è no, probabilmente conviene intervenire.

## Regola pratica

Le proprietà utili per UI Automation servono a dare ai controlli un'identità più chiara per strumenti assistivi e di test.
In WPF, usarle bene significa rendere la UI non solo visibile, ma anche interpretabile in modo più robusto.
