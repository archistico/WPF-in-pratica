# 05.01.03 - Le liste non si aggiornano

## Il problema parallelo a quello delle proprietà singole

Dopo il classico problema “la UI non si aggiorna”, arriva il suo fratello per le collezioni:
la lista viene mostrata, ma quando aggiungi o rimuovi elementi la UI non cambia come dovrebbe.

Questo errore è molto frequente con:
- `ListBox`
- `ListView`
- `DataGrid`
- `ComboBox`
- `ItemsControl`

e quasi sempre porta alla stessa domanda:
**che tipo di raccolta sto usando?**

## Il caso tipico

Supponiamo di avere:

```csharp
public List<string> Nomi { get; } = new();
```

e nella UI:

```xml
<ListBox ItemsSource="{Binding Nomi}" />
```

All'inizio tutto sembra corretto.
Poi fai:

```csharp
Nomi.Add("Anna");
```

ma la `ListBox` non mostra il nuovo elemento come ti aspetti.

## Perché succede

Perché una `List<T>` normale non notifica i cambiamenti strutturali della collection alla UI.

Per questo, in scenari dinamici di WPF, la scelta naturale è quasi sempre `ObservableCollection<T>`.

## La soluzione corretta

```csharp
using System.Collections.ObjectModel;

public ObservableCollection<string> Nomi { get; } = new();
```

Ora, se fai:

```csharp
Nomi.Add("Anna");
Nomi.Add("Luca");
```

la UI può aggiornarsi automaticamente.

## Il modello mentale corretto

### Proprietà singola
Ha bisogno di `INotifyPropertyChanged`.

### Collezione che cambia struttura
Ha bisogno spesso di `ObservableCollection<T>`.

Questa distinzione è fondamentale e va tenuta fermissima.

## Un secondo livello del problema

Attenzione: `ObservableCollection<T>` notifica:
- aggiunte;
- rimozioni;
- cambiamenti della collection.

Ma non sostituisce `INotifyPropertyChanged` sugli elementi interni.

Quindi, se hai:

```csharp
public ObservableCollection<Cliente> Clienti { get; } = new();
```

e cambi:

```csharp
Clienti[0].Nome = "Luca";
```

la UI si aggiornerà bene solo se `Cliente` implementa `INotifyPropertyChanged`.

Questo è uno dei punti più importanti da ricordare.

## Un caso realistico con `DataGrid`

Se una griglia è bindata a una `ObservableCollection<Prodotto>`:
- l'aggiunta di una nuova riga può comparire subito;
- ma il cambio del nome o del prezzo di una riga richiede anche notifica della proprietà nel tipo `Prodotto`.

Molti errori nascono proprio dal confondere questi due livelli.

## Un piccolo test mentale utile

Se la lista:
- mostra i dati iniziali;
- ma non reagisce a `Add` o `Remove`,

pensa subito alla collection.

Se invece:
- la lista si aggiorna quando aggiungi elementi;
- ma non aggiorna i campi di un elemento già presente,

pensa agli oggetti interni e a `INotifyPropertyChanged`.

## Errore tipico reale

Molti lettori usano `ObservableCollection<T>` ma lasciano il modello come classe con auto-property semplici.
Così la lista si aggiorna “a metà”: bene sulla struttura, male sui dettagli.

## Regola pratica

Se una lista WPF non si aggiorna, chiediti subito:
- sto usando `ObservableCollection<T>`?
- gli elementi interni notificano i cambi di proprietà?

Nella maggior parte dei casi, il problema è esattamente lì.
