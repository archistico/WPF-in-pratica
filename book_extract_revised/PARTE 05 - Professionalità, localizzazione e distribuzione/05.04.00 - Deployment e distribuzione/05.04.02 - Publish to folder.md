# 05.04.02 - Publish to folder

## La forma più diretta di pubblicazione

`Publish to folder` è il modo più semplice per ottenere una versione distribuibile della tua applicazione WPF.

In pratica, Visual Studio o il comando `dotnet publish` generano una cartella con i file necessari all'esecuzione dell'app. citeturn333254search3turn333254search6

Questa cartella poi può essere:
- copiata manualmente;
- messa su una share di rete;
- zippata;
- passata a un installer esterno;
- usata come output di rilascio tecnico.

## Che cosa produce davvero

La pubblicazione a cartella genera l'output dell'app in forma pronta alla distribuzione, secondo le opzioni di publish che scegli:
- runtime target;
- modalità framework-dependent o self-contained;
- eventuale single-file;
- altre proprietà di publish. citeturn333254search3turn333254search6turn333254search10

## Quando è utile

È molto utile quando:
- distribuisci internamente;
- vuoi controllare tu la consegna dei file;
- non hai bisogno di una piattaforma di installazione complessa;
- preferisci una soluzione lineare e trasparente;
- vuoi poi costruire un installer con altri strumenti.

## Che cosa non ti dà da solo

`Publish to folder` non è, di per sé, un sistema completo di installazione gestita o aggiornamenti automatici come ClickOnce. Ti dà i file pubblicati, non un modello completo di distribuzione con policy di update integrate. citeturn333254search1turn333254search3

Questo punto va tenuto molto fermo.

## Un esempio mentale semplice

Puoi pensarlo così:
- build → genera output di sviluppo;
- publish to folder → genera output di distribuzione.

Questa è una semplificazione utile.

## Come si usa in pratica

In Visual Studio puoi creare un profilo di publish verso cartella e configurare:
- percorso di output;
- runtime;
- deployment mode;
- eventuale single-file.

Da riga di comando puoi usare `dotnet publish`, che è il comando standard per pubblicare un'app .NET. citeturn333254search6turn333254search10

## Perché è spesso la prima strada da capire

Perché è la più trasparente:
vedi chiaramente quali file vengono prodotti, capisci meglio il rapporto tra runtime, dipendenze e output finale, e costruisci una base mentale molto utile anche per capire ClickOnce e MSIX.

## Regola pratica

`Publish to folder` è la strada giusta quando vuoi il massimo controllo sui file pubblicati e non hai bisogno, almeno subito, di un sistema integrato di installazione e aggiornamento.
