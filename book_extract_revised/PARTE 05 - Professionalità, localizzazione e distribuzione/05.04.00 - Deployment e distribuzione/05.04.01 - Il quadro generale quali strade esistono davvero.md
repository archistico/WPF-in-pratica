# 05.04.01 - Il quadro generale: quali strade esistono davvero

## Il punto da chiarire subito

Quando si arriva alla distribuzione di un'applicazione WPF, è molto facile confondere piani diversi:

- come compili e pubblichi i file;
- come installi l'app sul PC dell'utente;
- come gestisci aggiornamenti;
- quanto vuoi dipendere dalla presenza del runtime .NET sul computer di destinazione.

Per questo il primo passo corretto è fare ordine nelle strade reali che esistono davvero.

## Le strade principali

Per una WPF moderna su .NET, le strade che incontri più spesso sono:

- **Publish to folder**
- **ClickOnce**
- **MSIX**

A queste si aggiunge una distinzione trasversale molto importante:
- **framework-dependent**
- **self-contained**

Questa distinzione non sostituisce le tre strade sopra, ma le attraversa.

## `Publish to folder`

È la pubblicazione più semplice:
Visual Studio o `dotnet publish` generano una cartella con i file distribuiti dell'app. È il metodo più diretto quando vuoi copiare i file manualmente, distribuirli in rete interna o costruire poi un tuo installer. citeturn333254search3turn333254search6

## ClickOnce

ClickOnce è il sistema Microsoft di distribuzione e aggiornamento semplice per app desktop Windows. Per le app .NET Windows Desktop, Visual Studio supporta la pubblicazione ClickOnce anche per .NET Core 3.1, .NET 5 e successivi. È particolarmente orientato a installazione semplificata e aggiornamenti integrati. citeturn333254search1turn333254search12

## MSIX

MSIX è il formato di packaging moderno di Windows per distribuire app desktop in modo più strutturato. Può essere usato per distribuire tramite Store, web, enterprise o altri canali, e in Visual Studio si integra attraverso strumenti di packaging dedicati. citeturn333254search2turn333254search13

## Framework-dependent e self-contained

Questa è una scelta diversa, ma fondamentale.

### Framework-dependent
L'app pubblicata richiede che sul computer di destinazione sia presente un runtime .NET compatibile. citeturn333254search6turn333254search18

### Self-contained
L'app viene pubblicata includendo il runtime necessario, così il target non deve avere già installato .NET compatibile. citeturn333254search3turn333254search10

## Perché questa mappa è importante

Perché evita una confusione molto comune:
pensare che ClickOnce, MSIX e Publish to folder siano semplicemente “tre nomi per la stessa cosa”.

Non lo sono.

- `Publish to folder` è soprattutto un modo di ottenere i file pubblicati.
- ClickOnce è un modello di distribuzione/installazione con aggiornamenti integrati. citeturn333254search1
- MSIX è un packaging moderno di Windows, con logiche proprie di installazione e distribuzione. citeturn333254search2turn333254search13

## Un criterio mentale molto utile

Puoi leggere così il quadro generale:

- **voglio solo i file pubblicati** → Publish to folder
- **voglio installazione semplice e update integrati** → ClickOnce
- **voglio packaging Windows moderno e più strutturato** → MSIX

Poi, in ciascun caso, valuti anche:
- framework-dependent
- self-contained

## Un punto importante sui progetti WPF storici

La documentazione Microsoft distingue ancora i percorsi di deployment per WPF .NET Framework e per WPF su .NET moderno. Questo conta soprattutto quando leggi documentazione o progetti esistenti, perché certi strumenti o schermate di publish possono cambiare in base al target. citeturn333254search1turn333254search4

## Regola pratica

Prima di scegliere come distribuire una WPF, chiarisci tre cose:
- vuoi solo pubblicare i file o anche installare/aggiornare in modo strutturato?
- il target ha già il runtime?
- ti serve una distribuzione semplice, enterprise o da packaging Windows moderno?

Se queste tre risposte sono chiare, la strada giusta emerge molto più facilmente.
