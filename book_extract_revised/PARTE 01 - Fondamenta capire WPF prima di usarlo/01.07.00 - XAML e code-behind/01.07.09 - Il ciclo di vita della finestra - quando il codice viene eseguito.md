# 01.07.09 - Il ciclo di vita della finestra: quando il codice viene eseguito

Per usare bene XAML e code-behind, è utile avere una prima idea del ciclo di vita di una finestra WPF.

Il flusso mentale di base è questo:

1. viene creato l'oggetto finestra;
2. il costruttore esegue `InitializeComponent()`;
3. lo XAML viene caricato e i controlli vengono creati;
4. la finestra completa il proprio caricamento;
5. l'utente interagisce con la UI e il codice risponde agli eventi.

Già questa sequenza aiuta a evitare molti errori iniziali.

Per esempio:

- nel costruttore, prima di `InitializeComponent()`, i controlli dello XAML non sono pronti;
- dopo `InitializeComponent()`, puoi già accedere agli elementi nominati;
- alcuni lavori è meglio eseguirli in eventi come `Loaded`, quando la finestra è stata effettivamente caricata.

Esempio:

```csharp
public MainWindow()
{
    InitializeComponent();
}

private void Window_Loaded(object sender, RoutedEventArgs e)
{
    NomeTextBox.Focus();
}
```

E nello XAML:

```xml
<Window x:Class="EsempioWpf.MainWindow"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        Loaded="Window_Loaded">
</Window>
```

Questo non è ancora uno studio completo del ciclo di vita di WPF, ma è abbastanza per capire una cosa decisiva: il momento in cui scrivi il codice conta. Non tutto può essere fatto in qualunque punto.

Molti problemi iniziali nascono proprio da qui: il codice è corretto in sé, ma viene eseguito troppo presto o nel posto sbagliato.

Il punto da portare a casa è questo: XAML descrive la finestra, `InitializeComponent()` la costruisce, e gli eventi della vista ti permettono di intervenire nel momento giusto.
