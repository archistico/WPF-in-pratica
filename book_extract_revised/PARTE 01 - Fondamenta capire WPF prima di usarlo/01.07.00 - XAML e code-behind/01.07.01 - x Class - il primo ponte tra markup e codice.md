# 01.07.01 - `x:Class`: il primo ponte tra markup e codice

Quando apri una finestra WPF, di solito vedi una riga come questa nel tag radice:

```xml
<Window x:Class="EsempioWpf.MainWindow"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        Title="MainWindow" Height="450" Width="800">
</Window>
```

Quell'attributo `x:Class` è uno dei collegamenti più importanti di tutto WPF. Serve a dichiarare quale classe C# è associata a quel file XAML.

In altre parole, stai dicendo: questo markup rappresenta la parte dichiarativa della classe `EsempioWpf.MainWindow`.

Di solito, dall'altra parte trovi un file `MainWindow.xaml.cs` con una struttura simile:

```csharp
namespace EsempioWpf;

public partial class MainWindow : Window
{
    public MainWindow()
    {
        InitializeComponent();
    }
}
```

La parola chiave importante qui è `partial`. La classe è parziale perché una parte viene definita nel codice C# che scrivi tu, mentre un'altra parte viene generata a partire dal file XAML.

Questo è il punto da capire bene: XAML e code-behind non sono due mondi separati. Collaborano per costruire una singola classe.

Un errore comune all'inizio è pensare che `x:Class` sia solo una stringa decorativa. In realtà deve corrispondere in modo corretto al namespace e al nome della classe nel file `.xaml.cs`. Se non coincidono, il progetto può non compilare oppure il designer può iniziare a dare errori difficili da leggere.

Il ruolo di `x:Class`, quindi, è molto concreto:

- collega il file XAML a una classe CLR;
- permette al compilatore di generare il codice necessario;
- consente al code-behind di accedere ai controlli dichiarati nel markup;
- rende possibile la collaborazione tra interfaccia e logica.

La prima idea da fissare è questa: `x:Class` non serve a decorare lo XAML, ma a dirgli **chi deve diventare** nel mondo del codice.
