# 01.07.03 - `x:Name`: come il codice C# vede i controlli

Quando vuoi usare da C# un controllo dichiarato nello XAML, di solito gli assegni un nome:

```xml
<TextBox x:Name="NomeTextBox" Width="220" />
<Button x:Name="SalvaButton" Content="Salva" />
```

Grazie a `x:Name`, il code-behind può riferirsi a quei controlli in modo diretto:

```csharp
NomeTextBox.Text = "Mario";
SalvaButton.IsEnabled = true;
```

Dal punto di vista pratico, `x:Name` rende il controllo accessibile alla classe associata allo XAML. È il modo con cui il markup espone alcuni elementi alla parte C#.

Questo non significa che ogni controllo debba avere un nome. Anzi, uno degli errori più comuni all'inizio è nominare tutto. Se un elemento non viene mai usato dal codice, non c'è nessun vantaggio reale nel dargli un nome.

Di solito `x:Name` serve quando:

- devi leggere o impostare una proprietà da codice;
- devi cambiare visibilità, testo, stato o contenuto di un controllo;
- devi accedere a un elemento in risposta a un evento;
- hai davvero bisogno di riferirti a quel controllo nella classe C#.

Se invece il controllo esiste solo per motivi di layout o presentazione, spesso il nome è superfluo.

È utile anche capire una distinzione di mentalità: in WPF non dovresti vivere sempre "agganciandoti ai controlli" come spesso succedeva in WinForms. Se un dato può essere gestito bene tramite binding, il binding è spesso la strada più pulita. `x:Name` resta utile, ma non deve diventare il centro di tutta l'applicazione.

La regola pratica è questa: usa `x:Name` quando il codice deve davvero parlare con quel controllo. Non come abitudine automatica.
