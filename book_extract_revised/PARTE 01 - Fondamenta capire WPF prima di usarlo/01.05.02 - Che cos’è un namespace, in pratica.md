# 01.05.02 - Che cos’è un namespace, in pratica

In C#, un namespace serve a organizzare le classi e a evitare collisioni di nomi. È un modo per dire: questa classe appartiene a una certa area logica del progetto.

Per esempio:

```csharp
namespace LibroWpf.Controlli
{
    public class SelettoreCliente
    {
    }
}
```

Qui `SelettoreCliente` non è una classe “sospesa nel vuoto”. Vive dentro il namespace `LibroWpf.Controlli`.

In XAML il concetto resta simile, ma cambia il modo in cui lo usi. Quando scrivi un tag come `<Button />`, stai in realtà facendo riferimento a un tipo .NET. XAML deve quindi sapere dove trovare quel tipo. Per questo motivo si dichiarano i namespace nel tag radice.

Quando inizi a usare tipi creati da te, il concetto diventa ancora più evidente. Se hai una classe dentro il namespace `LibroWpf.Views`, dovrai permettere a XAML di raggiungerla tramite una dichiarazione `xmlns` adatta.

L’idea importante è questa: il namespace non è un dettaglio sintattico. È il “contenitore logico” in cui vive una classe. Senza questa informazione, XAML non può capire a quale tipo reale ti stai riferendo.

## Un paragone utile

Puoi immaginare il namespace come il cognome di una classe. Dire solo `MainWindow` o `SchedaCliente` a volte non basta. Dire `LibroWpf.Views.SchedaCliente` è molto più preciso.

## Errore tipico

All’inizio si tende a confondere il nome del file, il nome della cartella e il namespace reale della classe. In C# non sono obbligatoriamente la stessa cosa.

## Cosa portare a casa

Un namespace è l’organizzazione logica dei tipi .NET. In WPF è fondamentale perché XAML deve sapere con precisione a quali classi stai facendo riferimento.
