# 01.05.06 - Il ruolo di `x:Class`

Nel tag radice di una finestra o di un `UserControl` trovi spesso qualcosa del genere:

```xml
<Window x:Class="LibroWpf.MainWindow"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml">
</Window>
```

`x:Class` collega il file XAML a una classe .NET reale. È il punto in cui markup e code-behind diventano due parti della stessa classe parziale.

Per esempio, al file XAML corrisponde spesso un file C# simile a questo:

```csharp
namespace LibroWpf
{
    public partial class MainWindow : Window
    {
        public MainWindow()
        {
            InitializeComponent();
        }
    }
}
```

La parola importante qui è `partial`. Una parte della classe è descritta in XAML, l’altra parte nel file C#.

## Perché è importante

Quando Visual Studio compila il progetto, genera codice che mette in relazione il contenuto del file XAML con la classe C#. Se `x:Class` non coincide con il namespace e il nome reale della classe code-behind, iniziano i problemi.

## Cosa non fa `x:Class`

`x:Class` non serve a importare namespace esterni e non sostituisce le dichiarazioni `xmlns`. Il suo compito è un altro: collegare il markup alla classe che lo rappresenta.

## Errore tipico

Uno degli errori più comuni è rinominare la classe o spostarla di namespace in C# senza aggiornare `x:Class`. Il risultato tipico è un file che sembra corretto ma non compila oppure non si apre bene nel designer.

## Cosa portare a casa

`x:Class` è il ponte tra file XAML e classe C#. Non è un semplice dettaglio tecnico: è il meccanismo che permette al markup e al code-behind di appartenere alla stessa finestra o allo stesso controllo.
