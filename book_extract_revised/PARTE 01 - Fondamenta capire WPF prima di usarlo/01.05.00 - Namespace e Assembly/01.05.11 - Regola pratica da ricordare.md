# 01.05.11 - Regola pratica da ricordare

Fin qui abbiamo distinto namespace CLR, namespace XAML, assembly, `x:Class`, converter e `UserControl`. Ora conviene fermarsi e ridurre tutto a una regola pratica semplice.

## La regola

Quando lavori in WPF, puoi orientarti così:

- se il tipo è un controllo standard WPF, userai il namespace predefinito del file XAML;
- se il tipo è tuo e si trova nello stesso progetto, in genere basta `clr-namespace:...`;
- se il tipo è tuo ma si trova in un altro assembly, devi usare `clr-namespace:...;assembly=...`.

Questa piccola regola copre la grandissima maggioranza dei casi reali.

## Come leggerla in pratica

Se stai usando un `Button`, non devi fare quasi nulla: il namespace WPF standard è già pronto.

Se stai usando un tuo `UserControl` nello stesso progetto, aggiungi una dichiarazione come:

```xml
xmlns:views="clr-namespace:LibroWpf.Views"
```

Se invece quel controllo è in una libreria esterna, la forma diventa:

```xml
xmlns:views="clr-namespace:LibreriaControlli.Views;assembly=LibreriaControlli"
```

## Perché è utile

All’inizio si rischia di vivere namespace e assembly come una giungla di dettagli. In realtà i casi pratici più comuni seguono uno schema abbastanza lineare.

## Cosa portare a casa

Non serve memorizzare mille varianti. Ti basta ricordare questa progressione: standard WPF, stesso progetto, altro assembly. Da lì capirai quasi sempre quale forma di mapping usare.
