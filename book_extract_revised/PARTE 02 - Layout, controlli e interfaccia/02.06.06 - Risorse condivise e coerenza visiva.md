# 02.06.06 - Risorse condivise e coerenza visiva

## Il vero salto di qualità

Finché personalizzi pochi controlli direttamente dentro una finestra, tutto sembra semplice.
Quando però l'app cresce, questa strategia diventa fragile.

Se colori, margini, font e stili sono sparsi in tanti file XAML, prima o poi la UI perde coerenza e la manutenzione diventa pesante.

Qui entrano in gioco le risorse condivise.

## Perché servono

Le risorse condivise servono a:
- centralizzare valori ricorrenti;
- mantenere coerenza visiva;
- evitare duplicazioni;
- rendere più semplice cambiare l'aspetto dell'app nel tempo.

## Un primo esempio

```xml
<Window.Resources>
    <SolidColorBrush x:Key="PrimaryBrush" Color="#2D6CDF"/>

    <Style TargetType="Button">
        <Setter Property="Background" Value="{StaticResource PrimaryBrush}"/>
        <Setter Property="Foreground" Value="White"/>
        <Setter Property="Padding" Value="12,6"/>
        <Setter Property="MinWidth" Value="120"/>
    </Style>
</Window.Resources>
```

Qui non hai solo personalizzato i pulsanti. Hai già introdotto una logica condivisa:
- colore centrale;
- stile coerente;
- manutenzione più semplice.

## Il vantaggio pratico

Se domani vuoi cambiare il colore primario, modifichi una sola risorsa invece di cercare manualmente decine di occorrenze.

Questo è uno dei motivi per cui le risorse sono così importanti in WPF.

## Dove mettere le risorse

All'inizio puoi metterle:
- nella singola finestra, se servono solo lì;
- in `App.xaml`, se devono essere globali.

In un'app più strutturata spesso si usano anche dizionari di risorse separati, ma qui il punto principale è capire il principio: la coerenza visiva nasce dalla centralizzazione.

## Un esempio un po' più completo

```xml
<Application.Resources>
    <SolidColorBrush x:Key="PrimaryBrush" Color="#2D6CDF"/>
    <SolidColorBrush x:Key="BorderBrushSoft" Color="#B8B8B8"/>

    <Style TargetType="Button">
        <Setter Property="Background" Value="{StaticResource PrimaryBrush}"/>
        <Setter Property="Foreground" Value="White"/>
        <Setter Property="Padding" Value="12,6"/>
        <Setter Property="MinWidth" Value="120"/>
        <Setter Property="BorderThickness" Value="0"/>
    </Style>

    <Style TargetType="TextBox">
        <Setter Property="Padding" Value="8,4"/>
        <Setter Property="BorderBrush" Value="{StaticResource BorderBrushSoft}"/>
        <Setter Property="BorderThickness" Value="1"/>
        <Setter Property="MinWidth" Value="240"/>
    </Style>
</Application.Resources>
```

Questo è già un buon inizio per un'applicazione coerente.

## Quando usare risorse condivise

Usale quando:
- un valore ricorre spesso;
- vuoi dare un'identità visiva all'app;
- vuoi evitare incoerenze;
- vuoi mantenere più facilmente il progetto.

## Quando non serve complicare troppo

Se stai facendo un esempio minuscolo o una finestra di prova, puoi anche non centralizzare tutto.
Ma appena il progetto cresce un po', le risorse condivise diventano quasi indispensabili.

## Errori tipici

### Valori ripetuti ovunque
Ogni file ha i propri colori e spaziature.

### Nessuna regola visiva centrale
L'interfaccia sembra fatta per pezzi separati.

### Risorse messe a caso
Alcune globali, altre locali, altre duplicate senza motivo.

## Regola pratica

Se vuoi una UI coerente, non basta scegliere bene i singoli controlli.
Devi anche decidere **dove vive la tua coerenza visiva**.

In WPF, molto spesso, quella coerenza vive proprio nelle risorse condivise.
