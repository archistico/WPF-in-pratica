# 02.04.05 - Il vero punto: non attaccare tutto a `Click`

Questo è probabilmente il sottocapitolo più importante del capitolo.

All'inizio è naturale scrivere così:

```xml
<MenuItem Header="_Salva" Click="Salva_Click" />
<Button Content="Salva" Click="Salva_Click" />
```

E nel code-behind:

```csharp
private void Salva_Click(object sender, RoutedEventArgs e)
{
    MessageBox.Show("Documento salvato");
}
```

Funziona. Ma il problema non è farlo funzionare una volta. Il problema è che **questa impostazione scala male**.

## Perché scala male

Immagina una finestra reale. L'azione “Salva” può essere invocata da:

- una voce di menu;
- un pulsante nella toolbar;
- un pulsante normale nella finestra;
- una scorciatoia da tastiera.

Se colleghi tutto con `Click`, finisci facilmente con:

- logica duplicata;
- metodi sparsi;
- stato abilitato/disabilitato gestito a mano;
- codice meno coerente.

## Il problema non è l'evento in sé

Usare `Click` **non è sbagliato sempre**. È sbagliato quando l'azione rappresenta un comando logico dell'applicazione e può essere esposta da più punti della UI.

Per esempio:

- `Chiudi finestra introduttiva` può anche stare in un `Click`;
- `Salva documento`, `Elimina record`, `Copia`, `Aggiorna`, `Stampa` sono candidati molto più naturali per il sistema dei comandi.

## Esempio di approccio fragile

```xml
<MenuItem Header="_Elimina" Click="Elimina_Click" />
<Button Content="Elimina" Click="Elimina_Click" />
```

```csharp
private void Elimina_Click(object sender, RoutedEventArgs e)
{
    if (lstClienti.SelectedItem == null)
    {
        MessageBox.Show("Seleziona un cliente.");
        return;
    }

    MessageBox.Show("Cliente eliminato.");
}
```

Fin qui sembra accettabile. Ma appena devi disabilitare il comando se non c'è selezione, aggiungere una scorciatoia, riusare la stessa azione altrove o spostare la logica in MVVM, cominci a sentire i limiti.

## Il modello più pulito

In questi casi il pensiero corretto è:

> Non sto gestendo un clic. Sto esponendo un'azione dell'applicazione.

Questo porta naturalmente a usare:

- `Command`;
- `CommandBinding` o `ICommand`;
- `CanExecute`;
- eventualmente MVVM.

## Quando `Click` va ancora bene

`Click` resta lecito quando:

- l'azione è locale e semplice;
- riguarda solo quella vista;
- non deve essere riusata;
- non ha una vera semantica di comando applicativo.

Esempi:

- chiudere un popup locale;
- aprire un pannello informativo;
- cancellare il testo di un campo secondario;
- lanciare una piccola animazione locale.

## Regola pratica

Se la stessa azione potrebbe stare in **menu + toolbar + scorciatoia + pulsante**, non ragionare in termini di `Click`. Ragiona in termini di **comando**.
