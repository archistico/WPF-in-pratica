# 04.01.06 - `ICommand` e command personalizzati

## Il passaggio verso il modello moderno

Dopo il modello classico con `RoutedCommand`, è il momento di vedere il cuore del commanding più moderno e più vicino a MVVM:
`ICommand`.

Questa interfaccia è molto importante perché permette di rappresentare comandi personalizzati in modo pulito, riusabile e facilmente bindabile dal view model.

## Che cos'è `ICommand`

`ICommand` è un'interfaccia che rappresenta un comando eseguibile dalla UI.

Definisce tre elementi fondamentali:
- `Execute`
- `CanExecute`
- l'evento `CanExecuteChanged`

Questi tre pezzi racchiudono il comportamento base di un comando.

## La struttura dell'interfaccia

In forma concettuale:

```csharp
public interface ICommand
{
    bool CanExecute(object parameter);
    void Execute(object parameter);
    event EventHandler CanExecuteChanged;
}
```

## Come leggere questi membri

### `Execute`
Contiene l'azione vera e propria.

### `CanExecute`
Dice se l'azione è attualmente disponibile.

### `CanExecuteChanged`
Serve a notificare che la condizione di eseguibilità è cambiata.

Questo permette alla UI di abilitare o disabilitare automaticamente i controlli collegati.

## Perché `ICommand` è così importante

Perché sposta il comando da un modello più legato alla view a un modello molto più adatto a:
- view model;
- binding;
- separazione tra UI e logica;
- testabilità;
- architetture moderne.

È per questo che `ICommand` è così centrale in MVVM.

## Un esempio minimale di comando personalizzato

```csharp
using System;
using System.Windows.Input;

public class SalvaCommand : ICommand
{
    public event EventHandler? CanExecuteChanged;

    public bool CanExecute(object? parameter)
    {
        return true;
    }

    public void Execute(object? parameter)
    {
        // logica di salvataggio
    }
}
```

## Come si usa

Nel view model o nel code-behind puoi esporre il comando:

```csharp
public ICommand SalvaCommand { get; } = new SalvaCommand();
```

Poi nello XAML:

```xml
<Button Content="Salva"
        Command="{Binding SalvaCommand}" />
```

## Perché questo modello è così naturale in MVVM

Perché la view non ha bisogno di sapere *come* si salva.
Sa solo che esiste un comando bindato chiamato `SalvaCommand`.

La logica vive nel view model o in un componente associato, non nel code-behind del pulsante.

## Quando ha senso creare comandi personalizzati

Ha senso quando:
- vuoi usare binding ai comandi;
- stai entrando in MVVM;
- l'azione è parte del linguaggio applicativo della schermata;
- ti serve `CanExecute`;
- vuoi una UI meno accoppiata al code-behind.

## Un piccolo limite del comando scritto a mano

Scrivere una classe separata per ogni comando è istruttivo, ma a volte può risultare un po' verboso.
Per questo, nei progetti moderni, si usano spesso implementazioni generiche come `RelayCommand`.

Ma prima è molto utile capire bene il modello puro di `ICommand`.

## Errori tipici

### Vedere `ICommand` come pura “cerimonia MVVM”
In realtà rappresenta un'azione in modo molto pulito.

### Ignorare `CanExecute`
Si perde uno dei vantaggi più forti del comando.

### Creare comandi personalizzati senza capire bene il ruolo di `Execute` e `CanExecute`
Il modello resta meccanico e poco chiaro.

## Regola pratica

`ICommand` è il contratto base per rappresentare un'azione bindabile in modo moderno in WPF.
Se vuoi portare il commanding nel view model, è quasi sempre da qui che passi.
