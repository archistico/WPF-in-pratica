# 04.02.10 - Un esempio completo e realistico

## Mettiamo insieme tutto il modello del capitolo

Ora costruiamo un esempio completo, ma ancora leggibile, di navigazione interna con:
- una shell;
- un'area contenuto;
- `UserControl` come viste;
- `DataTemplate`;
- shell view model;
- comandi di navigazione;
- viste che restano tali;
- comunicazione tramite view model.

Questo è il punto in cui il capitolo deve diventare davvero concreto.

## Struttura concettuale

Immagina una piccola applicazione con due sezioni:
- Dashboard
- Clienti

La shell contiene:
- menu laterale;
- area centrale contenuto.

Il view model della shell decide quale sezione è attiva.

## `RelayCommand`

```csharp
using System;
using System.Windows.Input;

public class RelayCommand : ICommand
{
    private readonly Action<object?> _execute;
    private readonly Func<object?, bool>? _canExecute;

    public RelayCommand(Action<object?> execute, Func<object?, bool>? canExecute = null)
    {
        _execute = execute;
        _canExecute = canExecute;
    }

    public event EventHandler? CanExecuteChanged;

    public bool CanExecute(object? parameter)
    {
        return _canExecute?.Invoke(parameter) ?? true;
    }

    public void Execute(object? parameter)
    {
        _execute(parameter);
    }

    public void RaiseCanExecuteChanged()
    {
        CanExecuteChanged?.Invoke(this, EventArgs.Empty);
    }
}
```

## View model della shell

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;
using System.Windows.Input;

public class ShellViewModel : INotifyPropertyChanged
{
    private object? _currentViewModel;

    public object? CurrentViewModel
    {
        get => _currentViewModel;
        set
        {
            if (_currentViewModel != value)
            {
                _currentViewModel = value;
                OnPropertyChanged();
            }
        }
    }

    public ICommand ShowDashboardCommand { get; }
    public ICommand ShowClientiCommand { get; }

    public ShellViewModel()
    {
        ShowDashboardCommand = new RelayCommand(_ => CurrentViewModel = new DashboardViewModel());
        ShowClientiCommand = new RelayCommand(_ => CurrentViewModel = new ClientiViewModel());

        CurrentViewModel = new DashboardViewModel();
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

## Che cosa mostra questo esempio

### 1. La shell resta stabile
Menu laterale e area contenuto non cambiano struttura.

### 2. Il contenuto centrale cambia
Cambiano i view model e quindi le viste.

### 3. I `UserControl` restano viste
Non si assegnano il `DataContext` da soli e non contengono la logica di navigazione.

### 4. `DataTemplate` collega view model e view
Questo è il meccanismo più elegante del pattern.

### 5. La comunicazione passa dal view model
La shell decide il contenuto attivo.
Le viste leggono il loro stato dal `DataContext`.

## Perché questo esempio è realistico

Perché rappresenta bene una struttura molto comune in applicazioni desktop:
- finestra principale unica;
- menu fisso;
- sezioni interne;
- contenuto sostituibile;
- pattern MVVM-friendly;
- zero necessità di gestire manualmente i `UserControl` dal code-behind.

## Errori tipici che l'esempio evita

- creare le view manualmente nella shell;
- assegnare il `DataContext` dentro ogni `UserControl`;
- mettere logica di navigazione nelle viste;
- usare `UserControl` come mini-app autonome.

## Regola finale

Un buon esempio di navigazione interna con `UserControl` deve mostrare questo flusso:
shell stabile → view model corrente → `ContentControl` → `DataTemplate` → vista corretta.
Se questa catena è chiara, il modello del capitolo è davvero stato compreso.
