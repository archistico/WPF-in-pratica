# 04.04.03 - Preparare il target

## Il drag non basta: serve un punto che sappia riceverlo

Una sorgente può avviare un drag, ma senza un target preparato l'operazione non produce un drop utile.

Il target è la parte della UI che deve:
- accettare il trascinamento;
- valutare il formato del dato;
- decidere se il drop è ammesso;
- eseguire l'azione concreta quando l'utente rilascia il mouse.

Preparare il target è quindi la seconda metà essenziale del modello.

## Che cosa deve avere il target

Di solito un target WPF deve avere almeno:
- `AllowDrop="True"`
- gestione degli eventi di drag and drop rilevanti, in particolare `Drop`

Spesso si gestiscono anche eventi intermedi come:
- `DragEnter`
- `DragOver`
- `DragLeave`

ma il primo punto minimo è il `Drop`.

## Un esempio semplice

```xml
<ListBox x:Name="lstDestinazione"
         AllowDrop="True"
         Drop="ListBox_Drop" />
```

Nel code-behind:

```csharp
private void ListBox_Drop(object sender, DragEventArgs e)
{
    if (e.Data.GetDataPresent(typeof(Prodotto)))
    {
        var prodotto = (Prodotto)e.Data.GetData(typeof(Prodotto))!;
        // qui gestisci il dato ricevuto
    }
}
```

## Come leggere l'esempio

### `AllowDrop="True"`
Dichiara che il controllo può ricevere un drop.

### `Drop`
È l'evento finale in cui recuperi il dato trascinato.

### `GetDataPresent(...)`
Verifica se il payload contiene il tipo che ti aspetti.

### `GetData(...)`
Recupera il dato vero e proprio.

## Perché la verifica del formato è così importante

Perché il target non dovrebbe fidarsi ciecamente di tutto ciò che arriva.
Potresti ricevere:
- testo;
- file;
- oggetti non previsti;
- formati incompatibili.

Per questo un target ben progettato verifica sempre il formato prima di elaborare il payload.

## Un caso realistico

Se hai una `ListBox` di prodotti e vuoi accettare solo oggetti `Prodotto`, il target deve dirlo chiaramente nel proprio codice.
Non basta il gesto del mouse.

Questo punto è molto importante soprattutto quando l'app cresce o interagisce con sorgenti esterne.

## Che cosa succede davvero al `Drop`

Il drop non “trasferisce automaticamente” l'oggetto nella tua collection.
Sei tu, nel gestore, a decidere:
- se aggiungerlo;
- se ignorarlo;
- se copiarlo;
- se spostarlo;
- se mostrare un errore o un feedback.

Il drop è un punto decisionale, non un automatismo totale.

## Un principio molto utile

Il target dovrebbe rispondere a tre domande:
- posso accettare questo dato?
- che effetto devo mostrare?
- che cosa devo fare quando il drop avviene davvero?

Se questo schema è chiaro, il codice del target diventa molto più leggibile.

## Errore tipico

L'errore classico è preparare male il target:
- dimenticare `AllowDrop`;
- non verificare il formato;
- assumere che il payload sia sempre del tipo giusto.

Questi errori portano a drop che non funzionano o che si comportano in modo poco prevedibile.

## Regola pratica

Preparare il target significa:
- dichiarare che può ricevere il drop;
- controllare che il dato sia quello atteso;
- gestire in modo esplicito che cosa succede quando il dato arriva davvero.
