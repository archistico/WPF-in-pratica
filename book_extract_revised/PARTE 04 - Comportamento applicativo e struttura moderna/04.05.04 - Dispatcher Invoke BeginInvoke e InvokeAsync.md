# 04.05.04 - `Dispatcher.Invoke`, `BeginInvoke` e `InvokeAsync`

## Tre modi di chiedere al thread UI di fare qualcosa

Quando hai lavoro che nasce fuori dal thread UI, ma devi aggiornare la UI o eseguire codice sul thread corretto, entra in gioco il `Dispatcher`.

In WPF incontrerai spesso tre forme:

- `Dispatcher.Invoke`
- `Dispatcher.BeginInvoke`
- `Dispatcher.InvokeAsync`

Tutte servono a far eseguire del codice sul thread UI, ma non si comportano esattamente allo stesso modo.

## `Dispatcher.Invoke`

`Invoke` esegue il delegato sul thread UI in modo sincrono.

Esempio:

```csharp
Dispatcher.Invoke(() =>
{
    lblStato.Content = "Operazione completata";
});
```

### Che cosa significa “sincrono”
Il thread che chiama `Invoke` aspetta finché il codice inviato al `Dispatcher` non è stato eseguito.

In pratica:
- mandi il lavoro al thread UI;
- aspetti che venga eseguito;
- poi prosegui.

## Quando può avere senso

Ha senso quando:
- devi aggiornare subito la UI in un punto preciso;
- ti serve un comportamento strettamente ordinato;
- il codice da eseguire è breve e chiaro.

## Perché va usato con criterio

Perché essendo sincrono può introdurre attese inutili o peggiorare la reattività se abusato.

Non è “sbagliato”, ma va capito bene.

## `Dispatcher.BeginInvoke`

`BeginInvoke` pianifica il lavoro sul thread UI in modo asincrono e ritorna subito.

Esempio:

```csharp
Dispatcher.BeginInvoke(new Action(() =>
{
    lblStato.Content = "Elaborazione in corso...";
}));
```

Qui non stai aspettando l'esecuzione immediata.
Stai dicendo al `Dispatcher`:
“quando puoi, esegui questa operazione”.

Il chiamante continua.

## Perché era molto usato

Storicamente `BeginInvoke` è stato un modo molto usato per “postare” lavoro al thread UI senza bloccare il chiamante.

È ancora utile da conoscere, soprattutto leggendo codice esistente.

## `Dispatcher.InvokeAsync`

`InvokeAsync` è una forma moderna e molto comoda.
Programma il lavoro sul thread UI e restituisce un'operazione che puoi anche attendere.

Esempio:

```csharp
await Dispatcher.InvokeAsync(() =>
{
    lblStato.Content = "Fase completata";
});
```

Questo è spesso molto elegante perché si integra bene con il mondo `async/await`.

## Un confronto molto pratico

### `Invoke`
Esegue sul thread UI e il chiamante aspetta.

### `BeginInvoke`
Programma il lavoro e ritorna subito.

### `InvokeAsync`
Programma il lavoro e restituisce un'operazione awaitable, molto comoda nel codice moderno.

## Quando `InvokeAsync` è spesso la scelta più leggibile

È spesso la scelta più leggibile quando stai già lavorando in metodi `async` e vuoi restare in un flusso coerente con `await`.

Per esempio:

```csharp
await Task.Run(() =>
{
    // lavoro pesante
});

await Dispatcher.InvokeAsync(() =>
{
    progressBar.Value = 100;
    lblStato.Content = "Completato";
});
```

Qui il flusso si legge molto bene.

## Un punto importante

Questi metodi non rendono automaticamente “non bloccante” qualunque codice.
Servono a spostare l'esecuzione sul thread UI in modo corretto.

Se poi dentro il delegato fai lavoro lungo, stai comunque bloccando la UI.

Questo è un errore molto comune.

## Quando ti servono davvero

Ti servono soprattutto quando:
- stai lavorando da un thread secondario;
- devi aggiornare controlli WPF;
- devi modificare raccolte o proprietà strettamente legate alla UI;
- vuoi coordinare bene lavoro in background e aggiornamento visivo.

## Errore tipico

L'errore classico è usare `Dispatcher.Invoke` per blocchi grossi di codice, oppure usare il `Dispatcher` come se fosse un modo generico di fare lavoro asincrono.
Non è questo il suo ruolo.
Serve a eseguire lavoro sul thread UI, non a nascondere il problema del thread UI.

## Regola pratica

Se devi far eseguire qualcosa al thread UI:
- `Invoke` quando vuoi aspettare subito;
- `BeginInvoke` quando vuoi solo pianificare;
- `InvokeAsync` quando vuoi una forma moderna e comoda con `await`.
Ma in tutti i casi, il lavoro inviato al `Dispatcher` dovrebbe restare piccolo e mirato.
