# 04.03.10 - Eliminazione

## Il caso in cui collezione e database devono restare sincronizzati subito

L'eliminazione è l'operazione che mostra in modo molto netto la necessità di sincronizzare:
- persistenza;
- raccolta bindata.

Quando elimini un record, devi:
- rimuoverlo dal database;
- rimuoverlo anche dalla `ObservableCollection`.

Se dimentichi uno dei due livelli, la schermata o il database restano incoerenti.

## Il flusso corretto

Per eliminare correttamente un elemento:

1. individui l'oggetto selezionato;
2. chiedi al repository di eliminarlo da SQLite;
3. lo rimuovi dalla raccolta bindata;
4. aggiorni eventualmente la selezione corrente.

## Un esempio semplice

```csharp
public void DeleteCurrent()
{
    if (ProdottoSelezionato is null)
        return;

    _repository.Delete(ProdottoSelezionato.Id);
    Prodotti.Remove(ProdottoSelezionato);
    ProdottoSelezionato = null;
}
```

## Come leggere questo metodo

### `_repository.Delete(...)`
Elimina il record persistente.

### `Prodotti.Remove(...)`
Rimuove l'elemento dalla raccolta osservata dalla UI.

### `ProdottoSelezionato = null`
Pulisce lo stato di selezione.

Questa è una sequenza molto corretta e molto leggibile.

## Perché non basta rimuovere dalla collezione

Se fai solo:

```csharp
Prodotti.Remove(ProdottoSelezionato);
```

la riga sparisce dalla UI, ma il record resta nel database.

Quindi, al prossimo caricamento, il dato tornerà.

## Perché non basta eliminare dal database

Se fai solo il `DELETE` nel repository ma non rimuovi dalla raccolta, la UI continua a mostrare un record che non esiste più in SQLite.
Anche questo è un errore di coerenza.

## Una buona esperienza utente

Nella pratica, l'eliminazione è spesso accompagnata da:
- conferma utente;
- gestione del caso “nessun elemento selezionato”;
- aggiornamento della selezione successiva.

Questi aspetti sono secondari rispetto al modello mentale, ma sono molto realistici in una schermata vera.

## Un esempio con comando

```csharp
public ICommand EliminaCommand { get; }

public ProdottiViewModel()
{
    _repository.EnsureDatabase();
    LoadData();

    EliminaCommand = new RelayCommand(
        _ => DeleteCurrent(),
        _ => ProdottoSelezionato is not null);
}
```

Questo mostra bene ancora una volta come CRUD, comandi e binding lavorino bene insieme.

## Un punto interessante

A differenza dell'update, qui stai cambiando la struttura della raccolta.
Quindi `ObservableCollection<T>` è particolarmente utile, perché la UI vede subito la rimozione.

## Errore tipico

L'errore classico è cancellare solo “dal `DataGrid`” o solo “dal database”.
L'eliminazione corretta è sempre doppia:
- persistenza;
- raccolta bindata.

## Regola pratica

Quando elimini in un CRUD WPF con binding, devi pensare:
- togli il record persistente;
- togli anche l'oggetto dalla raccolta osservata dalla UI.
Se fai solo uno dei due, il sistema resta incoerente.
