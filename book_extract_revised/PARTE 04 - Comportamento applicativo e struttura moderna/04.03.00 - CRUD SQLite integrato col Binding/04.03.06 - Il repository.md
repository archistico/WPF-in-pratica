# 04.03.06 - Il repository

## Il ponte tra oggetti e database

Nel capitolo, il repository è il pezzo che collega:
- il database SQLite;
- gli oggetti `Prodotto` usati dalla UI.

Questo livello è molto importante perché evita di spargere SQL e accesso al database dentro:
- code-behind;
- view model;
- controlli.

Il repository concentra la responsabilità di lettura e scrittura dei dati.

## Che cosa fa davvero

Per il nostro CRUD, il repository fa tipicamente queste cose:
- assicura che il database/tabella esistano;
- legge tutti i prodotti;
- inserisce un prodotto;
- aggiorna un prodotto;
- elimina un prodotto.

In altre parole, è il punto in cui vive la persistenza.

## Un esempio semplice di repository

```csharp
using Microsoft.Data.Sqlite;
using System.Collections.Generic;

public class ProdottoRepository
{
    private readonly string _connectionString = "Data Source=prodotti.db";

    public void EnsureDatabase()
    {
        using var connection = new SqliteConnection(_connectionString);
        connection.Open();

        var command = connection.CreateCommand();
        command.CommandText =
        @"
            CREATE TABLE IF NOT EXISTS Prodotti (
                Id INTEGER PRIMARY KEY AUTOINCREMENT,
                Nome TEXT NOT NULL,
                Prezzo REAL NOT NULL
            );
        ";

        command.ExecuteNonQuery();
    }

    public List<Prodotto> GetAll()
    {
        var prodotti = new List<Prodotto>();

        using var connection = new SqliteConnection(_connectionString);
        connection.Open();

        var command = connection.CreateCommand();
        command.CommandText = "SELECT Id, Nome, Prezzo FROM Prodotti ORDER BY Nome;";

        using var reader = command.ExecuteReader();
        while (reader.Read())
        {
            prodotti.Add(new Prodotto
            {
                Id = reader.GetInt32(0),
                Nome = reader.GetString(1),
                Prezzo = reader.GetDecimal(2)
            });
        }

        return prodotti;
    }

    public int Insert(Prodotto prodotto)
    {
        using var connection = new SqliteConnection(_connectionString);
        connection.Open();

        var command = connection.CreateCommand();
        command.CommandText =
        @"
            INSERT INTO Prodotti (Nome, Prezzo)
            VALUES (@nome, @prezzo);
            SELECT last_insert_rowid();
        ";

        command.Parameters.AddWithValue("@nome", prodotto.Nome);
        command.Parameters.AddWithValue("@prezzo", prodotto.Prezzo);

        return Convert.ToInt32((long)command.ExecuteScalar()!);
    }

    public void Update(Prodotto prodotto)
    {
        using var connection = new SqliteConnection(_connectionString);
        connection.Open();

        var command = connection.CreateCommand();
        command.CommandText =
        @"
            UPDATE Prodotti
            SET Nome = @nome,
                Prezzo = @prezzo
            WHERE Id = @id;
        ";

        command.Parameters.AddWithValue("@id", prodotto.Id);
        command.Parameters.AddWithValue("@nome", prodotto.Nome);
        command.Parameters.AddWithValue("@prezzo", prodotto.Prezzo);

        command.ExecuteNonQuery();
    }

    public void Delete(int id)
    {
        using var connection = new SqliteConnection(_connectionString);
        connection.Open();

        var command = connection.CreateCommand();
        command.CommandText = "DELETE FROM Prodotti WHERE Id = @id;";
        command.Parameters.AddWithValue("@id", id);

        command.ExecuteNonQuery();
    }
}
```

## Come leggere bene questa struttura

### `EnsureDatabase()`
Prepara la tabella se non esiste.

### `GetAll()`
Legge i record e li trasforma in oggetti `Prodotto`.

### `Insert()`
Inserisce un prodotto e ritorna l'`Id` generato.

### `Update()`
Aggiorna il record esistente.

### `Delete()`
Elimina il record corretto.

Questa è già una forma molto sana di repository per un capitolo di libro.

## Perché è meglio di query sparse

Perché centralizza:
- connessione;
- SQL;
- mapping;
- responsabilità dati.

La UI non deve sapere come sono scritte le query.
Deve solo sapere che esiste un repository che le fornisce i dati e salva le modifiche.

## Perché ritorna l'Id nell'insert

Perché quando aggiungi un nuovo `Prodotto` alla collezione bindata, è molto utile conoscere l'`Id` assegnato dal database.
Questo mantiene coerenti:
- oggetto in memoria;
- record persistito.

## Un buon criterio

Il repository deve essere:
- chiaro;
- concentrato sul dato;
- privo di responsabilità di presentazione;
- semplice da usare dal view model.

Questo è l'equilibrio giusto.

## Errore tipico

L'errore classico è:
- mettere query nel code-behind della view;
- fare SQL direttamente negli handler della UI;
- mischiare CRUD e logica di binding nello stesso posto.

Il repository esiste proprio per evitare questo caos.

## Regola pratica

Il repository è il ponte tra SQLite e gli oggetti bindati.
Se il codice SQL è sparso nella UI, quasi sempre ti manca questo livello di struttura.
