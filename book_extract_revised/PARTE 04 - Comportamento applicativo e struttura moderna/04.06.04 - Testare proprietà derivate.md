# 04.06.04 - Testare proprietà derivate

## Uno dei casi migliori per iniziare a fare test utili

Le proprietà derivate sono tra gli elementi migliori da testare in un `ViewModel` o in modelli bindati.

Perché?
Perché sono:
- importanti per la UI;
- facili da verificare;
- spesso sensibili a regressioni;
- abbastanza isolate da produrre test chiari.

In un libro su WPF, questo è un punto molto utile da fissare bene.

## Che cos'è una proprietà derivata

È una proprietà il cui valore non viene immesso direttamente, ma calcolato a partire da altri dati.

Nel caso della cassa di ristorante, esempi tipici sono:
- `Subtotale`
- `TotaleCorrente`
- `IsOccupato`
- `StatoDescrittivo`

Sono proprio il tipo di proprietà che la UI mostra continuamente.

## Un esempio con `RigaConto.Subtotale`

Supponiamo di avere:

```csharp
public decimal Subtotale => PrezzoUnitario * Quantita;
```

Questo è un ottimo candidato per un test.

```csharp
using Xunit;
using CassaRistorante.Models;

namespace CassaRistorante.Tests
{
    public class RigaContoTests
    {
        [Fact]
        public void Subtotale_Should_Be_Prezzo_Per_Quantita()
        {
            var piatto = new Piatto
            {
                Id = 1,
                Nome = "Margherita",
                Prezzo = 7.50m
            };

            var riga = new RigaConto
            {
                Piatto = piatto,
                Quantita = 3
            };

            Assert.Equal(22.50m, riga.Subtotale);
        }
    }
}
```

## Perché questo test è buono

Perché è:
- breve;
- chiaro;
- direttamente legato a una regola del dominio;
- utile per la UI, che mostra proprio quel valore.

È il tipo di test che dà subito confidenza al lettore.

## Un esempio con `TotaleCorrente`

Nel `ViewModel`, il totale corrente può dipendere dalla somma dei subtotali delle righe del tavolo selezionato.

```csharp
[Fact]
public void TotaleCorrente_Should_Sum_All_Subtotali()
{
    var vm = new CassaRistoranteViewModel();

    vm.TavoloSelezionato!.RigheConto.Clear();

    vm.TavoloSelezionato.RigheConto.Add(new RigaConto
    {
        Piatto = new Piatto { Nome = "Acqua", Prezzo = 2.00m },
        Quantita = 2
    });

    vm.TavoloSelezionato.RigheConto.Add(new RigaConto
    {
        Piatto = new Piatto { Nome = "Tiramisù", Prezzo = 5.00m },
        Quantita = 1
    });

    Assert.Equal(9.00m, vm.TotaleCorrente);
}
```

## Perché questo è molto utile in una schermata WPF

Perché proprietà come `TotaleCorrente` hanno un peso molto forte nella UI.
Se si rompono:
- la schermata mostra dati sbagliati;
- il flusso operativo perde affidabilità;
- la fiducia dell'utente cala subito.

Quindi testarle ha grande valore.

## Un esempio con `IsOccupato`

```csharp
[Fact]
public void Tavolo_Should_Be_Occupato_When_Has_Righe()
{
    var tavolo = new Tavolo();

    tavolo.RigheConto.Add(new RigaConto
    {
        Piatto = new Piatto { Nome = "Acqua", Prezzo = 2.00m },
        Quantita = 1
    });

    Assert.True(tavolo.IsOccupato);
}
```

Anche questo è un caso molto buono:
stato semplice, regola chiara, valore pratico alto.

## Che cosa il lettore deve imparare qui

Deve imparare che le proprietà derivate non sono “solo formule comode”.
Sono parte importante del comportamento dell'applicazione, e quindi meritano verifica.

Questo è particolarmente vero in WPF, dove tali proprietà spesso guidano:
- contenuto mostrato;
- stili;
- stati visivi;
- abilità dei comandi.

## Regola pratica

Se una proprietà derivata influenza il comportamento o la leggibilità della schermata, è spesso un ottimo candidato per un test automatico.
Nel dubbio, parti da lì: sono tra i test più semplici e più utili da scrivere.
