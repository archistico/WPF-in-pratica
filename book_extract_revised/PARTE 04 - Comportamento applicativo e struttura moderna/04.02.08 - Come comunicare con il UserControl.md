# 04.02.08 - Come comunicare con il `UserControl`

## La domanda giusta non è “come chiamo un metodo della view?”

Quando si comincia con la navigazione interna, una domanda molto comune è:
“come faccio a comunicare con il `UserControl` caricato?”

La forma in cui poni la domanda conta moltissimo.

Se stai pensando:
“come faccio dalla shell a prendere il `UserControl` e chiamargli un metodo?”
allora stai spesso andando verso un modello troppo accoppiato.

La domanda migliore è:
“come faccio a far arrivare alla vista i dati o i comandi giusti?”

## La risposta più sana: tramite il view model

Nel pattern del capitolo, la comunicazione corretta avviene soprattutto tramite:
- proprietà del view model;
- comandi;
- stato condiviso;
- eventuali servizi o messaggi a un livello più strutturato.

Il `UserControl`, in quanto vista, riceve e mostra ciò che il suo view model espone.

Questa è la forma di comunicazione più pulita.

## Un esempio semplice

Supponiamo che `ClientiView` mostri una lista clienti e un testo di ricerca.
La shell non deve dire al `UserControl`:
“aggiorna la textbox”.

Più correttamente, la shell o la logica di navigazione può cambiare lo stato del `ClientiViewModel`, e la vista si aggiorna tramite binding.

## Comunicazione tramite proprietà

Per esempio, nel `ClientiViewModel`:

```csharp
private string _testoRicerca = "";

public string TestoRicerca
{
    get => _testoRicerca;
    set
    {
        if (_testoRicerca != value)
        {
            _testoRicerca = value;
            OnPropertyChanged();
        }
    }
}
```

La vista:

```xml
<TextBox Text="{Binding TestoRicerca, UpdateSourceTrigger=PropertyChanged}" />
```

La comunicazione è semplicemente nel dato esposto.

## Comunicazione tramite comando

Altro caso tipico:
la vista non esegue direttamente l'azione, ma espone un comando nel suo view model.

```xml
<Button Content="Cerca"
        Command="{Binding CercaCommand}" />
```

Qui la shell non ha bisogno di “entrare” nel `UserControl`.
La vista ha già il proprio linguaggio operativo, esposto dal view model.

## Un caso realistico

La shell può selezionare un contenuto e passargli un certo contesto iniziale creando il view model opportuno.

Per esempio:

```csharp
CurrentViewModel = new ClientiViewModel
{
    TestoRicerca = "Attivi"
};
```

Quando la vista viene mostrata, il `UserControl` riceve già il suo stato tramite il `DataContext`.

Questa è comunicazione, ma nel modo giusto.

## Quando evitare il contatto diretto shell → view

È bene evitarlo quando:
- il dato può passare via view model;
- il comportamento può essere esposto come comando;
- la shell non dovrebbe conoscere i dettagli visuali della vista interna.

Più la shell conosce la view concreta, più il sistema si irrigidisce.

## Quando un contatto più diretto può capitare

In casi speciali, piccoli o molto tecnici, può capitare di avere una comunicazione più diretta.
Ma dovrebbe essere l'eccezione, non il modello centrale.

## Errore tipico

L'errore classico è trattare il `UserControl` come un oggetto da pilotare manualmente da fuori, con chiamate dirette al code-behind.
Così si perde gran parte del vantaggio di MVVM e della shell a contenuti modulari.

## Regola pratica

La comunicazione con il `UserControl` dovrebbe passare soprattutto dal suo view model:
- proprietà;
- stato;
- comandi.
Se senti il bisogno di comandare la view concreta da fuori, fermati e chiediti se non stai accoppiando troppo la shell alla vista.
