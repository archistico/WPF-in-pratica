# 04.03.09 - Aggiornamento

## L'operazione che sembra semplice ma chiarisce molto il modello

L'aggiornamento in una schermata CRUD con binding è interessante perché l'oggetto è spesso già presente nella raccolta bindata e quindi già visibile nella UI.

Questo può dare l'illusione che il lavoro sia già finito.
In realtà manca ancora il passaggio fondamentale:
persistire la modifica nel database.

## Il doppio livello dell'aggiornamento

Quando aggiorni un elemento, hai due livelli distinti:

### 1. Modifica in memoria
Il binding può già aggiornare la UI se l'oggetto implementa `INotifyPropertyChanged`.

### 2. Persistenza nel database
Il repository deve eseguire `UPDATE`.

Se dimentichi il secondo passaggio, la schermata sembra corretta finché resta aperta, ma il dato non è davvero salvato.

## Un esempio semplice

Supponiamo di avere `ProdottoSelezionato` bindato alla UI.
Il metodo di salvataggio può essere:

```csharp
public void SaveCurrent()
{
    if (ProdottoSelezionato is null)
        return;

    _repository.Update(ProdottoSelezionato);
}
```

## Perché questo è già molto istruttivo

Perché mostra bene la differenza tra:
- oggetto già modificato dalla UI;
- persistenza effettiva delle modifiche.

Il repository non serve a cambiare il valore nella textbox.
Serve a scriverlo sul database.

## Un caso classico col `DataGrid`

Se l'utente modifica direttamente una cella del `DataGrid`, l'oggetto in memoria può già ricevere il nuovo valore tramite binding.

Questo però non significa che SQLite sia stato aggiornato.
Serve ancora un'azione di salvataggio esplicita o un meccanismo coerente per fare `UPDATE`.

## Strategie possibili

### Salvataggio esplicito
L'utente modifica e poi preme `Salva`.

### Salvataggio su evento/commit
Più automatico, ma anche più delicato da progettare.

Per un capitolo chiaro e didattico, il salvataggio esplicito è spesso la strada migliore da mostrare per prima.

## Perché la raccolta bindata spesso non va toccata nell'update

Se stai aggiornando un oggetto già presente nella `ObservableCollection`, e quell'oggetto implementa `INotifyPropertyChanged`, la UI è spesso già allineata.

Quindi, a differenza dell'inserimento o dell'eliminazione, non devi necessariamente modificare la struttura della raccolta.
Devi soprattutto salvare la modifica nel repository.

Questo è un punto molto importante da capire.

## Un piccolo esempio di comando

```csharp
public ICommand SalvaCommand { get; }

public ProdottiViewModel()
{
    _repository.EnsureDatabase();
    LoadData();

    SalvaCommand = new RelayCommand(
        _ => SaveCurrent(),
        _ => ProdottoSelezionato is not null);
}
```

## Perché è utile questo modello

Perché rende il flusso molto leggibile:
- la UI modifica l'oggetto;
- il view model decide quando salvare;
- il repository esegue l'`UPDATE`.

Questa separazione è molto sana.

## Errore tipico

L'errore classico è pensare:
“vedo già il valore modificato nel `DataGrid`, quindi è aggiornato”.
No.
È aggiornato in memoria, non necessariamente nel database.

## Regola pratica

Nel CRUD WPF, l'update spesso non richiede di rimpiazzare l'oggetto nella collezione bindata.
Richiede soprattutto di ricordarsi che la modifica visibile in UI deve essere persistita con un vero `UPDATE`.
