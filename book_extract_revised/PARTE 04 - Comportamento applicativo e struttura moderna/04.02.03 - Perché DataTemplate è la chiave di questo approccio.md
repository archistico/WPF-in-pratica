# 04.02.03 - Perché `DataTemplate` è la chiave di questo approccio

## Il punto che rende elegante la navigazione interna

Finora hai visto l'idea della shell con un'area contenuto.
Ma come fa WPF a capire quale vista caricare in base al contenuto corrente?

È qui che entra in gioco `DataTemplate`.

Ed è proprio `DataTemplate` a rendere questo approccio molto elegante.

## Il concetto centrale

In questo modello, spesso il `ContentControl` della shell non riceve direttamente un `UserControl`.
Riceve invece un **view model corrente**.

Poi WPF, grazie ai `DataTemplate`, decide quale vista usare per rappresentare quel view model.

Questo è un punto architetturale molto importante.

## Un esempio mentale semplice

Supponiamo di avere:
- `DashboardViewModel`
- `ClientiViewModel`
- `OrdiniViewModel`

La shell espone una proprietà:

```csharp
public object CurrentViewModel { get; set; }
```

Quando `CurrentViewModel` contiene un `DashboardViewModel`, la UI mostra `DashboardView`.
Quando contiene un `ClientiViewModel`, mostra `ClientiView`.

Questo collegamento automatico tra view model e view avviene proprio grazie ai `DataTemplate`.

## Un esempio di `DataTemplate`

```xml
<Window.Resources>
    <DataTemplate DataType="{x:Type vm:DashboardViewModel}">
        <views:DashboardView />
    </DataTemplate>

    <DataTemplate DataType="{x:Type vm:ClientiViewModel}">
        <views:ClientiView />
    </DataTemplate>
</Window.Resources>
```

Poi nella shell:

```xml
<ContentControl Content="{Binding CurrentViewModel}" />
```

## Perché questo è così potente

Perché la shell non deve sapere:
- come costruire la vista;
- quale `UserControl` istanziare manualmente;
- quale `switch` fare nel code-behind.

La shell lavora con un view model corrente.
Il sistema dei template si occupa della rappresentazione visiva.

Questa è una soluzione molto elegante e molto “WPF”.

## Perché `DataTemplate` è meglio di creare i controlli a mano

Potresti anche scrivere codice che dice:
- se è dashboard, crea `DashboardView`;
- se è clienti, crea `ClientiView`.

Ma staresti spostando nella logica una responsabilità che WPF sa già gestire molto bene a livello dichiarativo.

Con `DataTemplate`:
- la mappa view model → view è nello XAML;
- la shell resta più pulita;
- la navigazione interna è più estendibile.

## Il vantaggio architetturale

Questo approccio aiuta molto a mantenere:
- separazione tra shell e viste;
- separazione tra view model e istanziazione della view;
- pulizia del codice;
- estendibilità futura.

Aggiungere una nuova sezione può diventare molto semplice:
- nuovo view model;
- nuova view;
- nuovo `DataTemplate`.

## Quando questo modello è particolarmente efficace

È molto efficace quando:
- lavori in MVVM;
- vuoi che la shell ragioni per view model e non per controlli concreti;
- l'app ha più sezioni interne;
- vuoi evitare navigazione a mano con switch visivi nel code-behind.

## Errore tipico

L'errore classico è usare `ContentControl` con `CurrentViewModel`, ma poi creare le view manualmente altrove, perdendo il vantaggio più bello del modello.
Se usi questa strada, `DataTemplate` è davvero la chiave.

## Regola pratica

In una shell con navigazione interna moderna, il `ContentControl` ospita spesso il view model corrente, e `DataTemplate` traduce quel view model nella vista giusta.
È proprio questo che rende l'approccio pulito.
