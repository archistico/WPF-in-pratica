# 04.06.05 - Testare un comando

## Uno dei benefici più concreti di MVVM

Se c'è un punto in cui la testabilità del `ViewModel` mostra subito il proprio valore, è proprio il testing dei comandi.

Perché?
Perché in un progetto WPF reale i comandi rappresentano molte delle azioni importanti della schermata:
- aggiungere;
- rimuovere;
- incrementare;
- chiudere;
- annullare.

E quando vivono nel `ViewModel`, diventano molto più facili da verificare.

## Che cosa conviene testare in un comando

Ci sono due aspetti principali:

- `CanExecute`
- `Execute`

### `CanExecute`
Verifica se il comando è disponibile nello stato corrente.

### `Execute`
Verifica se l'azione modifica lo stato correttamente.

Questa distinzione è fondamentale.

## Un esempio con `ChiudiContoCommand`

Supponiamo di avere un comando così:

```csharp
ChiudiContoCommand = new RelayCommand(
    _ => ChiudiConto(),
    _ => TavoloSelezionato is not null && RigheContoCorrenti.Any());
```

Qui ha molto senso testare sia la disponibilità sia l'effetto.

## Testare `CanExecute`

```csharp
using Xunit;
using CassaRistorante.Models;
using CassaRistorante.ViewModels;

namespace CassaRistorante.Tests
{
    public class CassaRistoranteCommandTests
    {
        [Fact]
        public void ChiudiContoCommand_Should_Be_Disabled_When_Conto_Is_Empty()
        {
            var vm = new CassaRistoranteViewModel();

            vm.TavoloSelezionato!.RigheConto.Clear();

            Assert.False(vm.ChiudiContoCommand.CanExecute(null));
        }
    }
}
```

## Perché questo test è utile

Perché verifica una regola di interazione molto concreta:
non ha senso chiudere un conto vuoto.

La UI potrà anche mostrare un pulsante disabilitato, ma la regola vera vive nel comando.
Ed è quella che stiamo testando.

## Testare `Execute`

```csharp
[Fact]
public void ChiudiContoCommand_Should_Clear_Righe_Conto()
{
    var vm = new CassaRistoranteViewModel();

    vm.TavoloSelezionato!.RigheConto.Clear();

    vm.TavoloSelezionato.RigheConto.Add(new RigaConto
    {
        Piatto = new Piatto { Nome = "Acqua", Prezzo = 2.00m },
        Quantita = 2
    });

    Assert.True(vm.ChiudiContoCommand.CanExecute(null));

    vm.ChiudiContoCommand.Execute(null);

    Assert.Empty(vm.TavoloSelezionato.RigheConto);
    Assert.Equal(0m, vm.TotaleCorrente);
}
```

## Che cosa si vede bene qui

Si vede che il comando:
- può essere verificato come oggetto normale;
- non richiede di cliccare realmente un pulsante;
- non dipende dal rendering della view;
- modifica lo stato in modo leggibile e testabile.

Questo è esattamente il tipo di vantaggio che MVVM promette e che qui si vede molto bene.

## Un altro esempio molto naturale

Anche un comando come `AggiungiPiattoCommand` è ottimo da testare.

Per esempio, puoi verificare:
- che sia disabilitato se manca il tavolo selezionato;
- che sia disabilitato se manca il piatto selezionato;
- che aggiunga davvero una riga quando tutto è pronto.

Questi sono casi ottimi perché rappresentano il flusso vero dell'utente.

## Un criterio importante

Quando testi un comando, non stai testando il pulsante.
Stai testando la regola applicativa che il pulsante invoca.

Questa distinzione è molto utile da fissare nel lettore.

## Perché il comando è un candidato migliore dell'evento `Click`

Perché il comando vive già nel livello dove la logica ha più senso:
il `ViewModel`.

Quindi il test non deve:
- istanziare la finestra;
- trovare il controllo;
- simulare l'interazione visiva.

Può lavorare direttamente dove conta davvero.

## Regola pratica

I comandi sono tra gli elementi più utili da testare in un progetto WPF con MVVM.
Verifica sempre almeno due cose:
se l'azione è disponibile quando dovrebbe esserlo, e se modifica davvero lo stato nel modo corretto.
