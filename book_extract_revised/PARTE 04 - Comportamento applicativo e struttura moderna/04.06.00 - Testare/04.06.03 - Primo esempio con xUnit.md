# 04.06.03 - Primo esempio con xUnit

## Il modo migliore per capire la testabilità è vedere un test vero

Dopo aver chiarito perché il `ViewModel` è testabile e che cosa conviene testare, è il momento giusto per fare un primo esempio concreto.

Qui useremo **xUnit**, perché è molto diffuso, leggibile e comodo per progetti .NET moderni.
La logica sarebbe comunque simile anche con MSTest o NUnit.

L'obiettivo non è costruire una suite completa.
L'obiettivo è far vedere che un `ViewModel` può essere istanziato e verificato come una normale classe C#.

## Un `ViewModel` semplice da testare

Useremo una versione minima di `CassaRistoranteViewModel` con:
- tavoli;
- piatti;
- tavolo selezionato;
- totale derivato.

L'idea di fondo resta identica anche per un progetto più ricco.

## Il progetto di test

Una struttura tipica potrebbe essere:

```text
CassaRistorante
CassaRistorante.Tests
```

Nel progetto di test aggiungi:
- riferimento al progetto principale;
- pacchetti `xunit`, `xunit.runner.visualstudio` e `Microsoft.NET.Test.Sdk`.

## Primo test: il `ViewModel` nasce in uno stato coerente

```csharp
using Xunit;
using CassaRistorante.ViewModels;

namespace CassaRistorante.Tests
{
    public class CassaRistoranteViewModelTests
    {
        [Fact]
        public void Constructor_Should_Initialize_Default_State()
        {
            var vm = new CassaRistoranteViewModel();

            Assert.NotNull(vm);
            Assert.NotEmpty(vm.Tavoli);
            Assert.NotEmpty(vm.Piatti);
            Assert.NotNull(vm.TavoloSelezionato);
        }
    }
}
```

## Perché questo test è utile

Perché verifica una cosa molto concreta:
il `ViewModel` parte in uno stato coerente e pronto all'uso.

Non è un test “spettacolare”, ma è un ottimo primo esempio perché:
- è facile da capire;
- è leggibile;
- mostra che il `ViewModel` non dipende dalla view per esistere.

## Un secondo test molto semplice

Se il tuo `ViewModel` espone una proprietà come:

```csharp
public decimal TotaleCorrente
    => TavoloSelezionato?.RigheConto.Sum(r => r.Subtotale) ?? 0m;
```

puoi cominciare a testarla preparando lo stato e leggendo il risultato.

```csharp
using Xunit;
using CassaRistorante.Models;
using CassaRistorante.ViewModels;

namespace CassaRistorante.Tests
{
    public class CassaRistoranteViewModelTests
    {
        [Fact]
        public void TotaleCorrente_Should_Be_Zero_When_No_Righe()
        {
            var vm = new CassaRistoranteViewModel();

            vm.TavoloSelezionato!.RigheConto.Clear();

            Assert.Equal(0m, vm.TotaleCorrente);
        }
    }
}
```

## Che cosa si vede già da questi test

Si vede una cosa molto importante:
non stiamo aprendo la `MainWindow`.
Non stiamo simulando click reali.
Non stiamo toccando i controlli.

Stiamo invece:
- istanziando classi;
- impostando stato;
- verificando comportamento.

Ed è proprio questo il guadagno pratico della testabilità.

## Perché xUnit qui è una buona scelta

Perché rende il test:
- asciutto;
- leggibile;
- focalizzato;
- molto vicino al linguaggio naturale della logica che vuoi verificare.

Per un libro pratico è una scelta ottima, perché il lettore vede subito l'idea senza eccessiva cerimonia.

## Un punto importante

Il test migliore, qui, non è quello più lungo.
È quello che fa vedere con chiarezza che il `ViewModel` è trattabile come codice normale.

Questo è il messaggio che conta di più.

## Regola pratica

Il primo test su un `ViewModel` dovrebbe essere semplice e leggibile.
Deve far capire al lettore che può istanziare il `ViewModel`, preparare lo stato e verificare il comportamento senza passare dalla finestra reale.
