# 04.05.08 - E le collezioni bindate?

## Il punto delicato in cui multithreading e binding si incontrano

Quando si parla di multithreading in WPF, uno dei problemi più pratici e più insidiosi riguarda proprio le collezioni bindate.

Per esempio:
- `ObservableCollection<T>` collegata a una `ListBox`;
- `ObservableCollection<T>` mostrata in un `DataGrid`;
- raccolte aggiornate durante importazioni o scansioni.

Qui il lettore deve capire molto bene che non basta conoscere `Task.Run` o `await`.
Serve anche sapere come aggiornare correttamente strutture osservate dalla UI.

## Il problema di base

`ObservableCollection<T>` è spesso osservata direttamente dalla UI.

Se la modifichi da un thread sbagliato, puoi ottenere:
- eccezioni;
- comportamento non affidabile;
- aggiornamenti incoerenti;
- problemi di thread affinity.

Questo è uno dei casi più concreti in cui il rapporto tra dati bindati e thread UI va trattato con grande attenzione.

## Un esempio problematico

```csharp
await Task.Run(() =>
{
    for (int i = 0; i < 100; i++)
    {
        Prodotti.Add(new Prodotto { Nome = $"Prodotto {i}" });
    }
});
```

Questo codice è molto pericoloso, perché stai modificando una `ObservableCollection<T>` osservata dalla UI da un thread in background.

## Perché succede

Perché la collezione, se è usata dal binding WPF, entra in un ecosistema che dipende dal thread UI.

Quindi non puoi trattarla come una lista qualsiasi liberamente modificabile da qualunque thread.

## Il modello corretto: due strategie sane

### Strategia 1: raccogliere i dati in background e aggiornare la collection sul thread UI

Per esempio:

```csharp
var risultati = await Task.Run(() =>
{
    var lista = new List<Prodotto>();

    for (int i = 0; i < 100; i++)
    {
        lista.Add(new Prodotto { Nome = $"Prodotto {i}" });
    }

    return lista;
});

Prodotti.Clear();

foreach (var prodotto in risultati)
{
    Prodotti.Add(prodotto);
}
```

Qui il lavoro pesante o la raccolta dei dati avviene fuori dal thread UI, ma la modifica della `ObservableCollection<T>` avviene nel contesto corretto.

### Strategia 2: usare il `Dispatcher` per aggiornamenti puntuali

```csharp
await Task.Run(() =>
{
    for (int i = 0; i < 100; i++)
    {
        var prodotto = new Prodotto { Nome = $"Prodotto {i}" };

        Application.Current.Dispatcher.Invoke(() =>
        {
            Prodotti.Add(prodotto);
        });
    }
});
```

Funziona, ma va usata con criterio.
Se gli aggiornamenti sono moltissimi, può diventare più pesante o meno elegante.

## Quale strategia preferire

Molto spesso è preferibile:
- raccogliere o elaborare dati in background;
- aggiornare la collection in blocco o in modo ragionato sul thread UI.

Questo riduce il rumore e tiene più fluido il sistema.

## Un punto molto importante

Non ogni lavoro in background deve toccare direttamente la collection bindata passo per passo.

Molto spesso è più sano separare:
- fase di elaborazione;
- fase di aggiornamento UI.

Questa separazione migliora sia la correttezza sia la leggibilità del codice.

## E il progress reporting?

Puoi perfettamente avere entrambe le cose:
- lavoro in background;
- progress reporting verso la UI;
- aggiornamento della collection in punti controllati.

L'importante è non confondere:
- stato di avanzamento;
- modifica della collection osservata.

Sono due canali diversi, anche se si toccano entrambi nella UI.

## Un criterio pratico molto utile

Se una collection è bindata a WPF, chiediti sempre:
“questa modifica sta avvenendo sul thread UI?”

Se la risposta è no, fermati e rivaluta il flusso.

## Errore tipico

L'errore classico è pensare che una `ObservableCollection<T>` sia liberamente modificabile da un thread in background solo perché contiene oggetti normali.
Il binding WPF cambia completamente la situazione.

## Regola pratica

Le collezioni bindate in WPF vanno trattate con grande attenzione dal punto di vista dei thread.
Fai pure il lavoro pesante fuori dal thread UI, ma aggiorna la `ObservableCollection<T>` nel contesto corretto e in modo ragionato.
