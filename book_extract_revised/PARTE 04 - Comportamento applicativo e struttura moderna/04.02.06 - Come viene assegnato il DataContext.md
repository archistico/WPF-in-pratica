# 04.02.06 - Come viene assegnato il `DataContext`

## Un punto tecnico che deve essere molto chiaro

In una navigazione interna basata su shell, `ContentControl` e `DataTemplate`, una delle domande più importanti è:
come arriva il `DataContext` alla vista caricata?

Questo punto va capito bene, perché se resta confuso tutto il modello sembra più magico di quanto sia davvero.

## Il flusso corretto

La shell espone una proprietà come:

```csharp
public object? CurrentViewModel { get; set; }
```

La view la usa così:

```xml
<ContentControl Content="{Binding CurrentViewModel}" />
```

Poi esistono dei `DataTemplate` che dicono:
- se il contenuto è `DashboardViewModel`, usa `DashboardView`;
- se il contenuto è `ClientiViewModel`, usa `ClientiView`.

Il passaggio fondamentale è questo:
**il contenuto del `ContentControl` diventa il `DataContext` naturale della view creata dal `DataTemplate`.**

## Perché questo è così importante

Perché significa che il `UserControl` caricato non ha bisogno di farsi assegnare a mano il proprio `DataContext` nella maggior parte dei casi.

Se il pattern è costruito bene, WPF glielo fornisce naturalmente attraverso il contenuto che sta rappresentando.

## Un esempio molto chiaro

`ShellViewModel`:

```csharp
CurrentViewModel = new ClientiViewModel();
```

XAML della shell:

```xml
<Window.Resources>
    <DataTemplate DataType="{x:Type vm:ClientiViewModel}">
        <views:ClientiView />
    </DataTemplate>
</Window.Resources>

<ContentControl Content="{Binding CurrentViewModel}" />
```

Nel momento in cui il `ContentControl` mostra quel `ClientiViewModel`, il `ClientiView` creato tramite `DataTemplate` lavora naturalmente con quel view model come `DataContext`.

## Che cosa significa per la vista

Dentro `ClientiView.xaml` puoi scrivere tranquillamente:

```xml
<TextBox Text="{Binding TestoRicerca}" />
<ListBox ItemsSource="{Binding Clienti}" />
<Button Command="{Binding CercaCommand}" />
```

Perché il contesto dati della vista è proprio il `ClientiViewModel`.

## Perché è meglio non impostarlo manualmente nel `UserControl`

Un errore molto comune è questo:

```csharp
public ClientiView()
{
    InitializeComponent();
    DataContext = new ClientiViewModel();
}
```

Questo approccio rompe o almeno indebolisce il pattern, perché:
- la shell non controlla più davvero il contenuto corrente;
- il view model viene creato dentro la vista;
- perdi flessibilità;
- perdi testabilità e controllo del ciclo di vita.

In un modello shell + `DataTemplate`, di solito la vista **riceve** il `DataContext`, non se lo inventa da sola.

## Quando assegnarlo manualmente può avere senso

Può avere senso:
- in prototipi;
- in demo isolate;
- in schermate molto semplici non costruite attorno a una shell;
- in casi particolari consapevoli.

Ma nel pattern del capitolo, la strada naturale è lasciare che arrivi dal contenuto del `ContentControl`.

## Errore tipico

L'errore classico è non capire chi assegna il `DataContext` e quindi:
- sovrascriverlo inutilmente;
- romperlo nel code-behind del `UserControl`;
- rendere la vista meno riusabile.

## Regola pratica

In una navigazione shell + `ContentControl` + `DataTemplate`, il `DataContext` della vista arriva normalmente dal view model corrente assegnato al contenuto.
Se il pattern è corretto, non devi forzarlo nella vista.
