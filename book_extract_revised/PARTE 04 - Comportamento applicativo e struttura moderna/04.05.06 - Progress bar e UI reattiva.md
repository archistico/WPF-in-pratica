# 04.05.06 - Progress bar e UI reattiva

## Un caso pratico molto comune

Uno degli scenari più tipici in WPF è questo:
hai un'operazione lunga e vuoi mostrare una `ProgressBar` mentre la UI resta reattiva.

Qui il lettore vede molto bene se ha capito davvero:
- thread UI;
- `async/await`;
- lavoro in background;
- aggiornamento dello stato della schermata.

È quindi un punto molto importante del capitolo.

## Il problema classico

Molti provano a fare una cosa del genere:

```csharp
progressBar.Value = 0;

for (int i = 1; i <= 100; i++)
{
    progressBar.Value = i;
    Thread.Sleep(50);
}
```

Il risultato è spesso deludente:
- la finestra si blocca;
- la progress bar non si aggiorna fluidamente;
- il rendering resta congelato fino alla fine.

Perché?
Perché il thread UI è occupato dal lavoro stesso.

## Il modello corretto

Per una UI reattiva con progress bar, devi separare:
- il lavoro lungo;
- l'aggiornamento della UI.

L'operazione pesante o attesa deve procedere senza bloccare il thread UI.
La progress bar deve essere aggiornata in modo compatibile col contesto corretto.

## Un esempio semplice con `async/await`

```csharp
private async void BtnAvvia_Click(object sender, RoutedEventArgs e)
{
    btnAvvia.IsEnabled = false;
    progressBar.Value = 0;
    lblStato.Content = "Elaborazione in corso...";

    for (int i = 1; i <= 100; i++)
    {
        await Task.Delay(30);
        progressBar.Value = i;
    }

    lblStato.Content = "Completato";
    btnAvvia.IsEnabled = true;
}
```

## Che cosa mostra davvero questo esempio

Questo esempio non è un vero lavoro pesante, ma fa vedere bene una cosa:
l'uso di `await` lascia respirare il thread UI tra un aggiornamento e l'altro.

Quindi:
- la progress bar si ridisegna;
- la finestra resta reattiva;
- l'utente percepisce il progresso.

## Quando il lavoro è reale

Nelle applicazioni vere potresti avere:
- importazione file;
- scansione cartelle;
- download;
- parsing di dati;
- generazione di miniature;
- elaborazioni lunghe.

La progress bar non deve solo “muoversi”.
Deve accompagnare un lavoro reale mantenendo la schermata viva.

## Un principio molto importante

Aggiornare la progress bar non serve a nulla se il thread UI è bloccato.
Questa frase è fondamentale.

Molti pensano:
“ho scritto `progressBar.Value = ...`, quindi si vedrà”.
No.
Si vedrà bene solo se il thread UI ha davvero la possibilità di processare il rendering.

## Un caso realistico con metodo asincrono

```csharp
private async void BtnImporta_Click(object sender, RoutedEventArgs e)
{
    btnImporta.IsEnabled = false;
    lblStato.Content = "Importazione in corso...";
    progressBar.Value = 0;

    await ImportaAsync();

    lblStato.Content = "Importazione completata";
    progressBar.Value = 100;
    btnImporta.IsEnabled = true;
}
```

Il metodo `ImportaAsync()` deve essere progettato in modo che la UI possa continuare a vivere.
Questo è il punto chiave.

## Perché la progress bar è anche un test mentale

Se la progress bar:
- non avanza;
- avanza tutta insieme alla fine;
- la finestra si blocca;

allora quasi sempre c'è un problema di gestione del thread UI o del lavoro lungo.

È un ottimo indicatore didattico.

## Errore tipico

L'errore classico è fare lavoro lungo direttamente nel thread UI e aspettarsi che la progress bar si aggiorni lo stesso in tempo reale.
Non funziona così.

## Regola pratica

Una progress bar utile in WPF richiede una UI reattiva.
Se l'operazione lunga non lascia respirare il thread UI, il progresso mostrato sarà poco credibile o del tutto bloccato.
