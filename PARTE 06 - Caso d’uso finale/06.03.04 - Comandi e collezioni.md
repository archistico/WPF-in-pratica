# 06.03.04 - Comandi e collezioni

## Il punto in cui la schermata smette di essere statica

Una cassa di ristorante non è solo una schermata che mostra dati.
È una schermata operativa.

Questo significa che servono due grandi meccanismi ben progettati:
- collezioni che rappresentano lo stato dinamico;
- comandi che permettono all'utente di agire su quello stato.

Questo sottocapitolo è molto importante, perché qui il caso finale comincia a comportarsi davvero come una vera applicazione.

## Le collezioni principali

Nel caso finale, le collezioni più importanti sono:
- `Tavoli`
- `Piatti`
- `RigheConto` del tavolo corrente

Tutte hanno un ruolo diverso.

### `Tavoli`
Rappresenta i contesti selezionabili.

### `Piatti`
Rappresenta il catalogo degli articoli ordinabili.

### `RigheConto`
Rappresenta il contenuto operativo del conto corrente.

Questa tripla distinzione è molto sana.

## Perché `ObservableCollection<T>` è la scelta naturale

Per tutte queste strutture, `ObservableCollection<T>` è la scelta più didattica e più naturale, perché:
- la UI può reagire ad aggiunte e rimozioni;
- lo stato visibile cambia subito;
- il lettore vede bene la relazione tra dati e presentazione.

In un caso come questo, è esattamente il tipo di raccolta che vuoi avere.

## I comandi principali della schermata

Una versione didatticamente equilibrata del progetto dovrebbe esporre almeno comandi come:
- `AggiungiPiattoCommand`
- `RimuoviRigaCommand`
- `AumentaQuantitaCommand`
- `DiminuisciQuantitaCommand`
- `ChiudiContoCommand`

Volendo, si può aggiungere anche:
- `SvuotaContoCommand`
- `SelezionaTavoloCommand`

ma il nucleo operativo ruota già molto bene attorno ai primi cinque.

## Perché i comandi sono così importanti qui

Perché questo caso finale ha un flusso fortemente orientato ad azioni:
- aggiungi;
- correggi;
- rimuovi;
- chiudi.

Quindi è un ottimo terreno per usare davvero:
- `ICommand`
- `RelayCommand`
- `CanExecute`
- separazione tra gesture UI e operazione applicativa

Questo è uno dei punti in cui il lettore vede la forza pratica del modello.

## Un esempio di proprietà comando

```csharp
public ICommand AggiungiPiattoCommand { get; }
public ICommand RimuoviRigaCommand { get; }
public ICommand AumentaQuantitaCommand { get; }
public ICommand DiminuisciQuantitaCommand { get; }
public ICommand ChiudiContoCommand { get; }
```

## Un esempio di inizializzazione

```csharp
public CassaRistoranteViewModel()
{
    SeedDatiIniziali();

    AggiungiPiattoCommand = new RelayCommand(
        _ => AggiungiPiatto(),
        _ => TavoloSelezionato is not null && PiattoSelezionato is not null);

    RimuoviRigaCommand = new RelayCommand(
        _ => RimuoviRiga(),
        _ => TavoloSelezionato is not null && RigaSelezionata is not null);

    AumentaQuantitaCommand = new RelayCommand(
        _ => AumentaQuantita(),
        _ => RigaSelezionata is not null);

    DiminuisciQuantitaCommand = new RelayCommand(
        _ => DiminuisciQuantita(),
        _ => RigaSelezionata is not null);

    ChiudiContoCommand = new RelayCommand(
        _ => ChiudiConto(),
        _ => TavoloSelezionato is not null && RigheContoCorrenti.Any());
}
```

## Esempio di metodi operativi

```csharp
private void AggiungiPiatto()
{
    if (TavoloSelezionato is null || PiattoSelezionato is null)
        return;

    var rigaEsistente = TavoloSelezionato.RigheConto
        .FirstOrDefault(r => r.Piatto.Id == PiattoSelezionato.Id);

    if (rigaEsistente is not null)
    {
        rigaEsistente.Quantita++;
    }
    else
    {
        TavoloSelezionato.RigheConto.Add(new RigaConto
        {
            Piatto = PiattoSelezionato,
            Quantita = 1
        });
    }

    OnPropertyChanged(nameof(RigheContoCorrenti));
    OnPropertyChanged(nameof(TotaleCorrente));
}

private void RimuoviRiga()
{
    if (TavoloSelezionato is null || RigaSelezionata is null)
        return;

    TavoloSelezionato.RigheConto.Remove(RigaSelezionata);
    RigaSelezionata = null;

    OnPropertyChanged(nameof(RigheContoCorrenti));
    OnPropertyChanged(nameof(TotaleCorrente));
}

private void AumentaQuantita()
{
    if (RigaSelezionata is null)
        return;

    RigaSelezionata.Quantita++;
    OnPropertyChanged(nameof(TotaleCorrente));
}

private void DiminuisciQuantita()
{
    if (TavoloSelezionato is null || RigaSelezionata is null)
        return;

    RigaSelezionata.Quantita--;

    if (RigaSelezionata.Quantita <= 0)
    {
        TavoloSelezionato.RigheConto.Remove(RigaSelezionata);
        RigaSelezionata = null;
    }

    OnPropertyChanged(nameof(RigheContoCorrenti));
    OnPropertyChanged(nameof(TotaleCorrente));
}

private void ChiudiConto()
{
    if (TavoloSelezionato is null)
        return;

    TavoloSelezionato.RigheConto.Clear();
    RigaSelezionata = null;

    OnPropertyChanged(nameof(RigheContoCorrenti));
    OnPropertyChanged(nameof(TotaleCorrente));
}
```

## Perché questi metodi sono didatticamente molto efficaci

Perché fanno vedere in modo molto concreto:
- come si usa una collection bindata;
- come cambia il contenuto della schermata;
- come il totale dipende dalle righe;
- come i comandi incapsulano operazioni reali della UI.

Questo è esattamente il tipo di esperienza che il lettore deve fare nel caso finale.

## Il ruolo di `CanExecute`

Questo caso è ottimo anche per far capire il valore di `CanExecute`.

Per esempio:
- non puoi rimuovere una riga se non ne hai una selezionata;
- non puoi chiudere un conto vuoto;
- non puoi aggiungere un piatto se non hai un tavolo attivo.

Queste regole rendono la UI più intelligente e molto più professionale.

## Un punto importante sulle notifiche

Ogni volta che cambia:
- il tavolo attivo;
- una riga;
- la collection del conto;
- il totale derivato;

la UI deve sapere come aggiornarsi.

Questo rende il caso finale perfetto per consolidare il rapporto tra:
- collection osservabili;
- proprietà derivate;
- notifiche corrette;
- comandi coerenti.

## Regola pratica

Nel caso finale, le collezioni rappresentano lo stato vivo della schermata e i comandi rappresentano le azioni dell'operatore.
Se questi due livelli sono progettati bene, la UI WPF smette di essere passiva e diventa davvero operativa.
