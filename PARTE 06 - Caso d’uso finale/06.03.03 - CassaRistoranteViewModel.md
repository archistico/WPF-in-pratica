# 06.03.03 - `CassaRistoranteViewModel`

## Il punto in cui il caso finale prende davvero forma

Dopo aver definito le regole funzionali e i modelli, arriva il vero centro di coordinamento della schermata:
il `CassaRistoranteViewModel`.

Questo è il cuore applicativo del caso finale, perché qui si incontrano:
- stato corrente della schermata;
- selezione del tavolo;
- elenco piatti disponibili;
- righe del conto visibili;
- comandi operativi;
- dati derivati come totale e stato.

In altre parole, questo view model è il punto che trasforma il dominio in schermata WPF concreta.

## Quali responsabilità deve avere

Il `CassaRistoranteViewModel` deve esporre almeno:
- la collezione dei tavoli;
- la collezione dei piatti;
- il tavolo attualmente selezionato;
- la riga conto selezionata;
- eventualmente il piatto selezionato;
- il totale del conto corrente;
- i comandi dell'interfaccia.

Questa lista è molto importante, perché evita sia un view model troppo povero sia un view model eccessivamente gonfio.

## Una struttura iniziale sensata

### Esempio

```csharp
using System.Collections.ObjectModel;
using System.ComponentModel;
using System.Runtime.CompilerServices;
using System.Linq;

public class CassaRistoranteViewModel : INotifyPropertyChanged
{
    private Tavolo? _tavoloSelezionato;
    private RigaConto? _rigaSelezionata;
    private Piatto? _piattoSelezionato;

    public ObservableCollection<Tavolo> Tavoli { get; } = new();
    public ObservableCollection<Piatto> Piatti { get; } = new();

    public Tavolo? TavoloSelezionato
    {
        get => _tavoloSelezionato;
        set
        {
            if (_tavoloSelezionato != value)
            {
                _tavoloSelezionato = value;
                OnPropertyChanged();
                OnPropertyChanged(nameof(RigheContoCorrenti));
                OnPropertyChanged(nameof(TotaleCorrente));
            }
        }
    }

    public RigaConto? RigaSelezionata
    {
        get => _rigaSelezionata;
        set
        {
            if (_rigaSelezionata != value)
            {
                _rigaSelezionata = value;
                OnPropertyChanged();
            }
        }
    }

    public Piatto? PiattoSelezionato
    {
        get => _piattoSelezionato;
        set
        {
            if (_piattoSelezionato != value)
            {
                _piattoSelezionato = value;
                OnPropertyChanged();
            }
        }
    }

    public ObservableCollection<RigaConto> RigheContoCorrenti
        => TavoloSelezionato?.RigheConto ?? new ObservableCollection<RigaConto>();

    public decimal TotaleCorrente
        => TavoloSelezionato?.RigheConto.Sum(r => r.Subtotale) ?? 0m;

    public CassaRistoranteViewModel()
    {
        SeedDatiIniziali();
    }

    private void SeedDatiIniziali()
    {
        Piatti.Add(new Piatto { Id = 1, Nome = "Margherita", Prezzo = 7.50m, Categoria = "Pizze" });
        Piatti.Add(new Piatto { Id = 2, Nome = "Carbonara", Prezzo = 9.00m, Categoria = "Primi" });
        Piatti.Add(new Piatto { Id = 3, Nome = "Acqua", Prezzo = 2.00m, Categoria = "Bevande" });

        Tavoli.Add(new Tavolo { Id = 1, Nome = "Tavolo 1" });
        Tavoli.Add(new Tavolo { Id = 2, Nome = "Tavolo 2" });
        Tavoli.Add(new Tavolo { Id = 3, Nome = "Tavolo 3" });

        TavoloSelezionato = Tavoli.FirstOrDefault();
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

## Come leggere questa struttura

### `Tavoli`
Rappresenta la sala o il contesto selezionabile.

### `Piatti`
Rappresenta ciò che si può aggiungere al conto.

### `TavoloSelezionato`
Determina il contesto corrente della schermata.

### `RigheContoCorrenti`
Rappresenta l'ordine visibile nella UI.

### `TotaleCorrente`
È il dato derivato più importante.

Questa struttura è molto forte perché rende evidente il rapporto tra dominio e interfaccia.

## Perché il view model è così centrale

Perché evita che la schermata venga governata da:
- code-behind troppo pieno;
- logiche sparse;
- accessi casuali ai controlli;
- dipendenze visive troppo forti.

Il `CassaRistoranteViewModel` è il posto giusto in cui la schermata “sa” che cosa sta succedendo.

## Un punto importante su `RigheContoCorrenti`

Questa proprietà è interessante perché mostra una cosa molto WPF:
la UI non deve sempre lavorare su strutture indipendenti.
Può anche lavorare sul contenuto derivato dal contesto corrente.

È però importante notificare correttamente il cambio del tavolo selezionato, così la UI sa che il contenuto visibile è cambiato.

## Perché questo view model è didatticamente forte

Perché mette insieme:
- selezione;
- raccolte;
- proprietà derivate;
- stato corrente;
- inizializzazione dati;
- base per i comandi.

È un punto in cui il lettore vede davvero che MVVM non è teoria, ma struttura utile.

## Che cosa non deve contenere troppo presto

Per questa fase conviene evitare che il view model si trasformi subito in:
- repository completo;
- sistema di persistenza complesso;
- gestore di pagamenti;
- motore di business enorme.

Qui il suo ruolo è ancora soprattutto quello di orchestrare bene la schermata.

## Regola pratica

Il `CassaRistoranteViewModel` deve essere il centro di stato e comportamento della schermata:
sa quali tavoli esistono, quali piatti sono disponibili, quale conto è attivo e quali dati la UI deve vedere in ogni momento.
