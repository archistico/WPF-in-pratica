# 02.03.04 - Quando serve un `ItemTemplate`

`DisplayMemberPath` è perfetto quando devi mostrare una sola proprietà. Ma appena l’elemento deve avere una rappresentazione più ricca, entra in gioco `ItemTemplate`.

`ItemTemplate` serve a dire alla `ComboBox`:

> “Ogni elemento non deve essere mostrato come una singola stringa, ma con una piccola struttura visuale.”

## Quando `DisplayMemberPath` non basta più

Serve un `ItemTemplate` quando vuoi mostrare:

- più campi;
- testo principale e secondario;
- immagini o icone;
- elementi formattati in modo diverso;
- layout su più righe;
- contenuti visivi più esplicativi.

## Caso tipico

Immagina una lista di clienti:

```csharp
public class Cliente
{
    public int Id { get; set; }
    public string RagioneSociale { get; set; } = "";
    public string Citta { get; set; } = "";
}
```

Con `DisplayMemberPath` puoi mostrare solo una proprietà, per esempio:

```xml
<ComboBox ItemsSource="{Binding Clienti}"
          DisplayMemberPath="RagioneSociale" />
```

Va bene, ma l’utente vede solo il nome. Se hai clienti con nomi simili, può non bastare.

## Esempio con `ItemTemplate`

```xml
<ComboBox Width="300"
          ItemsSource="{Binding Clienti}"
          SelectedItem="{Binding ClienteSelezionato}">
    <ComboBox.ItemTemplate>
        <DataTemplate>
            <StackPanel>
                <TextBlock Text="{Binding RagioneSociale}"
                           FontWeight="SemiBold" />
                <TextBlock Text="{Binding Citta}"
                           FontSize="11" />
            </StackPanel>
        </DataTemplate>
    </ComboBox.ItemTemplate>
</ComboBox>
```

## Cosa sta succedendo

- `ItemsSource` fornisce la collezione;
- `SelectedItem` conserva l’oggetto scelto;
- `ItemTemplate` definisce il layout con cui visualizzare ogni elemento.

Ogni riga della dropdown ora mostra:

- ragione sociale;
- città;
- due livelli visivi distinti.

## Perché qui il template è la scelta giusta

Perché la sola stringa non comunica abbastanza.

Un `ItemTemplate` è utile quando il contenuto visuale deve aiutare l’utente a riconoscere rapidamente l’elemento corretto.

Questo è un criterio importante: non usare il template “perché si può”, ma perché **aiuta davvero a scegliere meglio**.

## Un esempio con immagine e testo

```csharp
public class Categoria
{
    public string Nome { get; set; } = "";
    public string Icona { get; set; } = "";
}
```

```xml
<ComboBox Width="260"
          ItemsSource="{Binding Categorie}"
          SelectedItem="{Binding CategoriaSelezionata}">
    <ComboBox.ItemTemplate>
        <DataTemplate>
            <StackPanel Orientation="Horizontal">
                <Image Source="{Binding Icona}"
                       Width="16"
                       Height="16"
                       Margin="0,0,8,0" />
                <TextBlock Text="{Binding Nome}" />
            </StackPanel>
        </DataTemplate>
    </ComboBox.ItemTemplate>
</ComboBox>
```

Qui l’elemento non è più una semplice stringa: è una piccola composizione grafica.

## Quando usare il template è una cattiva idea

Non conviene usare `ItemTemplate` quando:

- devi solo mostrare una proprietà semplice;
- stai complicando il controllo senza vantaggio reale;
- il layout del template è troppo ricco e rallenta la lettura;
- stai cercando di trasformare una `ComboBox` in una mini scheda dettagli.

## Attenzione a un dettaglio importante

Un `ItemTemplate` definisce **come appaiono gli elementi nella lista**. Non necessariamente controlla tutto l’aspetto della parte chiusa del controllo. In alcuni scenari, soprattutto con elementi complessi e modalità editabile, è bene verificare anche il comportamento della parte selezionata.

Questo punto tornerà utile nel sottocapitolo sul problema del nome della classe.

## Esempio completo più realistico

### C#

```csharp
public class Operatore
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";
    public string Reparto { get; set; } = "";
}

public List<Operatore> Operatori { get; } = new()
{
    new Operatore { Id = 1, Nome = "Anna Verdi", Reparto = "Grafica" },
    new Operatore { Id = 2, Nome = "Luca Bianchi", Reparto = "Sviluppo" },
    new Operatore { Id = 3, Nome = "Marta Neri", Reparto = "Amministrazione" }
};

public Operatore? OperatoreSelezionato { get; set; }
```

### XAML

```xml
<ComboBox Width="280"
          ItemsSource="{Binding Operatori}"
          SelectedItem="{Binding OperatoreSelezionato}">
    <ComboBox.ItemTemplate>
        <DataTemplate>
            <Grid Margin="2">
                <Grid.RowDefinitions>
                    <RowDefinition Height="Auto" />
                    <RowDefinition Height="Auto" />
                </Grid.RowDefinitions>

                <TextBlock Text="{Binding Nome}"
                           FontWeight="SemiBold" />
                <TextBlock Grid.Row="1"
                           Text="{Binding Reparto}"
                           FontSize="11"
                           Opacity="0.75" />
            </Grid>
        </DataTemplate>
    </ComboBox.ItemTemplate>
</ComboBox>
```

## Regola pratica

Usa `ItemTemplate` quando la rappresentazione dell’elemento deve essere **più ricca di una singola proprietà** e questa ricchezza aiuta davvero la scelta dell’utente.

## Il punto da portare a casa

`ItemTemplate` non serve a rendere il codice più sofisticato. Serve a rendere la `ComboBox` più chiara quando ogni elemento deve esprimere più informazioni. Se una sola proprietà non basta, il template è la strada giusta.
