# 02.03.05 - Il problema classico: quando chiusa mostra il nome della classe

Questo è uno dei problemi più classici con la `ComboBox` in WPF.

La dropdown si apre correttamente, la lista sembra popolata, magari gli elementi appaiono anche bene... ma appena scegli un elemento e la `ComboBox` si richiude, al posto del testo atteso compare qualcosa del genere:

```text
MioProgetto.Models.Cliente
```

oppure

```text
Namespace.Classe
```

Per chi inizia è un comportamento spiazzante, ma in realtà ha una spiegazione molto logica.

## Perché succede

Succede perché la `ComboBox` sta lavorando con **oggetti**, non con stringhe.

Se non le dici in modo chiaro quale rappresentazione testuale usare, il framework prova a convertire l’oggetto in stringa. E per farlo usa normalmente `ToString()`.

Se la classe non ridefinisce `ToString()`, il risultato predefinito è il nome completo del tipo.

## Esempio che genera il problema

### C#

```csharp
public class Cliente
{
    public int Id { get; set; }
    public string RagioneSociale { get; set; } = "";
}
```

### XAML

```xml
<ComboBox ItemsSource="{Binding Clienti}"
          SelectedItem="{Binding ClienteSelezionato}" />
```

Se non specifichi né `DisplayMemberPath` né un `ItemTemplate`, la `ComboBox` non sa cosa mostrare come testo leggibile.

## Primo modo corretto per risolvere: `DisplayMemberPath`

Se devi mostrare una sola proprietà, la soluzione migliore è questa:

```xml
<ComboBox ItemsSource="{Binding Clienti}"
          SelectedItem="{Binding ClienteSelezionato}"
          DisplayMemberPath="RagioneSociale" />
```

Questa è di solito la scelta più semplice e più pulita.

## Secondo modo corretto per risolvere: `ItemTemplate`

Se devi mostrare più informazioni:

```xml
<ComboBox ItemsSource="{Binding Clienti}"
          SelectedItem="{Binding ClienteSelezionato}">
    <ComboBox.ItemTemplate>
        <DataTemplate>
            <StackPanel>
                <TextBlock Text="{Binding RagioneSociale}" />
                <TextBlock Text="{Binding Id}" FontSize="11" />
            </StackPanel>
        </DataTemplate>
    </ComboBox.ItemTemplate>
</ComboBox>
```

Questa soluzione è giusta quando ti serve una rappresentazione visiva più ricca.

## Terzo modo possibile: ridefinire `ToString()`

Puoi anche fare così:

```csharp
public class Cliente
{
    public int Id { get; set; }
    public string RagioneSociale { get; set; } = "";

    public override string ToString()
    {
        return RagioneSociale;
    }
}
```

Questo funziona, ma va usato con prudenza.

## Perché `ToString()` non è quasi mai la prima scelta

Ridefinire `ToString()` può essere comodo in demo rapide, debugging o classi molto semplici. Però in un progetto reale spesso non è la scelta migliore, per vari motivi:

- la rappresentazione testuale utile nella UI non è sempre quella utile altrove;
- una classe può dover essere mostrata in modi diversi in schermate diverse;
- spostare la scelta visiva dentro il modello non è sempre pulito.

Per questo, in WPF, è quasi sempre meglio preferire:

- `DisplayMemberPath` per i casi semplici;
- `ItemTemplate` per i casi più ricchi.

## Il caso più insidioso

A volte la dropdown mostra bene gli elementi, ma la parte chiusa continua a visualizzare qualcosa di inatteso. Questo capita spesso quando si mescolano:

- oggetti complessi;
- `IsEditable=True`;
- template particolari;
- assenza di una proprietà testuale ben chiara.

In questi casi bisogna distinguere bene:

- come si visualizzano gli elementi nella lista;
- quale testo o contenuto appare quando l’elemento è selezionato.

## Un criterio pratico per scegliere la soluzione

### Usa `DisplayMemberPath` quando:
- basta una sola proprietà;
- vuoi il minimo indispensabile;
- vuoi una configurazione leggibile e robusta.

### Usa `ItemTemplate` quando:
- ogni elemento deve mostrare più campi;
- vuoi immagine, testo secondario o layout personalizzato;
- la resa visiva deve aiutare davvero la selezione.

### Usa `ToString()` solo quando:
- la classe ha una rappresentazione testuale naturale e stabile;
- il progetto è molto semplice;
- sai bene le conseguenze della scelta.

## Esempio corretto e realistico

```xml
<ComboBox Width="260"
          ItemsSource="{Binding Clienti}"
          SelectedItem="{Binding ClienteSelezionato}"
          DisplayMemberPath="RagioneSociale" />
```

Questa soluzione risolve il problema nella maggior parte dei casi.

## Errore tipico da evitare

L’errore più comune è pensare che il problema sia “grafico”. In realtà è quasi sempre un problema di **rappresentazione dell’oggetto**.

Non è la `ComboBox` che sbaglia: è il controllo che non ha ricevuto istruzioni sufficienti su come trasformare un oggetto in qualcosa di leggibile.

## Il punto da portare a casa

Se una `ComboBox` chiusa mostra il nome della classe, il messaggio implicito è semplice: stai passando oggetti, ma non hai ancora definito bene **come devono essere mostrati**. Nella maggior parte dei casi la soluzione giusta è `DisplayMemberPath`. Quando non basta, serve un `ItemTemplate`.
