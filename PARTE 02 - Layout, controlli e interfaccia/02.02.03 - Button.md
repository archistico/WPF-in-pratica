# 02.02.03 - `Button`

`Button` è il controllo che rappresenta un’azione. In una finestra reale lo usi per salvare, annullare, cercare, confermare, cancellare, aprire, chiudere, esportare.

Sembra il controllo più semplice del mondo, ma in WPF va capito bene per due motivi:

- può contenere molto più di una semplice stringa;
- spesso è il punto in cui UI e logica iniziano a collaborare.

## Il ruolo del pulsante

Un `Button` non serve a “mettere qualcosa di cliccabile”.
Serve a dichiarare in modo chiaro che l’utente può eseguire un’azione.

La domanda giusta non è: “mi serve un controllo cliccabile?”.
La domanda giusta è: “questa parte della UI rappresenta davvero un comando dell’utente?”.

Se la risposta è sì, `Button` è quasi certamente la scelta corretta.

## Esempio base

```xml
<Button Content="Salva"
        Width="100"
        Height="32" />
```

Va benissimo per iniziare. Ma in WPF un pulsante può essere molto più ricco.

## Un pulsante con contenuto composto

```xml
<Button Padding="12,6">
    <StackPanel Orientation="Horizontal">
        <TextBlock Text="💾"
                   Margin="0,0,8,0" />
        <TextBlock Text="Salva dati" />
    </StackPanel>
</Button>
```

Questo esempio è importante perché mostra una caratteristica decisiva di WPF: il contenuto del pulsante non è limitato al testo.

## Perché questo conta davvero

In altri ambienti UI, spesso il pulsante è pensato come un rettangolo con una stringa. In WPF, invece, il pulsante è molto più vicino all’idea di “contenitore di contenuto interattivo”.

Questo lo rende più flessibile, ma richiede anche più disciplina.

## Un esempio realistico in una barra comandi

```xml
<StackPanel Orientation="Horizontal"
            HorizontalAlignment="Right"
            Margin="16">
    <Button Content="Nuovo"
            Width="90"
            Margin="0,0,8,0" />
    <Button Content="Salva"
            Width="90"
            Margin="0,0,8,0" />
    <Button Content="Elimina"
            Width="90" />
</StackPanel>
```

Qui il ruolo del `Button` è perfettamente leggibile: ogni elemento corrisponde a un’azione chiara.

## Eventi o command?

All’inizio è naturale usare l’evento `Click`.

```xml
<Button Content="Salva"
        Click="BtnSalva_Click" />
```

Code-behind:

```csharp
private void BtnSalva_Click(object sender, RoutedEventArgs e)
{
    MessageBox.Show("Dati salvati.");
}
```

È un approccio lecito, soprattutto:

- nei primi esempi;
- nelle finestre semplici;
- quando vuoi mostrare il meccanismo base.

Ma non è sempre la soluzione migliore.

## Quando usare `Click`

Usa `Click` quando:

- il comportamento è piccolo e locale;
- l’esempio è didattico;
- la logica è strettamente legata alla finestra;
- non hai ancora introdotto commanding o MVVM.

## Quando iniziare a preferire i command

Se l’azione:

- deve essere riusabile;
- deve essere abilitata/disabilitata in base allo stato;
- non vuoi legarla troppo al code-behind;
- fa parte di un’architettura più pulita,

allora i command diventano più adatti.

In questo capitolo basta fissare l’idea: `Button` è il punto naturale in cui il discorso su command inizierà ad avere senso.

## Proprietà utili da conoscere

### `Content`
Definisce il contenuto del pulsante.

### `Padding`
Controlla lo spazio interno tra bordo e contenuto.

### `IsEnabled`
Abilita o disabilita il pulsante.

```xml
<Button Content="Salva"
        IsEnabled="{Binding PuòSalvare}" />
```

Questa proprietà è fondamentale nelle interfacce ben progettate: un pulsante disabilitato comunica che l’azione esiste, ma non è disponibile in quel momento.

## Esempio con stato della UI

```xml
<StackPanel Margin="16">
    <TextBox Width="240"
             Text="{Binding Nome}" />
    <Button Content="Conferma"
            Width="100"
            Margin="0,12,0,0"
            IsEnabled="{Binding NomeValido}" />
</StackPanel>
```

Qui il pulsante non è solo un oggetto cliccabile: riflette lo stato della schermata.

## Errori tipici

### Mettere troppa logica nel `Click`

All’inizio sembra comodo, ma dopo poco il code-behind si riempie di validazioni, accessi dati, aggiornamenti UI e logica di dominio tutti mescolati.

### Usare il pulsante per funzioni poco chiare

Se il testo dice “Vai”, “Esegui”, “Ok” senza contesto, l’utente capisce meno. Un pulsante dovrebbe dichiarare bene l’azione.

### Riempire il contenuto di layout troppo complessi

È possibile farlo, ma non sempre è una buona idea. Se il contenuto del pulsante diventa troppo ricco, la UI rischia di diventare pesante o confusa.

### Dimenticare lo stato `IsEnabled`

Lasciare sempre cliccabile un’azione non valida genera errori evitabili e peggiora l’esperienza utente.

## Quando usare `Button` e quando no

### Usa `Button` quando:
- l’utente deve lanciare un’azione;
- vuoi un comando chiaro e visibile;
- il controllo ha un ruolo operativo.

### Non usarlo quando:
- stai rappresentando una scelta persistente → forse serve `CheckBox`, `RadioButton` o `ComboBox`;
- il comportamento è più vicino a una navigazione o a una selezione in lista;
- stai forzando un pattern “a pulsanti” dove la UI richiederebbe un’altra metafora.

## Il punto da portare a casa

`Button` è molto più di un rettangolo con del testo. In WPF è un controllo ricco, flessibile e centrale nel dialogo tra utente e applicazione. Usarlo bene significa scegliere azioni chiare, contenuti leggibili e una logica che non soffochi subito nel code-behind.
