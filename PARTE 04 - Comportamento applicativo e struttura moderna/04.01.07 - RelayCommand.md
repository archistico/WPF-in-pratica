# 04.01.07 - `RelayCommand`

## La soluzione pratica che evita una classe per ogni comando

Dopo aver visto `ICommand`, è facile notare un problema:
scrivere una classe personalizzata per ogni comando può diventare verbose.

Per esempio, per ogni azione dovresti creare una classe con:
- `Execute`;
- `CanExecute`;
- `CanExecuteChanged`.

Dal punto di vista didattico va bene.
Dal punto di vista pratico, però, presto diventa scomodo.

È qui che entra in gioco `RelayCommand`.

## Che cos'è `RelayCommand`

`RelayCommand` è una classe di utilità che implementa `ICommand` e permette di definire il comportamento del comando passando delegati o lambda.

In pratica, invece di creare una classe separata per ogni comando, puoi scrivere:

- che cosa fare (`Execute`);
- opzionalmente, quando farlo (`CanExecute`);

direttamente nel view model.

## Un esempio classico di implementazione

```csharp
using System;
using System.Windows.Input;

public class RelayCommand : ICommand
{
    private readonly Action<object?> _execute;
    private readonly Func<object?, bool>? _canExecute;

    public RelayCommand(Action<object?> execute, Func<object?, bool>? canExecute = null)
    {
        _execute = execute;
        _canExecute = canExecute;
    }

    public event EventHandler? CanExecuteChanged;

    public bool CanExecute(object? parameter)
    {
        return _canExecute?.Invoke(parameter) ?? true;
    }

    public void Execute(object? parameter)
    {
        _execute(parameter);
    }

    public void RaiseCanExecuteChanged()
    {
        CanExecuteChanged?.Invoke(this, EventArgs.Empty);
    }
}
```

## Come si usa nel view model

```csharp
public class DocumentoViewModel
{
    public ICommand SalvaCommand { get; }

    public DocumentoViewModel()
    {
        SalvaCommand = new RelayCommand(
            _ => Salva(),
            _ => PuòSalvare());
    }

    private void Salva()
    {
        // logica di salvataggio
    }

    private bool PuòSalvare()
    {
        return true;
    }
}
```

## Perché è così comodo

Perché:
- riduce molto il codice ripetitivo;
- rende i comandi immediati nel view model;
- si integra benissimo con MVVM;
- tiene la logica vicino al contesto che la usa.

Questo è uno dei motivi per cui `RelayCommand` è diventato uno standard di fatto in moltissimi progetti WPF.

## Con o senza parametro

Puoi usare `RelayCommand` anche in forma molto semplice, senza sfruttare davvero il parametro:

```csharp
SalvaCommand = new RelayCommand(_ => Salva());
```

Oppure usando il parametro se serve:

```csharp
ApriClienteCommand = new RelayCommand(
    cliente => ApriCliente((Cliente)cliente!));
```

Questa flessibilità è molto utile.

## Perché si chiama “relay”

L'idea è che la classe “inoltra” l'esecuzione verso il codice che le passi.
Non definisce la logica del comando in sé, ma fa da ponte tra `ICommand` e le lambda/azioni del tuo view model.

## Un punto molto importante

`RelayCommand` non fa parte del framework WPF come tipo standard unico e universale.
È una convenzione molto diffusa, e spesso ogni progetto o libreria MVVM ne offre una propria implementazione.

Questo è bene saperlo.

## Errori tipici

### Pensare che `RelayCommand` sia magia
In realtà è solo una implementazione molto pratica di `ICommand`.

### Non prevedere `CanExecute` quando servirebbe
Si perde uno dei vantaggi più forti del comando.

### Non notificare `CanExecuteChanged` quando la condizione cambia
La UI non si aggiorna nell'abilitazione.

## Regola pratica

Se `ICommand` è il contratto, `RelayCommand` è molto spesso il modo più pratico di usarlo nel lavoro quotidiano con MVVM.
