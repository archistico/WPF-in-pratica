# 04.03.05 - Creazione della tabella

## Il primo passo reale sul database

Per poter fare CRUD, il database deve avere una tabella coerente con il modello.

Nel nostro esempio, il modello è `Prodotto`, quindi la tabella più naturale è `Prodotti`.

Questo sottocapitolo è importante perché chiarisce come il modello in memoria trovi una corrispondenza minima nel database SQLite.

## Una struttura semplice della tabella

Per il nostro modello:

```csharp
public class Prodotto
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";
    public decimal Prezzo { get; set; }
}
```

una tabella SQLite semplice può essere:

```sql
CREATE TABLE IF NOT EXISTS Prodotti (
    Id INTEGER PRIMARY KEY AUTOINCREMENT,
    Nome TEXT NOT NULL,
    Prezzo REAL NOT NULL
);
```

## Come leggere questa struttura

### `Id INTEGER PRIMARY KEY AUTOINCREMENT`
È la chiave primaria e SQLite la incrementa automaticamente per ogni nuovo record.

### `Nome TEXT NOT NULL`
Il nome del prodotto è obbligatorio.

### `Prezzo REAL NOT NULL`
Il prezzo è obbligatorio.

## Perché questa definizione è buona per il capitolo

Perché è semplice ma realistica.
Non introduce ancora complessità superflue, ma è sufficiente per:
- caricare dati;
- inserire record;
- aggiornare;
- eliminare.

È l'equilibrio giusto per un capitolo didattico.

## Dove fare la creazione della tabella

La creazione della tabella non va fatta nella view.
Non va fatta nel `DataGrid`.
Non va fatta nel modello.

Il posto più naturale è nel livello dati, di solito nel repository o in un servizio di inizializzazione.

Questo mantiene la responsabilità nel punto giusto.

## Un esempio C# con SQLite

```csharp
using Microsoft.Data.Sqlite;

public void EnsureDatabase()
{
    using var connection = new SqliteConnection("Data Source=prodotti.db");
    connection.Open();

    var command = connection.CreateCommand();
    command.CommandText =
    @"
        CREATE TABLE IF NOT EXISTS Prodotti (
            Id INTEGER PRIMARY KEY AUTOINCREMENT,
            Nome TEXT NOT NULL,
            Prezzo REAL NOT NULL
        );
    ";

    command.ExecuteNonQuery();
}
```

## Perché `IF NOT EXISTS` è importante

Perché consente di eseguire il controllo/creazione in modo sicuro senza tentare di ricreare ogni volta una tabella già esistente.

È una soluzione molto pratica per tool desktop e piccoli gestionali.

## Un punto importante sulla corrispondenza tipi

Nel capitolo non serve entrare troppo presto in tutte le sfumature dei tipi SQLite.
Ma è utile capire la corrispondenza base:

- `INTEGER` ↔ `int`
- `TEXT` ↔ `string`
- `REAL` ↔ `decimal`/`double` con la dovuta attenzione

Per un esempio introduttivo, questa mappa è più che sufficiente.

## Quando questa logica viene eseguita

Di solito:
- all'avvio dell'app;
- oppure all'inizializzazione del repository;
- oppure in un punto dedicato di bootstrap.

L'importante è che la struttura dati esista prima del caricamento vero e proprio.

## Errore tipico

L'errore classico è:
- creare la tabella da un punto casuale della UI;
- oppure supporre che il file SQLite sia già sempre corretto;
- oppure mischiare query di creazione, lettura e UI nello stesso posto.

## Regola pratica

La creazione della tabella è una responsabilità del livello dati.
Va eseguita in modo chiaro e centralizzato, così la schermata CRUD può partire su una base persistente affidabile.
