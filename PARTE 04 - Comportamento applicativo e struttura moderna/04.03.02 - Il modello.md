# 04.03.02 - Il modello

## Il primo mattoncino: l'oggetto che rappresenta il dato

Prima di parlare di repository, query e `DataGrid`, serve definire il modello dei dati.

Il modello è la classe che rappresenta l'entità su cui farai CRUD.
Nel capitolo conviene usare un esempio molto semplice e realistico:
un prodotto.

## Un esempio di modello

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;

public class Prodotto : INotifyPropertyChanged
{
    private int _id;
    private string _nome = "";
    private decimal _prezzo;

    public int Id
    {
        get => _id;
        set
        {
            if (_id != value)
            {
                _id = value;
                OnPropertyChanged();
            }
        }
    }

    public string Nome
    {
        get => _nome;
        set
        {
            if (_nome != value)
            {
                _nome = value;
                OnPropertyChanged();
            }
        }
    }

    public decimal Prezzo
    {
        get => _prezzo;
        set
        {
            if (_prezzo != value)
            {
                _prezzo = value;
                OnPropertyChanged();
            }
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

## Perché usare `INotifyPropertyChanged`

Qui il modello non serve solo a trasportare dati tra repository e database.
Serve anche a vivere bene nella UI WPF.

Per questo è molto utile che implementi `INotifyPropertyChanged`, così:
- il `DataGrid` può reagire ai cambi di proprietà;
- eventuali controlli bindati ai campi restano sincronizzati;
- l'oggetto funziona bene nella schermata CRUD.

## Quali proprietà ha senso mettere

Nel modello metti le proprietà dell'entità reale.
Nel nostro esempio:
- `Id`
- `Nome`
- `Prezzo`

Questo basta per mostrare bene il pattern.

In un'app reale potresti avere anche:
- `Descrizione`
- `Categoria`
- `Attivo`
- `DataCreazione`

ma per la comprensione del capitolo conviene restare essenziali.

## Il ruolo dell'`Id`

L'`Id` è molto importante perché rappresenta la chiave primaria nel database.
In una tabella SQLite sarà tipicamente usato per:
- identificare un record;
- capire se un oggetto esiste già nel database;
- aggiornare o eliminare la riga corretta.

Questo rende il modello molto più concreto anche dal lato CRUD.

## Perché il modello non deve contenere la logica del database

Un errore comune è iniziare a mettere dentro il modello:
- query SQL;
- metodi di salvataggio;
- accesso diretto a SQLite.

Questo complica il progetto e mescola responsabilità diverse.

Il modello dovrebbe rappresentare il dato.
La persistenza vive meglio nel repository.

## Perché il modello non coincide col `DataGrid`

Anche questo va chiarito bene:
il `DataGrid` non è il modello.
Il `DataGrid` mostra una collezione di modelli.

Questa distinzione semplice aiuta a capire tutta la struttura successiva del capitolo.

## Un criterio utile

Quando progetti il modello per una schermata CRUD WPF, chiediti:
- quali sono i campi dell'entità?
- quali proprietà devono essere mostrabili e modificabili nella UI?
- quali devono notificare il cambiamento?

Se rispondi bene a queste tre domande, la classe parte già nel modo giusto.

## Errore tipico

L'errore classico è usare un modello troppo povero per il binding oppure troppo carico di responsabilità database.
Qui la strada sana è una classe dati pulita, bindabile, ma non responsabile direttamente della persistenza.

## Regola pratica

Il modello di un CRUD WPF con SQLite deve essere:
- rappresentativo dei dati reali;
- comodo da bindare;
- separato dalla logica di accesso al database.
