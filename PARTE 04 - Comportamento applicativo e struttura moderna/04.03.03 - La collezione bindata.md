# 04.03.03 - La collezione bindata

## Il punto in cui la UI incontra davvero i dati

Dopo il modello, il secondo mattone fondamentale è la collezione bindata.

La UI, e in particolare controlli come:
- `ListBox`
- `ListView`
- `DataGrid`

non lavorano direttamente sul database SQLite.
Lavorano su una raccolta di oggetti in memoria.

Questa raccolta è il ponte immediato tra dati e presentazione.

## La scelta naturale: `ObservableCollection<T>`

Per uno scenario CRUD con UI dinamica, la scelta più naturale è quasi sempre:

```csharp
public ObservableCollection<Prodotto> Prodotti { get; } = new();
```

Perché?
Perché `ObservableCollection<T>` notifica quando:
- aggiungi un elemento;
- rimuovi un elemento;
- svuoti la raccolta.

Questo la rende perfetta per una UI bindata.

## Un esempio semplice in un view model

```csharp
using System.Collections.ObjectModel;
using System.ComponentModel;
using System.Runtime.CompilerServices;

public class ProdottiViewModel : INotifyPropertyChanged
{
    public ObservableCollection<Prodotto> Prodotti { get; } = new();

    private Prodotto? _prodottoSelezionato;

    public Prodotto? ProdottoSelezionato
    {
        get => _prodottoSelezionato;
        set
        {
            if (_prodottoSelezionato != value)
            {
                _prodottoSelezionato = value;
                OnPropertyChanged();
            }
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

## Perché la collezione bindata è così importante

Perché è l'oggetto che la UI osserva davvero.
Quando carichi i dati dal repository, non li stai “mostrando direttamente”.
Li stai mettendo in una raccolta che la UI può osservare e visualizzare.

Questo è un punto decisivo del modello mentale corretto.

## Perché non usare una `List<T>` normale

Una `List<T>` può contenere i dati, ma non notifica le modifiche strutturali.
Quindi se fai:
- `Add`
- `Remove`
- `Clear`

la UI non reagisce bene come con una `ObservableCollection<T>`.

In una schermata CRUD dinamica, questo è un limite molto importante.

## Il ruolo della selezione

In molti CRUD serve anche la proprietà dell'elemento selezionato:

```csharp
public Prodotto? ProdottoSelezionato { get; set; }
```

Meglio ancora se notificante.

Questo permette alla UI di:
- sapere quale riga è selezionata;
- compilare campi di dettaglio;
- aggiornare o eliminare l'elemento corretto.

## Che cosa contiene davvero la raccolta

La raccolta contiene oggetti `Prodotto` già caricati dal database o appena creati in memoria.

È importante capire che:
- non è una query viva;
- non è il database;
- non è solo una lista “temporanea casuale”.

È la rappresentazione in memoria su cui la UI lavora.

## Un piccolo esempio mentale

Quando il repository legge 20 prodotti da SQLite, quei 20 prodotti vengono trasformati in oggetti .NET e messi in `Prodotti`.
Il `DataGrid` mostra `Prodotti`.
Se aggiungi un nuovo `Prodotto` alla collezione, il `DataGrid` lo vede.

Questo è il ruolo corretto della raccolta bindata.

## Errore tipico

L'errore classico è:
- usare una `List<T>` e aspettarsi aggiornamenti automatici;
- oppure rimpiazzare continuamente la raccolta senza un criterio chiaro;
- oppure confondere la raccolta bindata con il repository o col database.

## Regola pratica

La collezione bindata è la rappresentazione in memoria dei dati mostrati dalla UI.
In un CRUD WPF, è quasi sempre una `ObservableCollection<T>` che la UI osserva direttamente.
