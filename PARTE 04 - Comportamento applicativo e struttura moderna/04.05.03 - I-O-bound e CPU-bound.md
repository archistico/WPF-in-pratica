# 04.05.03 - I/O-bound e CPU-bound

## Il punto che chiarisce davvero quando `await` basta e quando no

Uno degli equivoci più comuni su `async/await` è questo:
pensare che ogni operazione lunga si risolva automaticamente scrivendo `await`.

Non è così.

Per capire quando la UI resta davvero reattiva, devi distinguere molto bene tra:

- operazioni **I/O-bound**
- operazioni **CPU-bound**

Questa distinzione è decisiva.

## Che cos'è un'operazione I/O-bound

Un'operazione I/O-bound è un'operazione che passa molto tempo ad aspettare qualcosa di esterno, per esempio:
- disco;
- rete;
- database;
- file system;
- risposta di un servizio.

In questi casi, il lavoro non è tanto “calcolo pesante”, ma attesa.

Esempi:
- leggere un file grande;
- scaricare dati da un'API;
- eseguire query o accessi a storage;
- salvare file.

Qui `async/await` è spesso perfetto.

## Che cos'è un'operazione CPU-bound

Un'operazione CPU-bound è invece un lavoro di calcolo intenso che usa molto la CPU.

Esempi:
- elaborazione immagini;
- compressione;
- hashing massivo;
- parsing pesante;
- generazione di report complessi;
- trasformazioni su migliaia di record;
- analisi numeriche o algoritmiche.

Qui il problema non è aspettare risorse esterne.
Il problema è consumare tempo di CPU.

## Perché questa distinzione conta in WPF

Perché le due categorie richiedono strategie diverse per mantenere reattiva la UI.

### I/O-bound
Spesso basta usare API asincrone e `await`.

### CPU-bound
Molto spesso devi spostare il lavoro fuori dal thread UI, per esempio con `Task.Run(...)`.

Questa è una delle distinzioni più utili dell'intero capitolo.

## Un esempio I/O-bound

```csharp
private async Task<string> LeggiTestoAsync(string path)
{
    return await File.ReadAllTextAsync(path);
}
```

Qui stai aspettando un'operazione di I/O.
La UI può restare reattiva senza bisogno di calcolo pesante nel thread principale.

## Un esempio CPU-bound

```csharp
private string CalcoloPesante()
{
    // simulazione di lavoro intenso
    for (int i = 0; i < 500000000; i++)
    {
    }

    return "Fatto";
}
```

Se chiami questo metodo direttamente dal thread UI, la finestra si blocca.

Qui non basta “mettere await a caso”.
Serve eseguire il lavoro in modo appropriato, per esempio:

```csharp
private async Task<string> CalcoloPesanteAsync()
{
    return await Task.Run(() => CalcoloPesante());
}
```

## Il punto chiave

### I/O-bound
Aspetti qualcosa.

### CPU-bound
Stai facendo lavorare tantissimo la CPU.

Questa frase è una semplificazione molto utile.

## Un criterio pratico

Quando una schermata si blocca durante un lavoro lungo, chiediti:
- sto aspettando disco/rete/risorse esterne?
- oppure sto facendo calcolo intenso?

La risposta ti orienta subito sulla soluzione corretta.

## Perché questo chiarisce molti errori

Molti provano a usare:
- `await` su codice che in realtà resta CPU-bound e blocca comunque;
- oppure `Task.Run` dove basterebbe una API asincrona nativa.

Entrambi gli errori nascono da una cattiva distinzione tra I/O-bound e CPU-bound.

## Un esempio realistico WPF

### I/O-bound
Importazione di file da disco con letture asincrone.

### CPU-bound
Generazione di miniature e hashing di migliaia di immagini.

Nel primo caso `await` su API asincrone è naturale.
Nel secondo può servire spostare il calcolo fuori dal thread UI.

## Errore tipico

L'errore classico è pensare che “lungo” significhi sempre la stessa cosa.
In realtà un'operazione lunga può esserlo per attesa o per calcolo, e questo cambia completamente l'approccio.

## Regola pratica

Se l'operazione è I/O-bound, cerca prima una soluzione asincrona naturale con `await`.
Se è CPU-bound, chiediti se il lavoro debba essere eseguito fuori dal thread UI, spesso con `Task.Run`.
