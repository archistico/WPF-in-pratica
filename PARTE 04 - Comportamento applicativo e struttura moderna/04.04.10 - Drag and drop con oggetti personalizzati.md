# 04.04.10 - Drag and drop con oggetti personalizzati

## Il drag and drop non è limitato a testo o file

Uno dei punti più interessanti di WPF è che il drag and drop può lavorare molto bene anche con oggetti del tuo modello applicativo.

Questo significa che puoi trascinare:
- clienti;
- prodotti;
- righe documento;
- task;
- allegati;
- nodi di una struttura ad albero.

Non sei limitato a formati banali.

## Perché è così utile

Perché nelle applicazioni reali ciò che ti interessa spostare o copiare è spesso proprio un oggetto di dominio o di presentazione, non una semplice stringa.

Questo rende il drag and drop molto più potente e vicino al lavoro vero.

## Un esempio semplice

Modello:

```csharp
public class Prodotto
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";
    public decimal Prezzo { get; set; }
}
```

Avvio del drag:

```csharp
if (listBox.SelectedItem is Prodotto prodotto)
{
    DragDrop.DoDragDrop(
        listBox,
        prodotto,
        DragDropEffects.Copy | DragDropEffects.Move);
}
```

Ricezione nel target:

```csharp
if (e.Data.GetDataPresent(typeof(Prodotto)))
{
    var prodotto = (Prodotto)e.Data.GetData(typeof(Prodotto))!;
    // gestione del prodotto
}
```

## Perché questo approccio è molto naturale

Perché ti permette di trasferire direttamente l'oggetto che la UI già sta mostrando.
Non devi sempre serializzare a mano testo o ricostruire il contesto in modo artificiale.

Questo rende il codice spesso più pulito.

## Un punto importante sul significato dell'oggetto

Attenzione però:
se passi un oggetto personalizzato, il target riceve un riferimento a quell'oggetto .NET nel contesto dell'applicazione.

Questo è molto comodo, ma implica che devi progettare bene cosa significhi:
- copiarlo;
- spostarlo;
- riusarlo;
- modificarlo.

Soprattutto quando lavori su collezioni diverse.

## Caso `Copy` e caso `Move`

### `Move`
Puoi voler spostare lo stesso oggetto da una collection a un'altra.

### `Copy`
Potresti voler creare una copia logica dell'oggetto, non semplicemente aggiungere lo stesso riferimento in due posti.

Questo è molto importante da decidere nel codice del drop.

## Un esempio realistico

Immagina una schermata di produzione o gestione attività:
- trascini una `TaskItem` dalla lista “Da fare” alla lista “In corso”.

Il target può:
- aggiungere lo stesso oggetto alla nuova collection e rimuoverlo dalla vecchia;
- oppure creare una nuova istanza con stato diverso.

Dipende dal significato dell'operazione.

## Quando è molto comodo

È molto comodo in scenari interni alla tua applicazione, dove:
- sorgente e target condividono lo stesso modello;
- vuoi lavorare in modo fortemente typed;
- non hai bisogno di passare per stringhe o file.

## Quando serve più cautela

Serve più cautela quando:
- il payload potrebbe uscire dal perimetro della tua app;
- vuoi interoperabilità con altre app;
- il dato deve essere serializzato;
- hai bisogno di formati multipli.

Ma per la maggior parte degli esempi interni, l'oggetto personalizzato è una scelta ottima.

## Errore tipico

L'errore classico è pensare che trascinare un oggetto personalizzato equivalga automaticamente a “copiarlo bene”.
In realtà stai trasferendo un payload: poi devi decidere cosa farne davvero.

## Regola pratica

Nel drag and drop interno a un'app WPF, trascinare oggetti personalizzati è spesso la soluzione più naturale.
L'importante è ricordare che devi definire chiaramente il significato di copy, move e riuso dell'oggetto ricevuto.
