# 04.03.07 - Caricare i dati nella raccolta bindata

## Il passaggio che collega davvero repository e UI

A questo punto hai:
- il modello `Prodotto`;
- la raccolta bindata `ObservableCollection<Prodotto>`;
- il repository che legge da SQLite.

Ora serve il passaggio che unisce davvero questi pezzi:
caricare i dati letti dal repository nella raccolta osservata dalla UI.

Questo è uno snodo centrale del capitolo.

## Il principio corretto

Il repository restituisce oggetti.
La UI osserva una `ObservableCollection<Prodotto>`.

Quindi il caricamento consiste nel:
1. leggere i prodotti dal repository;
2. aggiornare la raccolta bindata.

## Un esempio in un view model

```csharp
using System.Collections.ObjectModel;
using System.ComponentModel;
using System.Runtime.CompilerServices;

public class ProdottiViewModel : INotifyPropertyChanged
{
    private readonly ProdottoRepository _repository = new();

    public ObservableCollection<Prodotto> Prodotti { get; } = new();

    private Prodotto? _prodottoSelezionato;

    public Prodotto? ProdottoSelezionato
    {
        get => _prodottoSelezionato;
        set
        {
            if (_prodottoSelezionato != value)
            {
                _prodottoSelezionato = value;
                OnPropertyChanged();
            }
        }
    }

    public ProdottiViewModel()
    {
        _repository.EnsureDatabase();
        LoadData();
    }

    public void LoadData()
    {
        Prodotti.Clear();

        foreach (var prodotto in _repository.GetAll())
        {
            Prodotti.Add(prodotto);
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

## Perché `Clear()` + `Add()` è istruttivo

Questa tecnica è molto chiara perché:
- svuoti la raccolta osservata dalla UI;
- la riempi con i dati letti dal repository;
- la UI reagisce agli aggiornamenti della collezione.

Per un capitolo didattico è una forma molto leggibile.

## Perché non assegnare sempre una nuova collezione al volo

Potresti anche sostituire l'intera collection, ma all'inizio conviene capire bene la logica:
la UI osserva una raccolta.
Quindi puoi aggiornare il contenuto di quella raccolta.

Questa forma rende più trasparente il rapporto tra:
- dati letti;
- raccolta bindata;
- notifiche alla UI.

## Quando si chiama `LoadData()`

Tipicamente:
- all'avvio della schermata;
- dopo certe operazioni di sincronizzazione;
- quando vuoi ricaricare lo stato dal database.

È un metodo molto importante perché rappresenta il momento in cui il database torna a riflettersi nella UI.

## Un punto da fissare bene

Il `DataGrid` non esegue la query.
La `ObservableCollection` non sa leggere da SQLite.
Il repository non aggiorna da solo la UI.

È il view model che orchestra:
- lettura;
- popolamento della raccolta;
- riflesso nella UI.

Questa distinzione è molto sana.

## Un piccolo test mentale utile

Se il database contiene 10 prodotti e la raccolta è vuota:
- `GetAll()` legge 10 oggetti;
- `LoadData()` li aggiunge alla `ObservableCollection`;
- il `DataGrid` mostra 10 righe.

Questo è il flusso corretto.

## Errore tipico

L'errore classico è:
- bindare la UI direttamente a una `List<T>` ritornata dal repository;
- oppure pensare che il repository debba “parlare” direttamente al `DataGrid`;
- oppure ricaricare tutto in modo poco chiaro senza una raccolta bindata stabile.

## Regola pratica

Il caricamento corretto in un CRUD WPF è:
repository → oggetti → raccolta bindata → UI.
Se questa catena è chiara, anche le operazioni successive diventano molto più semplici.
