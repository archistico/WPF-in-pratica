# 04.02.04 - Il ViewModel della shell

## Il centro di controllo della navigazione interna

Se la shell è la struttura principale della finestra, allora il suo view model è il punto che governa lo stato della navigazione.

Qui vive una delle idee più importanti del capitolo:
la shell non cambia view “a mano” nel code-behind, ma espone uno stato che dice quale contenuto deve essere attivo.

Questo stato vive molto bene nel view model della shell.

## Che cosa contiene di solito

Un `ShellViewModel` contiene spesso:
- il view model corrente mostrato nell'area contenuto;
- i comandi per cambiare sezione;
- eventuali dati comuni della shell;
- eventuale stato globale della finestra.

Il cuore, però, è quasi sempre la proprietà che rappresenta la vista o il view model attivo.

## Un esempio semplice

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;
using System.Windows.Input;

public class ShellViewModel : INotifyPropertyChanged
{
    private object? _currentViewModel;

    public object? CurrentViewModel
    {
        get => _currentViewModel;
        set
        {
            if (_currentViewModel != value)
            {
                _currentViewModel = value;
                OnPropertyChanged();
            }
        }
    }

    public ICommand ShowDashboardCommand { get; }
    public ICommand ShowClientiCommand { get; }

    public ShellViewModel()
    {
        ShowDashboardCommand = new RelayCommand(_ => CurrentViewModel = new DashboardViewModel());
        ShowClientiCommand = new RelayCommand(_ => CurrentViewModel = new ClientiViewModel());

        CurrentViewModel = new DashboardViewModel();
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

## Come leggere questo esempio

### `CurrentViewModel`
Rappresenta il contenuto corrente da mostrare nella shell.

### `ShowDashboardCommand` / `ShowClientiCommand`
Sono i comandi che cambiano il contenuto attivo.

### Costruttore
Imposta una vista iniziale e definisce la navigazione base.

Questo è già un modello di shell molto chiaro.

## Perché questo approccio è forte

Perché la shell non deve conoscere direttamente la logica della view.
Sa soltanto che:
- c'è una sezione corrente;
- i comandi la possono cambiare.

Questo è esattamente il tipo di responsabilità che un view model dovrebbe gestire bene.

## Perché usare `object` per `CurrentViewModel`

Perché ti permette di assegnare oggetti di tipi diversi:
- `DashboardViewModel`
- `ClientiViewModel`
- `OrdiniViewModel`

e lasciare che siano i `DataTemplate` a scegliere la vista corretta.

È una soluzione molto pratica in questo pattern.

## Un possibile miglioramento futuro

In progetti più strutturati, potresti usare:
- una classe base comune per i view model di navigazione;
- servizi di navigazione;
- gestione più centralizzata delle istanze.

Ma per un primo modello moderno e leggibile, questa struttura è già ottima.

## Quando il view model della shell è particolarmente importante

È importante quando:
- la finestra principale ha più sezioni;
- la navigazione è parte del comportamento applicativo;
- vuoi evitare logica sparsa nella view;
- vuoi una shell estendibile.

## Errori tipici

### Mettere la navigazione nel code-behind della finestra principale
Si perde una parte importante della pulizia del modello.

### Non avere una proprietà centrale che rappresenti il contenuto attivo
La navigazione diventa più frammentata.

### Far conoscere alla shell troppi dettagli delle singole viste
La separazione peggiora.

## Regola pratica

Il view model della shell deve essere il punto che decide quale contenuto è attivo e quali comandi permettono di cambiare sezione.
Se questa responsabilità è chiara, la navigazione interna diventa molto più ordinata.
