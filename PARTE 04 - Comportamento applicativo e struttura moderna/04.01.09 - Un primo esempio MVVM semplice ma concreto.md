# 04.01.09 - Un primo esempio MVVM semplice ma concreto

## L'obiettivo dell'esempio

Dopo tanta teoria, serve un esempio che faccia vedere MVVM in modo semplice ma reale.

L'obiettivo qui è mostrare una schermata con:
- una proprietà bindata;
- un comando;
- un view model;
- una view pulita, senza logica diretta del pulsante.

Non vogliamo ancora un esempio enorme.
Vogliamo un esempio che chiarisca bene il modello.

## Il view model

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;
using System.Windows.Input;

public class MainViewModel : INotifyPropertyChanged
{
    private string _nome = "";

    public string Nome
    {
        get => _nome;
        set
        {
            if (_nome != value)
            {
                _nome = value;
                OnPropertyChanged();
                ((RelayCommand)SalutaCommand).RaiseCanExecuteChanged();
            }
        }
    }

    public ICommand SalutaCommand { get; }

    public MainViewModel()
    {
        SalutaCommand = new RelayCommand(
            _ => Saluta(),
            _ => !string.IsNullOrWhiteSpace(Nome));
    }

    private void Saluta()
    {
        Nome = $"Ciao, {Nome}";
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

## Il `RelayCommand`

```csharp
using System;
using System.Windows.Input;

public class RelayCommand : ICommand
{
    private readonly Action<object?> _execute;
    private readonly Func<object?, bool>? _canExecute;

    public RelayCommand(Action<object?> execute, Func<object?, bool>? canExecute = null)
    {
        _execute = execute;
        _canExecute = canExecute;
    }

    public event EventHandler? CanExecuteChanged;

    public bool CanExecute(object? parameter)
    {
        return _canExecute?.Invoke(parameter) ?? true;
    }

    public void Execute(object? parameter)
    {
        _execute(parameter);
    }

    public void RaiseCanExecuteChanged()
    {
        CanExecuteChanged?.Invoke(this, EventArgs.Empty);
    }
}
```

## La view

```xml
<StackPanel Margin="20">
    <TextBlock Text="Nome" Margin="0,0,0,4"/>
    <TextBox Text="{Binding Nome, UpdateSourceTrigger=PropertyChanged}"
             Width="220"
             Margin="0,0,0,12"/>

    <Button Content="Saluta"
            Width="120"
            Command="{Binding SalutaCommand}"/>
</StackPanel>
```

## Il code-behind della finestra

```csharp
public partial class MainWindow : Window
{
    public MainWindow()
    {
        InitializeComponent();
        DataContext = new MainViewModel();
    }
}
```

## Che cosa mostra davvero questo esempio

### 1. La view non contiene la logica del pulsante
Il bottone non ha `Click`.

### 2. La UI si lega al view model
`TextBox` e `Button` lavorano su proprietà e comandi del `DataContext`.

### 3. Il comando sa quando può essere eseguito
Se `Nome` è vuoto o spazi, il pulsante può essere disabilitato.

### 4. Il modello mentale è già MVVM
Anche se l'esempio è piccolo, i ruoli sono distinti:
- view;
- view model;
- comando.

## Perché è un esempio buono

Perché non è troppo teorico e non è troppo grande.
Mostra già il vantaggio pratico:
la UI non governa direttamente l'azione, ma la espone tramite binding.

## Dove si vede la differenza col code-behind tradizionale

Nel modello tradizionale avresti:
- `Click`;
- lettura del testo da `TextBox`;
- logica del pulsante in finestra;
- abilitazione/disabilitazione gestita a mano.

Qui, invece, il view model espone:
- il dato (`Nome`);
- il comando (`SalutaCommand`);
- la condizione di eseguibilità.

## Errori tipici

### Pensare che MVVM richieda esempi enormi
Anche un esempio piccolo può mostrare perfettamente il modello.

### Non aggiornare `CanExecuteChanged` quando la proprietà cambia
Il pulsante non riflette bene lo stato.

### Lasciare logica del comando nella view
Si perde il vantaggio dell'architettura.

## Regola pratica

Il primo esempio MVVM corretto deve mostrare almeno questo:
- view pulita;
- view model con stato;
- comando bindato;
- logica fuori dal pulsante.
Se questi elementi ci sono, il modello è già chiaro.
