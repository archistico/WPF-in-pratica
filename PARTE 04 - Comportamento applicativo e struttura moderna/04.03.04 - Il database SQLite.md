# 04.03.04 - Il database SQLite

## Il livello persistente dell'applicazione

Dopo modello e raccolta bindata, arriva il livello che custodisce davvero i dati nel tempo:
il database SQLite.

In un CRUD WPF con SQLite, il database ha un ruolo molto chiaro:
- conserva i dati in modo persistente;
- li rende disponibili al caricamento;
- riceve inserimenti, aggiornamenti ed eliminazioni.

La UI non lo sostituisce.
La collezione bindata non lo sostituisce.
È il livello persistente del sistema.

## Perché SQLite è una buona scelta

SQLite è particolarmente adatto per applicazioni desktop perché:
- è leggero;
- non richiede un server separato;
- usa un semplice file locale;
- è molto pratico per tool, gestionali e software distribuiti su singola macchina;
- si integra bene con .NET.

Per un capitolo didattico ma realistico, è un'ottima scelta.

## Che cosa significa “database locale”

Di solito avrai un file, per esempio:

- `prodotti.db`

Questo file contiene:
- tabelle;
- record;
- struttura del database.

La tua applicazione WPF lo apre tramite una connection string SQLite.

## Un esempio di connection string

```csharp
string connectionString = "Data Source=prodotti.db";
```

Questa è la forma base più comune.

## Il rapporto con la UI

Questo punto va chiarito molto bene:
la UI non dovrebbe eseguire direttamente query SQL.
La UI dovrebbe lavorare su:
- view model;
- raccolte bindate;
- comandi.

L'accesso a SQLite vive meglio in un repository o in un livello dati dedicato.

## Il rapporto con il repository

Il repository usa SQLite per:
- creare la tabella se necessario;
- leggere i record;
- inserire nuovi dati;
- aggiornare quelli esistenti;
- eliminare righe.

Il repository è quindi il traduttore tra:
- file database;
- oggetti `Prodotto`.

## Un piccolo esempio mentale

Quando l'app parte:
1. apre il database SQLite;
2. il repository legge la tabella `Prodotti`;
3. crea oggetti `Prodotto`;
4. li passa alla raccolta bindata.

Quando l'utente salva:
1. la UI modifica o crea un oggetto;
2. il repository esegue `INSERT` o `UPDATE`;
3. la raccolta resta coerente col nuovo stato.

Questo è il flusso corretto.

## Perché è bene tenere SQLite al suo posto

Un errore molto comune è far entrare SQLite troppo in alto nella struttura:
- query nel code-behind;
- connessioni aperte dalla view;
- logica SQL sparsa.

Questo rende l'app più fragile.

Molto meglio pensare:
- SQLite = persistenza;
- repository = accesso ai dati;
- view model = orchestrazione e stato;
- UI = presentazione e input.

## Dove si vede il vantaggio

Si vede subito quando:
- devi cambiare il modo in cui carichi i dati;
- devi aggiungere validazione;
- vuoi passare in futuro a Entity Framework Core;
- vuoi testare meglio il comportamento della schermata.

Una struttura pulita rende tutto più gestibile.

## Errore tipico

L'errore classico è considerare SQLite come se fosse “la lista dei dati della UI”.
No.
SQLite è il magazzino persistente.
La UI lavora su oggetti caricati da lì.

## Regola pratica

Nel CRUD WPF con SQLite, il database è il livello persistente.
Va tenuto separato dalla UI e raggiunto tramite un livello dati chiaro, di solito il repository.
