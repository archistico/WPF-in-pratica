# 04.04.06 - Ricevere file trascinati da Esplora Risorse

## Uno degli scenari più utili nella pratica

Tra tutti i casi di drag and drop, uno dei più utili nelle applicazioni desktop è questo:
ricevere file trascinati da Esplora Risorse dentro la tua applicazione WPF.

Questo scenario compare molto spesso in software che devono:
- importare documenti;
- caricare immagini;
- analizzare file;
- allegare contenuti;
- elaborare cartelle o percorsi.

## Il formato che devi conoscere

Quando l'utente trascina file da Esplora Risorse, il payload tipico usa il formato:

```csharp
DataFormats.FileDrop
```

Questo è il punto chiave da ricordare.

## Un esempio minimo

XAML:

```xml
<Border AllowDrop="True"
        Drop="Border_Drop"
        Background="#F5F5F5"
        Padding="20">
    <TextBlock Text="Trascina qui uno o più file" />
</Border>
```

Code-behind:

```csharp
private void Border_Drop(object sender, DragEventArgs e)
{
    if (e.Data.GetDataPresent(DataFormats.FileDrop))
    {
        var files = (string[])e.Data.GetData(DataFormats.FileDrop)!;

        foreach (var file in files)
        {
            // qui gestisci il percorso del file
        }
    }
}
```

## Come leggere questo esempio

### `DataFormats.FileDrop`
Verifica che il payload contenga file trascinati dal sistema.

### `string[]`
Ricevi un array di percorsi completi.

### `foreach`
Puoi processare uno o più file.

Questo è già sufficiente per moltissimi casi reali.

## Perché è così comodo

Perché non devi implementare una selezione file manuale obbligatoria ogni volta.
Puoi offrire all'utente un gesto molto naturale:
- prendere file da Windows;
- trascinarli nella tua app;
- far partire l'elaborazione.

È una UX molto desktop e molto efficace.

## Un caso realistico

Immagina una schermata WPF per:
- importare immagini prodotto;
- caricare PDF;
- analizzare file di log;
- aggiungere allegati a una scheda.

Un riquadro con drag and drop da Esplora Risorse può essere molto più comodo di un semplice pulsante “Sfoglia...”.

## Eventi intermedi utili

Per migliorare il comportamento, è molto utile gestire anche `DragOver`:

```csharp
private void Border_DragOver(object sender, DragEventArgs e)
{
    if (e.Data.GetDataPresent(DataFormats.FileDrop))
        e.Effects = DragDropEffects.Copy;
    else
        e.Effects = DragDropEffects.None;

    e.Handled = true;
}
```

Questo dà feedback corretto all'utente prima del drop.

## Un punto importante

Ricevere i percorsi dei file non significa ancora averli importati o salvati nel tuo modello.
Come sempre:
- il drop ti consegna il dato;
- poi sei tu a decidere come elaborarlo.

Questo mantiene chiaro il ruolo del drag and drop.

## Quando il target giusto non è il controllo più piccolo

Spesso conviene rendere droppabile:
- un pannello;
- un contenitore;
- l'intero `UserControl`;
- una zona visivamente ampia.

Perché trascinare file su una piccola area poco chiara è scomodo per l'utente.

## Errore tipico

L'errore classico è non verificare `DataFormats.FileDrop` e assumere che il payload contenga sempre file.
Oppure dimenticare `AllowDrop="True"` sul target.

## Regola pratica

Per ricevere file da Esplora Risorse in WPF, controlla `DataFormats.FileDrop`, recupera l'array di percorsi e poi tratta quei file come input della tua applicazione, non come automatismo già concluso.
