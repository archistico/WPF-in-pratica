# 04.03.13 - Variante con Entity Framework Core

## Quando vuoi alzare il livello di astrazione

Finora il capitolo ha mostrato il CRUD con SQLite usando un repository che lavora con SQL e `SqliteConnection`.

È una scelta ottima per capire bene i meccanismi.

Ma in molti progetti moderni si preferisce una variante più astratta:
Entity Framework Core.

Questa sezione non sostituisce il modello precedente.
Lo affianca come variante utile da conoscere.

## Perché usare EF Core

Con Entity Framework Core puoi:
- modellare il database tramite classi;
- evitare molto SQL manuale;
- lavorare con un contesto tipizzato;
- usare LINQ;
- semplificare parte del repository.

Questo può essere molto comodo in applicazioni reali o in progetti che crescono.

## Il modello

```csharp
using System.ComponentModel;
using System.Runtime.CompilerServices;

public class Prodotto : INotifyPropertyChanged
{
    private int _id;
    private string _nome = "";
    private decimal _prezzo;

    public int Id
    {
        get => _id;
        set
        {
            if (_id != value)
            {
                _id = value;
                OnPropertyChanged();
            }
        }
    }

    public string Nome
    {
        get => _nome;
        set
        {
            if (_nome != value)
            {
                _nome = value;
                OnPropertyChanged();
            }
        }
    }

    public decimal Prezzo
    {
        get => _prezzo;
        set
        {
            if (_prezzo != value)
            {
                _prezzo = value;
                OnPropertyChanged();
            }
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged([CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}
```

## Il `DbContext`

```csharp
using Microsoft.EntityFrameworkCore;

public class AppDbContext : DbContext
{
    public DbSet<Prodotto> Prodotti => Set<Prodotto>();

    protected override void OnConfiguring(DbContextOptionsBuilder optionsBuilder)
    {
        optionsBuilder.UseSqlite("Data Source=prodotti.db");
    }
}
```

## Inizializzazione del database

```csharp
using var db = new AppDbContext();
db.Database.EnsureCreated();
```

Questo crea il database e la tabella se non esistono già.

## Un repository molto semplice

```csharp
using Microsoft.EntityFrameworkCore;
using System.Collections.Generic;
using System.Linq;

public class ProdottoRepositoryEf
{
    public List<Prodotto> GetAll()
    {
        using var db = new AppDbContext();
        return db.Prodotti
                 .OrderBy(p => p.Nome)
                 .ToList();
    }

    public void Insert(Prodotto prodotto)
    {
        using var db = new AppDbContext();
        db.Prodotti.Add(prodotto);
        db.SaveChanges();
    }

    public void Update(Prodotto prodotto)
    {
        using var db = new AppDbContext();
        db.Prodotti.Update(prodotto);
        db.SaveChanges();
    }

    public void Delete(Prodotto prodotto)
    {
        using var db = new AppDbContext();
        db.Prodotti.Remove(prodotto);
        db.SaveChanges();
    }
}
```

## Che cosa cambia rispetto alla variante precedente

### Prima
Scrivevi:
- query SQL;
- mapping dal reader agli oggetti;
- connessione esplicita.

### Con EF Core
Lavori più spesso con:
- `DbContext`;
- `DbSet`;
- LINQ;
- `SaveChanges()`.

Questo rende il codice spesso più sintetico.

## Che cosa non cambia

Non cambia il modello mentale corretto:
- la UI continua a lavorare su una raccolta bindata;
- il repository o il livello dati continua a fare da ponte;
- il binding continua a collegare UI e oggetti;
- il database continua a non coincidere con la `ObservableCollection`.

Questo punto va tenuto fermissimo.

## Quando preferire questa variante

Può essere preferibile quando:
- vuoi meno SQL manuale;
- il progetto usa già EF Core;
- il modello dati cresce;
- vuoi una persistenza più vicina al mondo ORM.

## Quando la variante manuale resta ottima

La variante manuale resta ottima quando:
- vuoi il massimo controllo;
- vuoi capire bene il flusso reale dei dati;
- il progetto è piccolo o molto locale;
- vuoi un capitolo più trasparente sul piano didattico.

## Errore tipico

L'errore classico è pensare che passando a EF Core cambi il modello di binding della UI.
Non cambia.
Cambia il modo in cui persisti i dati, non il fatto che la UI lavori su oggetti bindati.

## Regola pratica

Entity Framework Core semplifica il lato persistenza, ma non cambia la struttura mentale del CRUD WPF:
UI e binding lavorano su oggetti e collezioni in memoria, non direttamente sul database.
