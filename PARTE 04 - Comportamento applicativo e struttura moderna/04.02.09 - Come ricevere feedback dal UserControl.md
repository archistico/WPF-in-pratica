# 04.02.09 - Come ricevere feedback dal `UserControl`

## Anche qui la domanda va posta nel modo giusto

Dopo aver visto come “comunicare con” una vista interna, arriva l'altra metà del problema:
come ricevere feedback dal `UserControl`?

Anche qui, il rischio è formulare subito la questione in modo troppo diretto:
“come faccio a intercettare un evento del controllo figlio dalla shell?”

A volte si può fare, ma non è la forma più sana del modello.

La domanda migliore è:
“come faccio a far emergere verso l'esterno lo stato o l'esito di ciò che è accaduto nella vista interna?”

## La via più pulita: feedback tramite view model

Nel pattern MVVM, il feedback più sano passa spesso dal view model:
- proprietà che cambiano;
- stato esposto;
- comandi completati;
- richieste inviate tramite servizi o meccanismi di messaggistica.

In altre parole, la shell non riceve “il click del bottone interno”.
Riceve un cambiamento di stato o una conseguenza logica della sezione interna.

## Un esempio semplice

Immagina `ClientiViewModel` con una proprietà:

```csharp
private Cliente? _clienteSelezionato;

public Cliente? ClienteSelezionato
{
    get => _clienteSelezionato;
    set
    {
        if (_clienteSelezionato != value)
        {
            _clienteSelezionato = value;
            OnPropertyChanged();
        }
    }
}
```

La vista interna cambia la selezione.
Il feedback verso l'esterno, se serve, può essere letto o orchestrato a livello di stato del view model, non tramite chiamate dirette alla view.

## Un caso più concreto

Supponiamo che una vista interna permetta di confermare un documento.
La shell potrebbe dover:
- aggiornare un titolo;
- attivare una barra stato;
- mostrare una notifica.

Invece di intercettare il `Button.Click` interno, è più sano fare in modo che il view model della sezione:
- cambi uno stato;
- esponga un esito;
- notifichi una proprietà;
- invochi un servizio coordinato.

## Un possibile approccio semplice

Per esempio, il view model della shell può osservare o coordinare il view model corrente.

Oppure, in progetti un po' più evoluti, si possono usare:
- servizi di navigazione;
- servizi di dialogo;
- event aggregator o message bus locali al progetto.

Ma il punto del capitolo è questo:
il feedback non deve per forza passare dalla view concreta.

## Quando un evento dalla view può ancora avere senso

Può avere senso in scenari molto locali o tecnici:
- un'interazione puramente visiva;
- un dettaglio specifico del controllo;
- un comportamento che non vale la pena modellare nel view model.

Ma se il feedback ha significato applicativo, quasi sempre merita di emergere come stato o comando del view model.

## Un esempio mentale utile

### Soluzione meno sana
La shell prende il riferimento a `ClientiView` e intercetta cose interne alla sua UI.

### Soluzione più sana
La shell ragiona rispetto al `ClientiViewModel` e al suo stato.

Questa differenza è molto importante.

## Errore tipico

L'errore classico è usare il `UserControl` come se fosse un oggetto attivo da cui far “risalire eventi” di applicazione.
In molti casi il livello giusto di feedback è il view model, non la vista.

## Regola pratica

Se il feedback ha significato applicativo, fai in modo che emerga come stato, proprietà o comando del view model della vista interna.
Se senti il bisogno di ascoltare direttamente la UI figlia, chiediti prima se non stai saltando il livello giusto.
