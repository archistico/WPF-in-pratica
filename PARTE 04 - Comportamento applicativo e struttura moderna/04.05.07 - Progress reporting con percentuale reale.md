# 04.05.07 - Progress reporting con percentuale reale

## Non basta far muovere una barra: serve un progresso significativo

Una progress bar può essere cosmetica oppure informativa.

Se vuoi una UI davvero utile, il progresso dovrebbe rappresentare qualcosa di reale:
- file elaborati su totale;
- record processati su totale;
- immagini convertite su totale;
- passi completati su passi complessivi.

Questo è ciò che rende il progress reporting credibile.

## Il modello corretto

Il modello corretto è:
- conosci un totale;
- conosci quanti elementi o passi hai completato;
- calcoli una percentuale;
- riporti quella percentuale alla UI.

In formule:

```text
percentuale = completati / totale * 100
```

## Un esempio semplice

Supponiamo di dover elaborare 200 file.

```csharp
private async Task ElaboraFileAsync(List<string> files)
{
    int totale = files.Count;

    for (int i = 0; i < totale; i++)
    {
        await Task.Delay(20); // simulazione lavoro

        double percentuale = (i + 1) * 100.0 / totale;
        progressBar.Value = percentuale;
        lblStato.Content = $"Elaborati {i + 1} di {totale}";
    }
}
```

## Che cosa mostra questo esempio

### `totale`
È il numero complessivo di passi.

### `i + 1`
È il numero di passi completati finora.

### `percentuale`
È il valore reale da mostrare nella progress bar.

### `lblStato`
Aggiunge un feedback testuale utile, molto spesso apprezzabile quanto la barra stessa.

## Perché questo è meglio di avanzare “a caso”

A volte si vede codice che fa:
- `progressBar.Value += 1`
- oppure incrementi puramente estetici

senza rapporto col lavoro reale.

Questo può anche andare in demo o casi minimi, ma in software vero il progresso ha molto più valore se rappresenta una misura reale del lavoro completato.

## Un approccio ancora migliore: `IProgress<T>`

Nel codice moderno, un modo molto pulito per riportare progresso è usare `IProgress<T>`.

Esempio:

```csharp
private async Task ElaboraFileAsync(List<string> files, IProgress<int> progress)
{
    int totale = files.Count;

    for (int i = 0; i < totale; i++)
    {
        await Task.Delay(20);

        int percentuale = (i + 1) * 100 / totale;
        progress.Report(percentuale);
    }
}
```

Uso nella UI:

```csharp
private async void BtnAvvia_Click(object sender, RoutedEventArgs e)
{
    var progress = new Progress<int>(value =>
    {
        progressBar.Value = value;
        lblStato.Content = $"Completato: {value}%";
    });

    await ElaboraFileAsync(listaFile, progress);
}
```

## Perché `IProgress<T>` è così utile

Perché:
- separa il lavoro dal modo in cui la UI mostra il progresso;
- rende più pulito il codice;
- si integra bene con `async/await`;
- è molto leggibile in scenari reali.

## Un punto da non dimenticare

La percentuale reale richiede un totale noto o almeno stimabile.
Se il lavoro non ha un numero prevedibile di passi, potresti dover usare:
- progresso indeterminato;
- oppure una metrica diversa.

Questo è importante da capire, perché non tutti i lavori si prestano allo stesso tipo di progress bar.

## Errore tipico

L'errore classico è mostrare una progress bar “di scena” che non ha alcun legame reale col lavoro.
L'utente percepisce subito quando il progresso non è credibile.

## Regola pratica

Se puoi calcolare un totale e un numero di passi completati, fai progress reporting reale.
Una progress bar è molto più utile quando rappresenta davvero il lavoro che sta avanzando.
