# 04.03.01 - L’idea generale

## Il punto del capitolo

Questo capitolo mette insieme due mondi che in un'applicazione WPF reale devono collaborare molto bene:

- la persistenza dei dati in SQLite;
- la presentazione e modifica dei dati tramite binding.

Il punto non è solo “fare CRUD su un database”.
Il punto è capire come costruire una schermata in cui:
- i dati arrivano dal database;
- vengono caricati in una raccolta bindata;
- la UI li mostra e li modifica;
- le operazioni di inserimento, aggiornamento ed eliminazione restano coerenti sia in memoria sia nel database.

Questa è una delle situazioni più tipiche di un'applicazione desktop gestionale.

## Perché questo capitolo è importante

Molti principianti imparano:
- da una parte il binding;
- dall'altra le query SQL o SQLite.

Poi, però, faticano a farli convivere bene.

Il rischio è creare schermate in cui:
- il database viene aggiornato ma la UI no;
- la UI cambia ma il database no;
- la collezione bindata viene rimpiazzata in modo poco chiaro;
- il `DataGrid` mostra dati corretti solo a metà.

Per questo il capitolo insiste molto sul modello mentale, non solo sul codice.

## Il flusso corretto

Il flusso tipico di una schermata CRUD ben progettata è questo:

1. il database contiene i dati persistenti;
2. il repository li legge;
3. il view model o il livello di presentazione li carica in una raccolta bindata;
4. la UI lavora su quella raccolta;
5. quando l'utente inserisce, modifica o elimina, il repository aggiorna SQLite;
6. la raccolta bindata viene mantenuta coerente con lo stato persistito.

Questa catena va capita molto bene.

## Perché SQLite è una scelta ottima per esempi e strumenti desktop

SQLite è particolarmente adatto in WPF quando vuoi:
- un database locale leggero;
- nessun server esterno;
- distribuzione semplice;
- strumenti desktop autonomi;
- applicazioni gestionali piccole o medie;
- prototipi seri che possono diventare software utili.

È quindi una scelta molto pratica per un capitolo come questo.

## Il ruolo del binding

Il binding non salva dati nel database.
Il binding collega la UI a oggetti .NET.

Questa frase è importantissima.

Quello che salva dati nel database è:
- il repository;
- o un ORM come Entity Framework Core, nella variante successiva.

Il binding, invece, serve a:
- mostrare i dati;
- permettere la modifica;
- mantenere la UI allineata allo stato degli oggetti.

Se confondi questi ruoli, il capitolo diventa molto più difficile di quanto sia davvero.

## Un esempio mentale semplice

Immagina una schermata prodotti con:
- `DataGrid` dei prodotti;
- textbox di modifica;
- pulsanti `Aggiungi`, `Salva`, `Elimina`.

L'utente vede un elenco.
Ma quel che la UI sta davvero mostrando non è “il database”.
Sta mostrando una collezione in memoria, bindata.

Il database è sotto.
La UI non lavora direttamente sul file SQLite: lavora sugli oggetti caricati.

Questa distinzione è fondamentale.

## Perché non bisogna fare tutto direttamente sul controllo

Un errore molto comune è pensare:
“modifico il `DataGrid`, quindi ho modificato il database”.

No.
Hai modificato la rappresentazione dati nella UI.

Per fare CRUD vero, serve anche il livello che scrive e legge da SQLite.

## Il principio chiave

In questo capitolo il lettore deve fissare bene questa idea:

- SQLite custodisce i dati persistenti;
- la collezione bindata custodisce i dati mostrati dalla UI;
- il repository fa da ponte tra i due.

Se questa tripla distinzione è chiara, tutto il CRUD con binding diventa molto più ordinato.

## Regola pratica

Un CRUD WPF ben fatto con SQLite non mette in comunicazione diretta il `DataGrid` con il database.
Mette in comunicazione:
- UI ↔ oggetti bindati
- oggetti bindati ↔ repository
- repository ↔ SQLite
