# 04.01.10 - Perché `CanExecute` è così utile

## Uno dei vantaggi più forti del commanding

Se dovessi scegliere una caratteristica del commanding che mostra subito il suo valore pratico, sceglierei `CanExecute`.

Perché?
Perché permette di legare in modo naturale la disponibilità di un'azione allo stato dell'applicazione.

Questo è molto più elegante che abilitare e disabilitare manualmente i controlli in tanti punti diversi.

## Che cos'è `CanExecute`

`CanExecute` risponde a una domanda molto semplice:

**questo comando può essere eseguito in questo momento?**

Se la risposta è:
- `true` → il controllo collegato può risultare abilitato;
- `false` → può risultare disabilitato.

## Perché è così potente

Perché sposta la logica di disponibilità dell'azione dentro il comando stesso, anziché disperderla nella UI.

In pratica non stai più dicendo:
“disabilita questo pulsante”.

Stai dicendo:
“questa azione, in questo stato, non è eseguibile”.

È una differenza molto importante.

## Un esempio classico

```csharp
SalvaCommand = new RelayCommand(
    _ => Salva(),
    _ => DocumentoModificato);
```

Qui il comando `Salva` è eseguibile solo se `DocumentoModificato` è `true`.

La UI collegata al comando può adattarsi automaticamente.

## Un esempio molto semplice

```csharp
public ICommand CercaCommand { get; }

public MainViewModel()
{
    CercaCommand = new RelayCommand(
        _ => Cerca(),
        _ => !string.IsNullOrWhiteSpace(TestoRicerca));
}
```

Se `TestoRicerca` è vuoto, il comando non può essere eseguito.
Il pulsante collegato può essere disabilitato senza logica manuale sparsa.

## Perché questo è meglio del classico `IsEnabled`

Senza comandi, potresti finire a scrivere:

```csharp
btnSalva.IsEnabled = DocumentoModificato;
menuSalva.IsEnabled = DocumentoModificato;
toolbarSalva.IsEnabled = DocumentoModificato;
```

Con `CanExecute`, la regola vive nel comando e tutti gli attivatori della UI ne beneficiano in modo coerente.

## `CanExecute` migliora anche la leggibilità progettuale

Perché rende esplicita una cosa molto importante:
non tutte le azioni sono sempre disponibili.

Questo porta la UI più vicino al vero stato dell'applicazione.

## Un caso lavorativo molto tipico

Azioni che hanno senso solo in certe condizioni:
- salva solo se ci sono modifiche;
- elimina solo se c'è un elemento selezionato;
- conferma solo se i dati sono validi;
- stampa solo se il documento esiste;
- cerca solo se il filtro non è vuoto.

Tutti questi casi si esprimono benissimo con `CanExecute`.

## Il ruolo di `CanExecuteChanged`

Perché la UI si aggiorni quando la condizione cambia, il comando deve notificare che `CanExecute` va rivalutato.

Per questo, in `RelayCommand`, compare spesso un metodo tipo:

```csharp
RaiseCanExecuteChanged();
```

È un dettaglio pratico molto importante.

## Errori tipici

### Non usare `CanExecute` e poi gestire l'abilitazione a mano
Si perde una parte centrale del modello.

### Usarlo ma non notificare i cambiamenti
La UI non si aggiorna bene.

### Mettere in `CanExecute` logiche troppo complesse o poco leggibili
Come sempre, serve chiarezza.

## Regola pratica

`CanExecute` è utile perché lega la disponibilità dell'azione allo stato reale dell'applicazione.
È uno dei motivi più forti per preferire i comandi ai semplici eventi nei casi non banali.
