# 04.04.11 - Drag and drop e `UserControl`

## Quando il drag and drop vive dentro una vista composta

In applicazioni WPF moderne, il drag and drop non succede quasi mai solo su controlli isolati messi direttamente in una `Window`.
Molto spesso vive dentro:
- `UserControl`;
- viste modulari;
- sezioni della shell;
- componenti riusabili.

Per questo Ã¨ importante capire come si inserisce nel contesto dei `UserControl`.

## Il primo principio corretto

Anche qui il `UserControl` resta una vista.

Questo significa che il drag and drop nella vista dovrebbe:
- lavorare con il `DataContext`;
- usare le raccolte o i comandi del view model;
- evitare di diventare un centro di logica applicativa pesante.

Il drag and drop puÃ² stare nella view, ma il significato dei dati e delle operazioni dovrebbe restare coerente col modello MVVM.

## Un esempio mentale molto utile

Immagina un `UserControl` chiamato `ClientiView` dentro una shell.
Contiene:
- una `ListBox` di clienti disponibili;
- una `ListBox` di clienti selezionati.

Il drag and drop avviene dentro la vista.
Ma le raccolte che modifichi vivono nel `ClientiViewModel`.

Questa Ã¨ la forma piÃ¹ sana del modello.

## Un esempio pratico

XAML del `UserControl`:

```xml
<UserControl x:Class="MioProgetto.Views.ClientiView"
             xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml">
    <Grid Margin="20">
        <Grid.ColumnDefinitions>
            <ColumnDefinition Width="*"/>
            <ColumnDefinition Width="40"/>
            <ColumnDefinition Width="*"/>
        </Grid.ColumnDefinitions>

        <ListBox x:Name="lstDisponibili"
                 ItemsSource="{Binding Disponibili}"
                 DisplayMemberPath="Nome"
                 PreviewMouseMove="LstDisponibili_PreviewMouseMove"/>

        <ListBox x:Name="lstSelezionati"
                 Grid.Column="2"
                 ItemsSource="{Binding Selezionati}"
                 DisplayMemberPath="Nome"
                 AllowDrop="True"
                 DragOver="LstSelezionati_DragOver"
                 Drop="LstSelezionati_Drop"/>
    </Grid>
</UserControl>
```

Code-behind minimo:

```csharp
private void LstDisponibili_PreviewMouseMove(object sender, MouseEventArgs e)
{
    if (e.LeftButton == MouseButtonState.Pressed)
    {
        var listBox = (ListBox)sender;

        if (listBox.SelectedItem is Cliente cliente)
        {
            DragDrop.DoDragDrop(listBox, cliente, DragDropEffects.Move);
        }
    }
}

private void LstSelezionati_DragOver(object sender, DragEventArgs e)
{
    e.Effects = e.Data.GetDataPresent(typeof(Cliente))
        ? DragDropEffects.Move
        : DragDropEffects.None;

    e.Handled = true;
}

private void LstSelezionati_Drop(object sender, DragEventArgs e)
{
    if (DataContext is not ClientiViewModel vm)
        return;

    if (e.Data.GetDataPresent(typeof(Cliente)))
    {
        var cliente = (Cliente)e.Data.GetData(typeof(Cliente))!;

        if (!vm.Selezionati.Contains(cliente))
        {
            vm.Selezionati.Add(cliente);
            vm.Disponibili.Remove(cliente);
        }
    }
}
```

## PerchÃ© questo esempio Ã¨ buono

PerchÃ©:
- il `UserControl` gestisce l'interazione UI del drag and drop;
- il significato dei dati vive nel view model;
- le collection bindate vengono aggiornate correttamente;
- la vista non si inventa un proprio stato scollegato dal `DataContext`.

## Dove serve attenzione

Il pericolo nei `UserControl` Ã¨ trasformare il drag and drop in un grande blocco di logica applicativa dentro il code-behind.

Meglio tenere nel code-behind solo la parte:
- strettamente UI;
- di recupero payload;
- di invio ai dati o alle collection corrette.

Se la logica cresce molto, puÃ² essere opportuno spostare parte della decisione nel view model o in un servizio.

## Quando il `UserControl` puÃ² essere il target intero

A volte il target non Ã¨ solo una `ListBox`, ma l'intero `UserControl`.
Per esempio:
- drop di file sull'intera sezione;
- drop di immagini in una vista dedicata;
- drop su un pannello contenitore.

In quei casi puÃ² avere senso mettere `AllowDrop="True"` su un contenitore piÃ¹ ampio o sul `UserControl` stesso.

## Errore tipico

L'errore classico Ã¨ usare il `UserControl` come se fosse una mini-app scollegata dal suo view model.
In quel caso il drag and drop finisce per rompere la coerenza dell'architettura.

## Regola pratica

Nel `UserControl`, il drag and drop appartiene alla vista come gesto e interazione.
Ma il significato dei dati e lo stato da aggiornare dovrebbero continuare a vivere nel view model o nelle raccolte bindate del controllo.
