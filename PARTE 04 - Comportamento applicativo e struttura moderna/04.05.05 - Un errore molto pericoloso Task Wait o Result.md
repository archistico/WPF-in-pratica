# 04.05.05 - Un errore molto pericoloso: `Task.Wait()` o `.Result`

## Uno degli errori più insidiosi in una UI WPF

Tra tutti gli errori legati ad `async/await`, questo è uno dei più pericolosi:

- `Task.Wait()`
- `.Result`

A prima vista sembrano scorciatoie innocue.
In realtà, in un'applicazione WPF, possono facilmente:
- bloccare la UI;
- creare deadlock;
- rendere il comportamento dell'app inspiegabilmente congelato.

Per questo vanno trattati con molta cautela.

## Che cosa fanno davvero

### `Wait()`
Blocca il thread corrente finché il task non è completato.

### `.Result`
Recupera il risultato del task, ma se il task non è ancora completato blocca il thread corrente.

Questa parola è la chiave: **bloccano**.

## Perché questo è pericoloso sul thread UI

Perché il thread UI deve restare libero per:
- processare il `Dispatcher`;
- eseguire continuazioni;
- reagire agli eventi;
- ridisegnare la finestra.

Se lo blocchi con `Wait()` o `.Result`, puoi creare una situazione in cui:
- il task attende di riprendere sul contesto UI;
- ma il contesto UI è bloccato proprio mentre aspetta il task.

Ed ecco il deadlock classico.

## Un esempio pericoloso

```csharp
private void BtnCarica_Click(object sender, RoutedEventArgs e)
{
    var testo = CaricaTestoAsync().Result;
    lblStato.Content = testo;
}
```

Se `CaricaTestoAsync()` usa `await` e vuole riprendere sul contesto UI, questa chiamata può bloccare tutto.

## La versione corretta

```csharp
private async void BtnCarica_Click(object sender, RoutedEventArgs e)
{
    var testo = await CaricaTestoAsync();
    lblStato.Content = testo;
}
```

Qui il thread UI non viene bloccato in modo sincrono.
Il metodo si sospende e la finestra resta reattiva.

## Perché l'errore è così subdolo

Perché `Wait()` e `.Result` a volte sembrano funzionare.
Quindi il lettore pensa:
“allora va bene”.

Ma il problema emerge:
- in certe condizioni di timing;
- con certe continuazioni;
- quando entrano in gioco il contesto UI e il `Dispatcher`.

È proprio questo che li rende pericolosi.

## Un altro esempio

```csharp
var task = ImportaDatiAsync();
task.Wait();
```

Anche qui il thread corrente resta fermo ad aspettare.
Se quel thread è il thread UI, sei in una zona molto rischiosa.

## Il modello corretto

In WPF, se sei in codice che può essere asincrono, la regola generale è:
usa `await` invece di bloccare il task in modo sincrono.

Questa è quasi sempre la strada giusta.

## Esistono eccezioni?

In contesti molto specifici e consapevoli, fuori dal thread UI, puoi anche vedere `.Wait()` o `.Result`.
Ma nel flusso tipico di una finestra WPF sono strumenti da evitare, salvo casi davvero particolari e molto controllati.

## Perché `await` è la risposta naturale

Perché:
- non blocca il thread UI nello stesso modo;
- si integra con il `Dispatcher`;
- mantiene più leggibile il flusso;
- riduce enormemente il rischio di deadlock.

## Errore tipico

L'errore classico è pensare:
“mi serve il risultato subito, quindi uso `.Result`”.

In WPF questa scorciatoia è spesso molto più costosa del beneficio apparente.

## Regola pratica

Sul thread UI di WPF, evita `Task.Wait()` e `.Result` quasi sempre.
Se stai aspettando un task, la strada naturale e sicura è normalmente `await`.
